# 链路17：异常检测精度回测与算法择优

## 第一步：上传标注数据

### 操作页面
**侧边栏** → **智能分析室** → **异常检测** → **异常检测基准**

### 操作步骤

1. **点击「新建基准」按钮**

2. **填写基准信息**
   - **资产**：选择目标资产
   - **指标名称**：输入 `cpu_usage`
   - **算法**：选择 `MAD`（或其他算法）
   - **标注数据**：上传 CSV 文件（包含历史值 + 是否异常标签）
     - 格式：`timestamp,value,is_anomaly`
     - 示例：
       ```
       2026-07-01 00:00:00,45.2,false
       2026-07-01 00:05:00,46.1,false
       2026-07-01 00:10:00,95.8,true
       ```

3. **点击「提交」**
   - 系统调用 `POST /anomaly/api/benchmark`
   - 后台执行 `anomaly_eval_service.record_benchmark()`：
     - 计算 Precision、Recall、F1 分数
     - 写入 `AnomalyBenchmark` 表（asset_id/metric_name/algorithm/precision/recall/f1/created_at）

### 看到什么
- 基准列表显示新创建的基准记录
- 显示计算得到的 Precision、Recall、F1 分数

## 第三步：获取算法推荐

### 操作页面
**侧边栏** → **智能分析室** → **异常检测** → **异常检测配置**

### 操作步骤

1. **新建异常检测规则**
   - 点击「新建规则」按钮
   - **指标名称**：输入 `cpu_usage`
   - **资产**：选择目标资产

2. **点击「推荐算法」按钮**
   - 系统调用 `GET /anomaly/api/benchmark/recommend?asset_id=&metric_name=`
   - 后台执行 `anomaly_eval_service.recommend_algorithm()`：
     - **优先查标注数据**：查询 90 天内该 asset+metric 的各算法 F1，选 max F1 算法
     - **无标注数据时降级**：调用 `_recommend_by_metric特征()` 做指标特征匹配：
       - `cpu` → MAD（CPU 噪声干扰强，MAD 抗噪）
       - `mem/memory` → EWMA（内存渐变趋势）
       - `disk/io` → 3σ（磁盘 I/O 平稳）
       - `latency/p99/p95` → MAD（延迟偏态分布）
       - `error/fail/5xx` → Isolation Forest（稀疏事件）
       - `qps/throughput` → STL（周期性波动）
       - `order/revenue/业务` → Prophet（业务时序）
     - **再回退到统计推断**：查 MetricRecord 采样计算变异系数 CV/偏度/MAD：
       - CV > 1.0 → Isolation Forest
       - 偏度 > 2 → MAD
       - std/mad > 3 → MAD
       - 其他 → EWMA

3. **查看推荐结果**
   - 系统自动填充推荐算法到表单
   - 显示推荐理由（如 "基于标注数据，MAD 算法 F1 分数最高"）

### 看到什么
- 算法字段自动填充为推荐算法
- 显示推荐理由和置信度
### 结果
异常检测闭环：标注数据 → 基准回测 → 算法推荐 → 配置规则。

# 链路17（续）：Trace 异常配置管理

Trace 异常配置管理用于监控分布式链路调用的延迟和错误率，当服务/端点的性能超过阈值时自动告警。

## 第一步：新建 Trace 异常配置

### 操作页面
**侧边栏** → **智能分析室** → **Trace异常配置**

### 操作步骤

1. **点击「新建配置」按钮**

2. **填写配置信息**
   - **服务名称**：输入要监控的服务名，如 `order-service`
   - **端点**：输入要监控的 API 端点路径，如 `/api/orders`
   - **延迟阈值(ms)**：输入 P95 延迟阈值，如 `500`（毫秒，超过此值视为异常）
   - **错误率阈值(%)**：输入错误率阈值，如 `5`（百分比，超过此值视为异常）
   - **最小 Trace 数**：输入 `10`（评估窗口内至少需要这么多 Trace 才触发检查，避免数据稀疏误报）
   - **评估窗口(分钟)**：输入 `5`（滑动窗口大小，在此时间段内聚合统计）

3. **点击「保存」**
   - 系统调用 `POST /trace-anomaly/api/configs`
   - 新增配置默认启用（`is_active=true`）

### 看到什么
- Toast 提示「创建成功」
- 新配置出现在配置列表

# 链路18：SOP 知识自动生成

## 第一步：从故障单生成 SOP

### 操作页面
**侧边栏** → **值班驾驶舱** → **故障管理** → **故障详情**

### 操作步骤

1. **选择故障单**
   - 在故障列表中，点击某条已解决的故障单
   - 进入故障详情页

2. **点击「生成 SOP 知识」按钮**
   - 在故障详情页，点击「生成 SOP 知识」按钮
   - 系统调用 `POST /knowledge/api/auto-gen/sop/incident/{incident_id}`

3. **自动生成 SOP**
   - 后台执行 `knowledge_autogen_service.generate_sop_from_incident(incident_id, db)`：
     - 获取故障单详情（`Incident`）+ 关联资产（`Asset`）+ 关联告警（`Alert`）
     - 查 AIProvider（`enabled=True`）取 default_provider
     - 调用 LLM 生成结构化 SOP：
       - `title`：故障标题衍生
       - `symptom`：告警消息汇总
       - `root_cause`：根因分析
       - `solution`：修复建议
       - `sop_steps`：JSON 数组，每个元素包含：
         - `step`：步骤号
         - `action`：动作
         - `command`：命令
         - `expectation`：预期结果
       - `tags`：标签
       - `severity`：严重级别
       - `asset_type`：资产类型
     - 创建 `KnowledgeDraft` 记录：
       - `source_type`：`sop`
       - `status`：`pending`
       - `sop_steps`：JSON 字符串

### 看到什么
- Toast 提示「SOP 知识已生成，等待审批」
- 草稿出现在「AI 知识草稿」页面

## 第三步：审批 SOP 草稿

### 操作步骤

1. **查看全局概览**
   - 页面加载时并发调用 6 个 API：
     - `GET /api/ops-analytics/overview`：全局 KPI 统计
     - `GET /api/ops-analytics/mtta-mttr`：MTTA/MTTR 计算
     - `GET /api/ops-analytics/alert-trend`：告警趋势
     - `GET /api/ops-analytics/remediation-effect`：自愈效果
     - `GET /api/ops-analytics/ai-efficiency`：AI 效能
     - `GET /api/ops-analytics/notification-stats`：通知统计

2. **查看 6 大 KPI 卡片**
   - **总告警数**：历史告警总数
   - **总故障数**：历史故障总数
   - **MTTR**：平均修复时间（分钟）
   - **自愈成功率**：自动修复成功率（%）
   - **AI 任务完成率**：AI Agent 任务完成率（%）
   - **通知送达率**：通知发送成功率（%）

3. **查看 6 张 ECharts 图**
   - **告警趋势图**：按天/小时聚合告警数量趋势
   - **MTTR 趋势图**：MTTR 随时间变化
   - **自愈效果图**：自愈成功率 + 各规则效果
   - **AI 效能图**：Agent 评估 + 工具调用统计
   - **通知统计图**：通知类型 + 成功率
   - **工具排行图**：TOP-10 工具调用次数

4. **切换时间范围**
   - 点击时间范围切换按钮（7天 / 30天 / 90天）
   - 页面数据更新为对应时间范围的统计

### 看到什么
- 6 大 KPI 卡片显示整体运营指标
- 6 张 ECharts 图直观展示趋势和分布
- 时间范围切换后数据更新

## 第二步：执行诊断工具

### 操作页面
**侧边栏** → **运维工作台** → **任务中心** → **诊断工具中心**

### 操作步骤

1. **选择工具**
   - 点击某工具卡片（如 `os.overview`）
   - 右侧弹出执行面板

2. **选择资产**
   - 在执行面板中，选择目标资产
   - 输入资产 IP 或名称

3. **执行工具**
   - 点击「执行」按钮
   - 系统调用 `POST /api/diagnostic-tools/execute`
   - 后端查 `Asset(ip/connection_config)` → SSH 连接 → 执行工具 `command_template`

4. **查看结果**
   - 右侧面板显示执行结果：
     - **输出内容**：命令执行的 stdout
     - **退出码**：exit_code（0=成功）
     - **执行耗时**

- 成功结果以绿色徽章显示
- 失败结果以红色徽章显示

## 第三步：执行 Flexible 层工具（需校验）

### 操作页面
**侧边栏** → **运维工作台** → **任务中心** → **诊断工具中心**

### 操作步骤

2. **输入命令**

   - **MySQL 工具**：输入 SQL 查询（如 `SHOW PROCESSLIST`）

3. **命令校验**
   - 系统调用 `POST /api/diagnostic-tools/validate`
   - 后端执行 `validate_command()` 做 AST 白名单校验：
     - **安全命令通过**：`df`、`ls`、`cat`、`top` 等只读命令
     - **危险命令拦截**：`rm -rf`、`kill`、`shutdown`、`reboot`、`fork bomb` 等

   - 校验通过后，点击「执行」
   - 系统执行命令并返回结果

## 常见问题

### Q1：异常基准 F1 分数为 0？
- **检查**：标注数据格式是否正确（`timestamp,value,is_anomaly`）
- **检查**：标注数据是否有异常样本（`is_anomaly=true`）
- **检查**：算法是否适合该指标类型

### Q2：SOP 生成失败？
- **检查**：故障单是否有关联告警和资产
- **检查**：AIProvider 是否配置正确
- **检查**：LLM 调用是否超时

### Q3：运营分析看板数据为空？
- **检查**：是否有告警/故障/自愈/Agent 评估数据
- **检查**：时间范围是否正确
- **检查**：API 是否正常返回

### Q5：Trace 异常配置没有触发告警？
- **检查**：配置是否已启用（`is_active=true`）
- **检查**：`最小 Trace 数` 是否过大（窗口内数据不足会跳过检查）
- **检查**：`评估窗口` 是否合理（太小导致波动大，太大导致响应慢）
- **检查**：后端 `check_trace_anomalies` 后台任务是否在运行

### Q6：Trace 异常配置编辑后不生效？
- **检查**：配置的 `service` 和 `endpoint` 是否正确匹配实际链路数据
- **检查**：后端 `trace_anomaly_service` 是否读取了更新后的配置

### Q4：诊断工具执行失败？
- **检查**：资产是否在线
- **检查**：SSH 凭据是否正确
- **检查**：命令是否被白名单拦截
- **检查**：Flexible 层命令是否通过校验

## 专业术语

| 术语 | 英文 | 说明 |
|------|------|------|
| Benchmark 回测 | Benchmark Backtesting | 用标注数据评估异常检测算法性能 |
| Precision | 精确率 | 预测为异常的样本中实际为异常的比例 |
| Recall | 召回率 | 实际为异常的样本中被预测为异常的比例 |
| F1 Score | F1 分数 | Precision 和 Recall 的调和平均 |
| Trace Anomaly Detection | Trace 异常检测 | 基于延迟和错误率阈值的分布式链路调用异常检测 |
| Sliding Window | 滑动窗口 | 按时间窗口聚合统计，窗口不断向前滑动 |
| P95 Latency | P95 延迟 | 排序后第 95 百分位的响应时间，排除极端值干扰 |
| Toggle | 开关切换 | 启用/禁用配置的即时切换，无需删除重建 |
| MTTA | Mean Time To Acknowledge | 平均确认时间 |
| MTTR | Mean Time To Resolve | 平均修复时间 |
| SOP | Standard Operating Procedure | 标准操作流程 |
| AST | Abstract Syntax Tree | 抽象语法树，用于命令安全校验 |
