**关键决策**：
- fail-soft 风格：所有新 admin 端点异常返回 200 + warning，不返回 5xx
- 审计中间件 fail-soft：审计记录失败不影响主流程，仅 logger.warning
- 密码字段脱敏：`password/token/secret/api_key/private_key/credential/kubeconfig` 等字段存储前替换为 `***`
- 路径推断 target_type：正则匹配 16 类业务路径（asset/incident/user_permission/ai_provider/config/chaos/remediation/knowledge/alert/sre/script/deploy/change/tenant/dashboard/notification/datasource）
- 告警聚类用进程内缓存（TTL=30s），不持久化（避免脏数据）；RAG 评估缓存 60s
- RAG 评估数据集从 `knowledge_base` 自动构造（query=title, relevant_doc_id=关联 KbDocument），无关联文档时 fail-soft 返回 warning
- 审计覆盖率三指标分层：**能力覆盖率**（中间件可拦截，=100%）/ **精确实际覆盖率**（按 method+route_path 精确匹配 audit_logs）/ **类型动作覆盖率**（同 type/action 有记录，辅助参考）；修复了旧版按 type/action 聚合导致 88% 假象的问题
- AuditLog 表新增 `route_path` 列存路由模板（如 `/api/tags/{tag_id}`），与 `path`（实际路径如 `/api/tags/5`）分离，支撑精确覆盖率计算

**专业名词学习**：
- Field Drift Detection（字段漂移检测）— 文档约定的字段名与代码实际使用不一致，导致静默数据丢失
- Alert Correlation / Convergence（告警收敛）— 多条告警按服务/时间/拓扑聚类，关联根因，减少工单量
- recall@k / MRR / nDCG@k — 信息检索标准指标：前 k 命中率 / 平均倒数排名 / 归一化折损累计增益
- Audit Coverage Matrix（审计覆盖矩阵）— 写端点 × 是否记审计 × 记录字段的二维矩阵，量化审计完整性
- Fail-Soft（失效软化）— 异常不抛 5xx，返回 200 + warning，保证前端页面可用

**专业提问话术**：
- "我们的字段契约检测能否挂进 CI/CD，违规即 fail？当前 68 个高置信违规是否需要按优先级排期修复？"
- "告警聚类的 30s 缓存是否会导致根因推荐延迟？是否需要事件驱动刷新（新告警入队即重算）？"
- "RAG 评估的标注集只来自 knowledge_base，是否需要引入用户反馈（点击/采纳）作为补充标注？"
- "审计覆盖矩阵的 88% 覆盖率，未覆盖的 36 个端点是什么类型？是否需要补齐？"

## 一、P0 紧急（基建空白，必须先补）

### 1. 测试体系近乎空白

**现状**：100+ 路由只有 `tests/` 6 个文件，覆盖率约等于 0。MEMORY.md 里所有"端到端验证"都是手 curl，无回归保护。

**目标**：
- 引入 `pytest + httpx + pytest-asyncio + coverage`
- 按 router 维度批量生成冒烟测试：每个端点至少 1 条 happy path + 1 条 4xx
- 覆盖率起步 40%
- 把 MEMORY.md 里"端到端全通过"的脚本固化成 pytest 用例

**收益**：消除"修了 A 破 B"的反复横跳，PR 阶段就能拦截回归。

### 2. CI/CD 完全缺失

**现状**：无 `.github/workflows`、无 `.gitlab-ci.yml`。每次靠人肉 `npm run build` + `python -m py_compile` 验证。

**目标**：
- 加 GitHub Actions 流水线：`ruff check + pytest + npm run build`
- PR 合并前强制通过
- 自动跑 CONTRACT.md 字段漂移检测（见 P2.8）

**收益**：所有"修了 A 破 B"的问题在合并前自动拦截，不再依赖人工验证。

### 3. SQLite 单机瓶颈 + 无版本化迁移

**现状**：
- `aiops.db` 7MB + WAL 6MB
- FastAPI 多 worker 下 SQLite 写锁会成瓶颈
- 告警/审计高频写入必踩坑
- `main.py:_MIGRATIONS` 是手写幂等 ALTER TABLE，无版本号无 downgrade

**目标**：
- 迁 PostgreSQL（`psycopg2-binary` 已在可选依赖）
- 引入 **Alembic** 做版本化迁移，支持 upgrade/downgrade
- 借迁移一次性修 CONTRACT.md 标注的几十处"❌ 缺 is_/_at/泛型名"字段

**收益**：解决并发写锁 + schema 变更可追溯 + 字段规范一次性对齐。

## 二、P1 高（架构层）

### 4. 路由扁平堆叠

**现状**：`app/routers/` 100+ 文件一字铺开，无领域边界。找代码靠 glob，维护成本随模块数线性涨。

**目标**：按业务域分包：
```
app/domains/{assets,alerts,k8s,ai,sre,chaos,knowledge,incident,remediation}/
  ├─ router.py
  ├─ service.py
  ├─ models.py
  └─ schemas.py
```

**收益**：领域内聚、跨域解耦，新模块接入有清晰模板。

### 5. AI Provider 健康度路由

**现状**：已有 fallback（按 id 升序逐个试），但第一个连不上要等 15s 超时才切下一个（MEMORY 里 OpenAI 超时 18s 才到 zm）。

**目标**：
- 加 **熔断器（Circuit Breaker）**
- 健康度表：记录每个 provider 的 `last_success_at / failure_count / p95_latency`
- 按健康度排序选 provider，连续 N 次失败自动熔断 60s
- 半开态试探恢复

**收益**：AI 调用 P95 延迟从 18s+ 降到 3s 级，可用性从"最终能成"到"快速成"。

### 6. 背景任务无可观测性

**现状**：`main.py:577` 注册了 trace_anomaly / slo 计算 / 草稿生成等定时任务，但没地方看跑没跑、成功没成功、跑了多久。当前 trace_anomaly "运行中但无反应"的坑就是因为不可见。

**目标**：加 `GET /api/admin/background-tasks` 看板：
- 每个任务的 `last_run_at / last_status / last_duration / error_count / next_run_at`
- 失败任务红色高亮
- 支持手动触发 / 暂停

**收益**：定时任务从"黑盒"到"白盒"，问题第一时间暴露。

### 7. 前端首屏体积

**现状**：90+ 视图，未确认分包策略。

**目标**：
- 确认 `router/index.js` 用 `() => import(...)` 动态导入
- Vite `manualChunks` 把 `element-plus / echarts / vendor` 分包
- 监控首屏 bundle 体积，目标 < 500KB gzip

**收益**：首屏加载从"白屏几秒"到"秒开"。

## 三、P2 中（质量层）

### 8. CONTRACT.md 字段漂移检测

**现状**：契约列了几十处"❌ 缺 is_/_at/泛型名"的违规字段，但没工具校验代码是否还在用旧名。违反契约会导致静默数据丢失（CONTRACT.md 开篇已警告）。

**目标**：写 `scripts/check_contract.py`：
- 解析 `models.py` 的 Column 名
- 解析 `routers/*.py` 的 dict key
- 对照 CONTRACT.md 输出 diff 报告
- 挂进 CI，违规即 fail

**收益**：把"文档约定"变成"工具强制"，杜绝字段漂移。

### 9. 告警收敛能力（AIOps 核心闭环）

**现状**：已有 `alert_storm.py / alert_silence.py`，但 AIOps 核心的"告警聚合→同一根因→单一工单"链路缺顶层设计。当前告警→工单→知识草稿是线性的。

**目标**：补"告警聚类→关联拓扑→根因推荐"闭环：
- 同拓扑路径 / 同时间窗 / 同服务的告警聚类
- 聚类结果关联到拓扑图（topology_service 已有）
- 根因推荐复用 pagerank_rca / graph_inference

**收益**：从"一条告警一个工单"到"一类根因一个工单"，工单量降一个数量级。

### 10. RAG 检索质量无指标

**现状**：已有 `rag_engine_v2 / vector_store / embedding_service`，但没看到 `recall@k / MRR / nDCG` 评测。"RAG 效果好坏"全靠主观。

**目标**：
- 加 `GET /api/rag/eval` 跑标注集出指标
- 标注集来源：knowledge_base 已审批通过的知识 + 用户反馈
- 指标看板：`recall@5 / MRR / nDCG @10 / 平均检索延迟`
- 指标劣化自动告警

**收益**：RAG 优化有数据支撑，不再凭感觉调参。

### 11. 审计日志完整性

**现状**：`audit.py` 存在，但审批设置端点权限漏洞（MEMORY 2026-07-19 记录）说明审计不完整。

**目标**：
- 确认所有写操作（资产变更/审批/配置修改/脚本执行/Token 管理）都过审计中间件
- 写一份"审计覆盖矩阵"：每个写端点 × 是否记审计 × 记录字段
- 缺失项补齐

**收益**：合规审计可追溯，权限漏洞无藏身之处。

## 四、P3 优化（运维层）

### 12. 自举监控

**现状**：AIOps 给别人监控但自己没 `/metrics`（Prometheus）端点。医者不能自医。

**目标**：
- 加 `prometheus-fastapi-instrumentator`
- 暴露 QPS / p95 / 错误率 / DB 连接池 / 后台任务积压
- 接入 Grafana 看板

**收益**：系统自身健康度可观测，问题主动发现而非被动报告。

### 13. API 类型对齐

**现状**：FastAPI 自动生成 OpenAPI，但前端是手写 axios。字段漂移靠人眼发现。

**目标**：
- `openapi-typescript` 从 OpenAPI spec 生成 TS 类型
- 前端 axios 强类型化
- 编译期发现字段漂移

**收益**：前后端字段漂移在编译期暴露，不再等运行时静默丢数据。

### 14. 移动端 E2E

**现状**：MEMORY 记录了一堆 uni-app H5 编译缓存 / publicPath 覆盖 / @tap 拦截的坑，都是手测发现。

**目标**：
- Playwright 跑 H5 关键链路
- 覆盖：登录 / 告警列表 / 工单审批 / 资产查看
- 挂进 CI

**收益**：移动端改动不再靠"改完手动点一遍"。

### 15. 多租户隔离验证

**现状**：`tenant_management.py / tenant_service.py / tenant_context.py` 存在，但未验证所有查询都过 `tenant_id` 过滤。

**目标**：
- 写测试：租户 A 登录后查所有 GET 端点，断言返回数据都属于租户 A
- 写测试：租户 A 尝试访问租户 B 资源 ID，断言返回 404
- 缺失过滤项补齐

**收益**：杜绝租户数据越权，合规底线。

## 五、专业名词学习

| 名词 | 含义 |
|------|------|
| Test Pyramid（测试金字塔） | 单元测试 > 集成测试 > E2E 的数量配比，当前是倒金字塔（几乎全是手测 E2E） |
| Circuit Breaker（熔断器） | 连续失败 N 次后短路，避免雪崩级联等待；半开态试探恢复 |
| Alembic Migration（版本化迁移） | 数据库 schema 变更版本控制，支持 upgrade/downgrade，替代手写 ALTER TABLE |
| Domain-Driven Packaging（领域驱动分包） | 按业务域而非技术层组织代码，降低模块间耦合 |
| OpenAPI Codegen（接口契约代码生成） | 从 OpenAPI spec 反向生成前端 TS 类型 / 后端 client，消除手写字段漂移 |
| Self-Observability（自举可观测性） | 监控系统自身也需要被监控，否则"医者不能自医" |
| Prometheus Instrumentation（普罗米修斯插桩） | 通过 `/metrics` 端点暴露指标，被 Prometheus 拉取 |
| Alert Correlation / Clustering（告警关联/聚类） | 同根因告警聚合为单一事件，降低告警风暴 |
| recall@k / MRR / nDCG | RAG 检索质量标准指标：召回率 / 平均倒数排名 / 归一化折损累计增益 |
| Tenant Isolation（租户隔离） | 多租户系统中数据按 tenant_id 严格隔离，杜绝越权 |

- "AI Provider 的健康度路由是否做了熔断 + 优先级排序？"
- "数据库迁移是否版本化？SQLite 到 PostgreSQL 的迁移路径是否打通？"
- "CONTRACT.md 的字段规范是否有自动化校验？还是只靠人工对照？"
- "后台定时任务的成功率/耗时是否可观测？"
- "RAG 检索质量是否有量化指标（recall@k / MRR）？"
- "监控系统自身的运行指标是否暴露了 /metrics？"
- "告警是否做了同根因聚类？还是一条告警一个工单？"
- "多租户场景下，是否所有查询都过 tenant_id 过滤？有无自动化测试？"

## 七、执行建议

**第一阶段（1-2 周）**：P0 三项 — 测试体系 + CI/CD + 数据库迁移
- 这是后续所有优化的地基，没地基都是空中楼阁
- 测试体系先建框架 + 冒烟测试，覆盖率 40% 即可
- CI 先跑起来，后续 P1/P2 都靠它兜底

**第二阶段（2-3 周）**：P1 四项 — 领域分包 + AI 熔断 + 背景任务看板 + 前端分包
- 架构层优化，一次性把"扩展性"和"可观测性"补齐
- AI 熔断收益最直观（18s → 3s）

**第三阶段（1-2 周）**：P2 四项 — 契约校验 + 告警收敛 + RAG 指标 + 审计矩阵
- 质量层加固，把"约定"变"强制"

**第四阶段（持续）**：P3 五项 — 自举监控 + API 类型对齐 + 移动 E2E + 租户隔离测试
- 运维/合规向，按需推进

---

> 本方案为方向性建议，每项落地前需细化任务拆分与工时评估。
> 建议按阶段推进，每阶段完成后更新 MEMORY.md 记录决策与发现。
