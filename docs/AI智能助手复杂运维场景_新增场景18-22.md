## 测试场景 18：夜间无人值守告警与冷静判断（复杂）

### 场景目标

验证 AI 在凌晨无人值守时段收到告警时，能做出冷静判断，区分"真实故障"与"瞬态抖动"，在需要时自动执行诊断操作而不等待人工介入，并在判断为误报时给出推荐操作而非盲目升级。

### 测试数据准备

```
时间：凌晨 03:15
告警：server-001 CPU 92%（持续 1 分钟，然后恢复至 30%）
背景：该服务器夜间的正常 CPU 基线为 15-25%
历史：24 小时内有 3 次类似的瞬时 CPU 尖刺（分别发生在 01:00、02:30、03:15）
每次尖刺持续时间 < 2 分钟，之后自动恢复
没有伴随的内存/磁盘/IO 告警
```

### 多轮对话测试步骤

#### 第 1 轮：用户通报夜间告警

**用户输入：**
```
刚收到告警，server-001 半夜 CPU 飙到 92% 了，这个严重吗？需要现在处理吗？
```

**期望 AI 行为：**
- 查询该资产近期告警历史（24 小时内）
- 对比当前告警与历史模式的异同
- 判断是否为反复出现的瞬态峰值的规律性模式
- 给出冷静判断：区分"系统性故障"与"周期性抖动"
- 如果不严重，明确建议"暂不处理"并说明理由
- 同时建议设置自动规则来抑制同类告警

**期望输出示例：**
```
冷静分析中...

📊 历史模式识别：
我查了 server-001 近 24 小时的 CPU 历史：
| 时间   | CPU 峰值 | 持续时间 | 伴随告警       |
|--------|---------|---------|---------------|
| 01:00  | 89%     | 90秒    | 无             |
| 02:30  | 91%     | 75秒    | 无             |
| 03:15  | 92%     | 60秒    | 无（刚刚发生） |

🔍 判断：这看起来是周期性瞬态抖动，不是系统性故障
- 三次尖刺的模式高度一致（CPU 88-92%，持续 60-90 秒，自动恢复）
- 没有伴随内存/磁盘/网络告警（排除资源泄漏）
- 每次都在凌晨发生，可能是定时任务（备份/批处理/日志压缩）
- 当前 CPU 已恢复到 30%（正常范围）

🛌 建议：现在可以继续睡，不需要紧急处理
- 建议明天白天去确认 crontab 查看凌晨是否有定时任务

📋 改进建议（明天处理）：
1. 确认 01:00/02:30/03:15 的定时任务来源
2. 如果确认是定期批处理，为 CPU 告警规则加抑制条件
3. 或在告警规则里加时间段过滤
```

---

#### 第 2 轮：用户确认定时任务存在

**用户输入：**
```
被你猜中了，确实是 cron 定时任务，凌晨跑日志压缩的。
那这个告警还要留着吗？要不要做点什么优化？
```

**期望 AI 行为：**
- 确认根因是已知的定时任务
- 给出告警静默/告警规则优化建议
- 提供性能优化建议
- 考虑是否生成知识沉淀

**期望输出示例：**
```
确认了是定时日志压缩任务，那这个就不再是真正的告警了。

🛠 建议操作（按优先级排序）：

1️⃣ 立即静默该告警规则 24 小时，避免今晚再收到同样误报

2️⃣ 明天优化告警规则阈值
   当前：CPU > 90% 立即触发
   建议改为：CPU > 90% 持续 3 分钟才触发

3️⃣ 优化定时任务，减少资源争抢

4️⃣ 记录到知识库作为运维笔记

```

```
也改了定时任务加了 nice 19。帮我验证一下效果。
```

**期望 AI 行为：**
- 验证告警规则是否已更新
- 验证历史相同模式不再触发
- 给出确认结论

## 测试场景 19：知识遗忘与重新发现——知识库陈旧（复杂）

### 场景目标

验证 AI 在参考知识库历史案例时，能意识到"知识可能已过时"（如版本升级、架构变更），并能根据当前系统状态验证知识的有效性。发现过时知识时，主动建议更新而非盲目遵循。

### 测试数据准备

```
知识库条目 #5：【Mysql 调优】"数据库连接池设 200 最佳"
- 创建时间：2025-03-15（1 年 4 个月前）
- 适用版本：MySQL 5.7
- 当前系统：MySQL 8.0（半年前已从 5.7 升级到 8.0）

当前系统实际：
- database-01 已升级到 MySQL 8.0（6 个月前）
- 8.0 版本推荐连接池 = CPU 核数 × 4（当前机器 32 核 → 推荐 128）
- 连接池当前设为 200（还在用旧知识）
- 当前连接数均值 120/200，峰值 180/200（偏高）
```

### 多轮对话测试步骤

#### 第 1 轮：知识库查询

**用户输入：**
```
database-01 的连接池应该设多大？帮我查查知识库
```

**期望 AI 行为：**
- 搜索知识库获取相关知识
- 检查知识条目的创建/更新时间
- 对比当前系统版本与知识适用的版本
- 识别知识可能已过时
- 验证当前系统版本的官方建议

**期望输出示例：**
```
搜索知识库中...

📚 找到 1 条相关知识：
【Mysql 调优】数据库连接池设 200 最佳
- 创建时间：2025-03-15（1 年 4 个月前）
- 适用版本：MySQL 5.7

⚠️ 知识过时预警
- 当前 database-01 版本：MySQL 8.0（半年前已完成升级）
- 8.0 推荐公式：连接池 = CPU 核数 × 4 → 当前机器 32 核 → 推荐 128
- 当前使用旧知识的 200 虽然可用，但高峰期接近上限存在风险

建议更新知识库条目，增加 MySQL 8.0 的调优说明。
```

#### 第 2 轮：用户确认知识过时

```
确实忘了已经升级到 8.0 了。那 128 是推荐值的话，改成 128 安全吗？
改之前要不要先验证一下峰值连接数？
```

**期望 AI 行为：**
- 先做验证（查历史峰值，分析业务高峰期连接需求）

**期望输出示例：**

```
| 工作日夜间  | 85      | 110     |
| 周末        | 60      | 95      |

🎯 推荐安全方案：分步调整
第 1 步：从 200 降到 160（安全下调 20%，留有余量）
第 2 步：观察一周，确认峰值 < 150 再降到 128

```

## 测试场景 20：自愈闭环验证——自动恢复后的根因确认（复杂）

### 场景目标

验证 AI 在系统自动恢复（自愈）后，能主动分析自愈是否真正解决了根因，还是只是暂时掩盖了症状。这是自动化运维中"信任但验证"的核心能力。

### 测试数据准备

```
线上故障记录：
- 时间线：
  14:00:00 — server-001 CPU 95% 告警触发
  14:00:15 — 自动自愈流程触发（restart nginx）
  14:00:45 — nginx 重启完成，CPU 降至 40%
  14:01:00 — 告警自动恢复
- 当前时间：14:30（故障已自动恢复 30 分钟）
- server-001 CPU 当前：42%（正常范围）
- 但 nginx 错误日志显示：重启前有大量 "worker_connections exceed" 警告
- 该告警在过去一周内触发了 4 次，每次都通过自愈恢复
```

### 多轮对话测试步骤

#### 第 1 轮：查看自愈恢复后的总结

**用户输入：**
```
刚收到通知，server-001 的 CPU 告警已经自动恢复了。
但我看到这个告警这周已经出现了好几次，是不是自愈只是临时解决了？
```

**期望 AI 行为：**
- 分析自愈恢复后的状态
- 查询自愈触发历史（同一规则/同一资产近期告警次数）
- 分析自愈前日志（查找 nginx error log 中的根本原因）
- 判断自愈是否真正解决根因
- 给出结论

**期望输出示例：**
```
自愈记录（近 7 天）：
| 日期     | 时间     | 触发告警         | 自愈动作      |
|----------|---------|------------------|--------------|
| 07-11    | 10:30   | CPU 92%          | restart nginx |
| 07-13    | 16:20   | CPU 94%          | restart nginx |
| 07-15    | 09:15   | CPU 91%          | restart nginx |
| 07-17    | 14:00   | CPU 95%          | restart nginx |

nginx error log 每次自愈前都有相同模式：
[emerg] 768 worker_connections are not enough while connecting to upstream

🔍 判断：自愈（重启 nginx）只是临时释放了连接，没有解决根因
- worker_connections 设置为 1024，实际高峰期需要 1500+
- 每次重启后连接数清零，但业务量上来后又迅速填满
- 这是典型的"重启掩盖症状"

🛠 建议永久修复方案：
1. 立即：将 nginx worker_connections 从 1024 提升到 2048
2. 短期：检查 nginx upstream 是否有慢响应导致连接长期占用
3. 长期：评估是否需要扩容 nginx 实例数
```

---

#### 第 2 轮：用户要求永久修复

**用户输入：**
```
分析得很清楚。那就按你说的，把 nginx worker_connections 改到 2048，
同时把自愈流程也优化一下。
```

**期望 AI 行为：**
- 分步提议：修改配置 + 更新自愈流程
- 展示配置变更的执行计划

**期望输出示例：**
```
变更 #1：修改 nginx worker_connections 1024 → 2048
执行步骤：
  1. 修改 /etc/nginx/nginx.conf
  2. nginx -t 验证配置语法
  3. nginx -s reload 热加载

变更 #2：优化自愈流程
当前：检测 CPU > 90% → restart nginx
优化后：
  检测 CPU > 90%
    → 检查 nginx error log 是否有 worker_connections exceed
      → 如果没有：restart nginx（当前逻辑）
      → 如果有：扩大 worker_connections 到当前值 ×2 + reload
    → 等待 30 秒
    → 验证 CPU 是否下降
```

## 测试场景 21：跨多云环境排障（复杂）

### 场景目标

验证 AI 能处理跨多个云平台（阿里云 + AWS + 自建机房）的混合云环境排障，识别跨云网络延迟、云厂商故障等复杂场景。

### 测试数据准备

```
拓扑：
  用户 → CDN → 阿里云（杭州）api-gateway
                   → 阿里云（杭州）app-service
                   → AWS（新加坡）payment-service
                   → 自建机房 database-01

问题表现：
- 杭州用户反馈"支付页面加载慢"（P95 5.8s）
- 新加坡用户反馈"支付正常"（P95 1.2s）
- 阿里云到 AWS 的专线延迟从 25ms 突增到 380ms
- AWS payment-service CPU 和内存正常
- database-01 正常
```

### 多轮对话测试步骤

#### 第 1 轮：跨区域调用分析

**用户输入：**
```
杭州用户说支付很慢，但新加坡用户说正常。
payment 服务部署在新加坡 AWS，杭州的 api-gateway 调新加坡的 payment。
帮我看看是不是跨云网络有问题？
```

**期望 AI 行为：**
- 查询跨区域调用延迟数据
- 对比杭州 vs 新加坡本地访问的差异
- 分析网络链路各跳的延迟分布
- 识别跨云专线是否为瓶颈
- 给出建议

**期望输出示例：**
```
🌐 跨区域延迟分析：
| 链路                      | P95 正常   | P95 当前   | 变化  |
|---------------------------|-----------|-----------|-------|
| 杭州用户 → 阿里云杭州     | 25ms      | 35ms      | 正常  |
| 阿里云杭州 → AWS 新加坡   | 25ms      | 380ms     | +14倍 |
| AWS 新加坡 → 自建数据库   | 8ms       | 10ms      | 正常  |

🎯 根因定位：阿里云 → AWS 跨云专线延迟突增（25ms → 380ms）
- 杭州用户的请求 90% 时间卡在跨云链路上
- 新加坡用户不经过跨云链路，所以正常

🛠 建议措施：
1. 立即：启用 payment 杭州本地缓存（减少跨云调用频率）
2. 短期：联系云厂商确认专线状态
3. 长期：在阿里云杭州部署 payment 服务副本
```

---

#### 第 2 轮：云厂商故障确认

**用户输入：**
```
联系云厂商了，他们说今天亚太区域有光缆故障，正在修复中。
那我能不能临时把新加坡的流量切到杭州来？
```

**期望 AI 行为：**
- 确认云厂商故障是已知原因
- 评估临时方案的可行性和风险
- 说明临时方案的局限性

**期望输出示例：**
```
临时降级方案评估：
在杭州 api-gateway 启用 payment 结果缓存 60 秒：
- 效果：60% 请求可缓存（商品价格/库存查询）
- 风险：实时支付结果延迟 60 秒
- 如果客户能接受"最终一致性"，这个方案可用

影响：
- 支付功能不会中断
- 实时性降低（60 秒缓存）
- 光缆恢复后立即切回实时模式
```

## 测试场景 22：安全事件应急响应（复杂）

### 场景目标

验证 AI 在收到安全类告警时，能做出正确的安全应急响应——取证、隔离、溯源。安全事件与普通运维故障的处理方式完全不同，AI 需要区分两者并采取不同的响应策略。

### 测试数据准备

```
安全告警信息：
- 告警内容：server-003 检测到异常外连流量，目标 IP 45.33.32.156（已知恶意 IP）
- 时间：14:23
- 源端口：随机高端口
- 流量特征：每 5 分钟发送 1KB 数据包
- 相关指标：server-003 CPU 85%（异常高），/tmp/ 目录下发现可疑文件 .systemd-update
- 登录记录：14:15 有来自 185.220.101.xxx（Tor 出口节点）的 SSH 登录成功
- 资产：server-003（Web 服务器）
```

### 多轮对话测试步骤

#### 第 1 轮：安全告警响应

**用户输入：**
```
收到一个安全告警，server-003 有异常外连流量，目标 IP 45.33.32.156 已知是恶意 IP。
现在该怎么办？要不要直接隔离这台服务器？
```

**期望 AI 行为：**
- 立即确认安全事件类型（C2 通信/数据外泄）
- 区分安全事件与普通运维故障：安全事件用"取证隔离"而非"修"
- 给出分级的应急响应步骤（隔离 → 取证 → 溯源 → 清除）
- 不执行普通的"重启服务"等运维操作（可能破坏证据）

**期望输出示例：**
```
🚨 确认安全事件，级别：高危

安全事件特征：
1. C2 通信：外连已知恶意 IP（45.33.32.156）— 命令控制服务器
2. 入侵入口：14:15 来自 Tor 出口节点的 SSH 登录成功
3. 后门程序：/tmp/.systemd-update 可疑文件
4. CPU 异常：85%（可能是挖矿或扫描活动）

🛑 响应步骤（按顺序执行）：

步骤 1 - 立即隔离
  对 server-003 执行网络隔离（切断网络，保留 SSH 供取证）
  不要重启或关机（会丢失内存证据）

步骤 2 - 内存取证
  导出可疑文件、检查进程列表、网络连接、计划任务

步骤 3 - 溯源分析
  检查 SSH 登录记录、authorized_keys、lastlog

步骤 4 - 清除与恢复
  清除后门、轮换凭证、从快照重建

建议先执行步骤 1（隔离），我同时开始取证分析。
```

---

#### 第 2 轮：取证分析

**用户输入：**
```
server-003 已经隔离了，网络已切断，保留 SSH 可以访问。
现在帮我分析一下那个可疑文件和登录记录。
```

**期望 AI 行为：**

**期望输出示例：**

```
取证结果：

  → 连接 XMR 矿池

入侵时间线：
14:15 — SSH 登录（来自 Tor 节点）

入侵判定：挖矿木马（非勒索/非数据窃取）
影响范围：仅 server-003

清除步骤：
1. 杀掉挖矿进程
2. 删除可疑文件
```

#### 第 3 轮：知识沉淀与复盘

**用户输入：**

## 验收标准补充

### 夜间无人值守场景（场景 18）
- [ ] AI 能识别告警的规律性模式（周期性 vs 一次性）
- [ ] 区分"系统性故障"与"瞬态抖动"，给出冷静判断
- [ ] 建议告警规则优化（持续时间过滤/时间段过滤）
- [ ] 提供定时任务的性能优化建议

### 知识遗忘场景（场景 19）
- [ ] AI 能检查知识条目的创建/更新时间，识别过时知识
- [ ] 对比当前系统版本与知识适用的版本
- [ ] 给出逐步调整的安全建议（而非一刀切）

### 自愈闭环验证场景（场景 20）
- [ ] AI 能识别"自愈只是掩盖了症状"，分析自愈历史
- [ ] 从自愈前日志中识别根因（连接数超限而非进程挂死）
- [ ] 给出永久修复方案（修改配置而非重启）

### 跨多云排障场景（场景 21）
- [ ] AI 能识别跨云网络延迟问题
- [ ] 区分本地 vs 跨区域调用的性能差异
- [ ] 给出降级方案评估（缓存/最终一致性）

### 安全事件响应场景（场景 22）
- [ ] AI 能区分安全事件与普通运维故障
- [ ] 采取"取证隔离"而非"重启修复"的方式
- [ ] 给出分级应急响应步骤（隔离/取证/溯源/清除）
- [ ] 进行 Why-5 根因分析，给出安全加固建议
- [ ] 生成安全事件类型的知识沉淀
