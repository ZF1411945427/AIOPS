# AIOps 用户操作手册 — 链路2-4：K8s指标接入 & 事件告警 & 移动端推送

> **适用角色**：运维工程师、SRE、K8s 管理员
> **前置条件**：已登录系统，拥有「运维工作台」菜单权限，有可用的 K8s 集群
> **预计耗时**：约 10-15 分钟

---

## 概述

本手册覆盖三条链路：
- **链路2**：K8s 指标接入（资源容量 + 事件采集）+ HPA 推荐 + 资源优化
- **链路3**：K8s 事件接入告警
- **链路4**：移动端告警推送与控制

> **链路2 K8s指标**　①配置集群 → ②资源查询 → ③指标采集(后台) → ④HPA推荐 → ⑤资源优化
> **链路3 K8s事件**　⑥事件→告警(后台)
> **链路4 移动端**　⑦WS连接 → ⑧查看告警 → ⑨确认告警 → ⑩静默告警 → ⑪解除
>

# 链路2：K8s 指标接入

## 第一步：配置 K8s 集群

### 操作页面
**侧边栏** → **运维工作台** → **K8s 资源** → **K8s 集群管理**

### 操作步骤

2. **填写集群信息**

   - **认证方式**：选择「Token」或「kubeconfig」
   - **Token**：粘贴 ServiceAccount Token
   - **命名空间**：输入 `prod,staging`（逗号分隔，或留空表示全部）

3. **点击「保存」**
   - 系统验证连接成功后，集群出现在列表中

### 看到什么
- 集群列表显示新添加的集群，状态为「已连接」
- 集群详情显示：节点数、Pod 数、命名空间数

## 第三步：K8s 指标采集（后台自动）

### 说明
K8s 指标采集通过 `datasource_service._scrape_kubernetes()` 执行，采集两类数据：

### 采集内容

**1. 资源容量指标**（真实采集）

| 指标名 | 说明 | 来源 |
|--------|------|------|
| `node_cpu_capacity` | Node CPU 核数 | K8s Node.Status.Capacity.cpu |
| `node_memory_capacity` | Node 内存字节数 | K8s Node.Status.Capacity.memory |
| `pod_restarts` | Pod 重启次数 | K8s Pod.Status.ContainerStatuses[].RestartCount |
| `pod_containers` | Pod 容器数量 | K8s Pod.Spec.Containers[].Length |
| `deployment_replicas` | Deployment 副本数 | K8s Deployment.Status.Replicas |
| `deployment_available` | Deployment 可用副本数 | K8s Deployment.Status.AvailableReplicas |

**2. K8s 事件采集**（真实采集）

- 调用 `v1.list_event_for_all_namespaces()` 获取全部 Event
- 过滤 `Warning`/`Error` 类型事件
- 写入 `K8sEvent` 表（cluster/namespace/name/kind/reason/message/severity/count）
- **Severity 自动分级**：
  - 包含 `OOMKilling`/`CrashLoopBackOff`/`NodeNotReady` → `critical`
  - 包含 `Warning`/`Failed`/`Evicted` → `warning`
  - 其他 → `info`

### 看到什么
- 无需手动操作
- 采集的指标在「可观测性」→「指标监控」页面展示
- K8s 事件在「K8s 资源」→「事件列表」页面展示

### ⚠️ 注意事项
当前 K8s 指标采集**仅获取资源容量**（CPU 核数、内存字节数），**未采集运行时使用率**（CPU%、内存%）。如需获取运行时指标，需对接 Metrics Server 或 Prometheus Adapter。

# 链路2（续）：HPA 推荐与资源优化

## 第四步：查看 HPA 推荐

### 操作页面
**侧边栏** → **运维工作台** → **K8s 资源** → **HPA 推荐**

### 说明
HPA 推荐功能基于 Metrics Server 采集的 Pod 实际 CPU/内存使用率，自动计算推荐的 HPA 配置参数，帮助优化 Pod 弹性伸缩策略。

### 操作步骤

1. **选择集群**
   - 页面顶部下拉选择 K8s 集群
   - 默认可选集群：所有已配置的 K8s 集群

2. **查看 HPA 推荐列表**
   - 页面显示可进行 HPA 的 Deployment 列表
   - 每条记录显示：
     - **命名空间**：Deployment 所在的 namespace
     - **Deployment 名称**
     - **当前副本数**：实际运行副本数
     - **CPU 使用率**：Metrics Server 采集的当前 CPU 使用量（mCore）
     - **内存使用率**：Metrics Server 采集的当前内存使用量（MiB）
     - **推荐最小副本**：基于当前使用率向上取整保留余量
     - **推荐最大副本**：min_replicas × 3（含突发余量）
     - **推荐 CPU 目标**：基于 P75 实际使用率 + 安全余量
     - **推荐内存目标**：基于 P75 实际使用率 + 安全余量

3. **复制 HPA 配置**
   - 点击某条推荐的「复制配置」按钮
   - 自动生成 YAML 格式的 HPA 配置
   - 内容示例：
     ```yaml
     apiVersion: autoscaling/v2
     kind: HorizontalPodAutoscaler
     metadata:
       name: my-app-hpa
       namespace: default
     spec:
       minReplicas: 2
       maxReplicas: 6
       metrics:
       - type: Resource
         resource:
           name: cpu
           target:
             type: Utilization
             averageUtilization: 75
       - type: Resource
         resource:
           name: memory
           target:
             type: Utilization
             averageUtilization: 80
     ```

### 看到什么
- 推荐列表按 namespace 分组展示
- 每条推荐显示当前值和建议值的对比
- YAML 配置可直接应用到 K8s 集群

## 第五步：查看资源优化建议

### 操作页面
**侧边栏** → **运维工作台** → **K8s 资源** → **资源优化**

### 说明
资源优化功能逐 Pod 分析容器资源 requests/limits 的配置合理性，识别超配（资源浪费）和欠配（限流风险）问题，并给出调整建议。

### 诊断规则

| 问题类型 | 诊断条件 | 严重级别 | 建议 |
|---------|---------|---------|------|
| **重度超配** | request > usage × 2 | warning | 将 requests 降至 usage × 1.2 |
| **中度超配** | request > usage × 1.5 | info | 考虑适当降低 requests |
| **内存紧平衡** | usage > request × 0.9 | warning | 增加 requests 避免 OOM |
| **CPU 紧平衡** | usage > request × 0.9 | info | 适当增加 CPU requests |
| **限流风险** | usage > limit × 0.85 | critical | 增加 limits 防止容器被限流 |
| **未设置 requests** | request 为空或 0 | warning | 建议设置合理的 requests |
| **未设置 limits** | limit 为空或 0 | info | 建议设置 limits 防止资源争抢 |
| **limit > request × 5** | limit 远大于 request | info | 缩小差距，避免突增影响 |

### 操作步骤

1. **选择集群**
   - 页面顶部下拉选择 K8s 集群

2. **查看优化建议列表**
   - 页面显示所有 Pod 的容器资源分析结果
   - 每条记录显示：
     - **命名空间 / Pod / 容器**
     - **CPU request / limit / 实际使用量**
     - **内存 request / limit / 实际使用量**
     - **问题类型**：重度超配 / 中度超配 / 限流风险 / 紧平衡 / 未设置
     - **严重级别**：critical / warning / info（带颜色徽章）
     - **诊断详情**：具体问题和调整建议

3. **筛选优化建议**
   - **严重级别筛选**：全部 / critical / warning / info
   - **问题类型筛选**：全部 / 超配 / 欠配 / 未设置

4. **按建议调整配置**
   - 查看诊断详情中的具体数值对比
   - 参考建议值修改 Deployment 的 resources 配置
   - 重新部署生效

### 看到什么
- 优化建议列表按严重级别排序（critical 优先）
- 每条建议显示当前值与推荐值的对比
- 超配问题显示可节省的资源量
- 欠配问题标注已触发的风险和阈值

### 结果
HPA 推荐闭环：Metrics Server 采集 → 使用率分析 → 推荐 HPA 配置 → 应用到集群。
资源优化闭环：容器配置分析 → 超配/欠配诊断 → 调整建议 → 优化部署。

# 链路3：K8s 事件接入告警

## 第四步：K8s 事件自动映射为告警

### 说明
K8s 事件告警是**后台定时任务**，无需手动操作。

### 映射过程

1. **定时触发**：后台任务 `alert_service.check_k8s_events()` 每轮循环执行
2. **查询 K8sEvent**：查 `K8sEvent` 表中 `severity IN ('warning', 'critical')` 的事件
3. **事件映射**：按 `_K8S_EVENT_ALERT_MAP` 映射表转换：
   - **Critical 事件**：OOMKilling、CrashLoopBackOff、NodeNotReady、ImagePullBackOff 等
   - **Warning 事件**：BackOff、FailedScheduling、Evicted、Unhealthy 等
4. **模糊匹配**：不在映射表中的事件，通过 reason 字符串包含 "oom"/"crash"/"fail"/"notready" 模糊判断
5. **去重检查**：检查是否存在同 `asset_id` + 同 `metric_name`（`k8s_event_{reason}`）+ 同时间窗口（30分钟）+ 同状态的已存在告警，有则跳过
6. **创建告警**：
   - `metric_name`：`k8s_event_{reason}`（如 `k8s_event_OOMKilling`）
   - `actual_value`：事件重复次数（`count`）
   - `severity`：事件严重级别
   - `message`：拼接 `reason + ":" + message + ",涉及" + kind + "/" + name`
   - `status`：`triggered`
7. **通知推送**：调用 `notification_service.notify_new_alerts()` 发送通知

### 看到什么
- 无需手动操作
- K8s 事件触发的告警在「告警列表」页面显示
- 告警标题格式：`OOMKilling: Pod xxx 因 OOM 被 Kill，涉及 Pod/xxx`

### 结果
K8s 事件自动映射为告警，可进入告警确认/故障单流程。

# 链路4：移动端告警推送与控制

## 第五步：移动端 WebSocket 连接

### 操作页面
**移动端** → **告警中心** → **告警列表**

### 连接过程

1. **打开移动端告警列表页**
   - 页面 `onLoad` 时自动建立 WebSocket 连接
   - 连接地址：`ws://host/api/ws/alerts?token=xxx`

2. **连接确认**
   - 连接成功后，服务端发送：`{"type": "connected", "msg": "告警频道已连接"}`
   - 移动端显示连接成功提示

3. **自动重连**
   - 断开后指数退避重试：3s → 6s → 12s → 最大 30s
   - 重连成功后自动恢复推送

### 看到什么
- 页面顶部显示「已连接」状态指示器
- 断开时显示「重连中...」提示

## 第七步：移动端确认告警

### 操作页面
**移动端** → **告警中心** → **告警列表** → **告警详情**

### 操作步骤

1. **单条确认**
   - 点击某条告警
   - 进入告警详情页
   - 点击「确认告警」按钮
   - **结果**：告警状态变为 `acknowledged`

2. **批量确认**
   - 在告警列表页，长按进入多选模式
   - 勾选多条告警（checkbox）
   - 底部工具栏显示「已选 X 条」
   - 点击「批量确认」按钮
   - **结果**：所有选中告警状态变为 `acknowledged`

3. **批量解决**
   - 在多选模式下，点击「批量解决」按钮
   - 弹出二次确认对话框
   - 点击「确认」
   - **结果**：所有选中告警状态变为 `resolved`

### 看到什么
- 单条确认：详情页状态从「待处理」变为「已确认」
- 批量确认：列表页选中告警状态批量更新
- 操作成功后显示 Toast 提示「确认成功」/「解决成功」

### 结果
告警已确认/解决，状态实时同步到后端数据库。

## 第九步：移动端静默告警（受限）

### 操作页面
**移动端** → **告警中心** → **告警列表** → **告警详情**

### 说明
⚠️ **移动端静默功能当前未实现**，点击「静默」按钮会提示：
> "移动端暂不支持直接静默，请通过 AI 助手或 Web 端执行静默操作"

### 替代方案

**方案 A：通过 Web 端静默**
1. 在 Web 浏览器打开系统
2. 进入「告警列表」→ 点击告警 → 「静默告警」
3. 填写静默时长和原因 → 提交
4. 后端调用 `POST /alerts/{alert_id}/silence` 写入 `AlertSilence` 记录

**方案 B：通过 AI 助手静默**
1. 在移动端打开 AI 助手
2. 输入 "请静默告警 {alert_id}，静默 1 小时，原因是维护窗口"
3. AI 助手调用 `execute_silence_alert` 工具执行静默

## 常见问题

### Q1：K8s 集群连接失败？
- **检查**：API Server 地址是否正确
- **检查**：Token 是否有效（是否过期）
- **检查**：网络是否连通（能否访问 API Server）

### Q2：K8s 事件没有生成告警？
- **检查**：`K8sEvent` 表是否有 Warning/Error 事件
- **检查**：`check_k8s_events()` 是否在后台循环中执行
- **检查**：是否有重复告警（30 分钟去重窗口）

### Q3：移动端 WebSocket 连接失败？
- **检查**：Token 是否有效
- **检查**：网络是否连通（能否访问后端 API）
- **检查**：防火墙是否放行 WebSocket 端口

### Q5：HPA 推荐显示为空？
- **检查**：集群是否配置了 Metrics Server（HPA 推荐依赖 `/apis/metrics.k8s.io/v1beta1`）
- **检查**：集群中是否有可用的 Deployment 资源
- **检查**：Metrics Server 是否采集到了 Pod 的 CPU/内存使用数据

### Q6：资源优化建议显示「未设置 requests/limits」？
- **检查**：Pod 的容器配置中 `resources.requests` / `resources.limits` 是否填写
- **建议**：始终为关键业务容器设置 requests 和 limits，避免资源争抢

### Q4：移动端批量操作失败？
- **检查**：是否进入多选模式（长按触发）
- **检查**：是否勾选了告警（checkbox 选中）
- **检查**：网络是否稳定

## 专业术语

| 术语 | 英文 | 说明 |
|------|------|------|
| K8s Metrics Server | Kubernetes Metrics Server | K8s 内置指标采集组件，提供 Pod/Node 的 CPU/Memory 实时使用率 |
| Resource Capacity | 资源容量 | 资源总量（如 CPU 核数、内存字节数） |
| Resource Utilization | 资源利用率 | 实际使用占比（如 CPU 使用率 80%） |
| Event-driven Alerting | 事件驱动告警 | 基于 K8s Event 流实时生成告警 |
| Idempotent Deduplication | 幂等去重 | 确保相同条件的事件/告警不会重复创建 |
| WebSocket Heartbeat | WebSocket 心跳 | 保持长连接的保活机制 |
| Exponential Backoff | 指数退避 | 断线重连时的等待时间策略 |

| Metrics Server | K8s 指标采集器 | 采集 Pod/Node 的 CPU 和内存实时使用率 |
| Resource Overallocation | 资源超配 | requests 远高于实际使用量，造成资源浪费 |
| Resource Underallocation | 资源欠配 | 实际使用量接近/超过 requests 或 limits，有 OOM/限流风险 |
