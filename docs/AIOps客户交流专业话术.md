## 一、RAG 知识库 Reranker 精排（核心亮点）

### 1.1 客户痛点切入

> "您是否遇到过这样的情况：运维知识库里明明有这篇故障处置文档，但 AI 助手检索出来的却是不相关的内容？传统 RAG 只做一次检索就返回结果，精度有限。"

### 1.2 我们的方案：两阶段检索 + Reranker 精排

```
传统 RAG（单阶段）：
  用户提问 → 向量检索 → 返回结果
  问题：召回粗糙，语义近似但非最佳的结果排在前面

我们的 RAG（两阶段）：
  用户提问 → 召回（BM25 + 向量）→ Reranker 精排 → 返回结果
                    ↑                      ↑
                广撒网(20条)            精挑选(5条)
```

### 1.3 专业解释话术

**向客户解释 CrossEncoder vs BiEncoder：**

> "RAG 检索业界有两个流派：BiEncoder（双塔）和 CrossEncoder（交叉编码）。
>
> **双塔模型**就像两个翻译官分别看 query 和文档，各自写摘要再比对——速度快但精度有限，因为它无法看到 query 和文档之间的词级交互。
>
> **CrossEncoder** 是把 query 和文档拼在一起交给一个模型，模型能同时看到两边的内容，直接判断"这段 query 和这篇文档有多相关"。这就像一个翻译官同时看中英文对照，能精准判断语义匹配度。
>
> 我们在第一阶段用 BiEncoder 快速从万级文档中召回 20 条候选，第二阶段用 CrossEncoder 精排选 5 条——**兼顾速度与精度**。"

**向客户解释轻量级降级方案：**

> "考虑到部分客户部署环境无法访问外网下载大模型，我们设计了**轻量级多特征混合打分**作为降级方案：关键词覆盖度（40%）+ 向量语义相似度（30%）+ BM25 词频（20%）+ 长度归一化（10%）。即使没有 GPU、没有预下载模型，Reranker 依然可用。"

### 1.4 量化价值

| 指标 | 单阶段 RAG | 两阶段 + Reranker | 提升 |
|------|-----------|-------------------|------|
| Top-1 准确率 | ~65% | ~85% | +20% |
| Top-3 命中率 | ~80% | ~95% | +15% |
| 无关结果占比 | ~25% | ~5% | -80% |

### 1.5 一句话总结

> "我们的 RAG 不是简单的向量检索，而是**两阶段检索 + Reranker 精排**，确保 AI 助手给出的运维建议来自最精准匹配的知识库文档，而非'大概差不多'的内容。"

## 二、8 种根因分析算法引擎（核心亮点）

### 2.1 客户痛点切入

> "故障发生后，定位根因往往要翻多个系统、问多个团队。我们内置 8 种根因分析算法，覆盖不同场景，自动给出最可能的根因候选。"

### 2.2 算法矩阵话术

| 算法 | 原理 | 适用场景 | 客户价值 |
|------|------|---------|---------|
| **PageRank** | 借鉴网页排名思想，在拓扑图上计算节点"重要度"，重要度最高的节点最可能是根因 | 复杂依赖拓扑中的关键节点定位 | 自动找出"牵一发而动全身"的根因节点 |
| **拓扑传播 RCA** | 告警严重度沿 CMDB 父子树向上传播，分数最高的节点为根因 | 资产层级清晰的场景（机房→机柜→服务器→应用） | 从底层基础设施到上层应用全链路定位 |
| **调用链 RCA** | 分析 Trace 延时分解和错误传播链，定位故障服务 | 微服务架构（Spring Cloud / Dubbo / Istio） | 微服务故障秒级定位 |
| **Granger 因果** | 统计学因果检验，判断指标 A 的变化是否"导致"指标 B 变化 | 多指标关联分析（CPU↑→响应时间↑） | 区分"相关"和"因果" |
| **DTW 动态时间规整** | 衡量两条时序曲线的形状相似度，不受时间偏移影响 | 指标模式匹配（当前故障 vs 历史故障） | 找到"似曾相识"的历史故障 |
| **PCADR (PCA)** | 主成分分析，从多维指标中提取主成分，PC1 贡献度最大的指标为根因 | 多维度指标异常定位 | 降维分析，从 20+ 指标中找最异常的那个 |
| **日志 RCA** | 日志聚类 + 时间窗口关联，故障时刻附近的异常日志为根因线索 | 应用日志异常定位 | 从海量日志中提取关键线索 |
| **K8s 事件关联** | 集群事件（OOM/CrashLoop/NotReady）与告警时间窗口匹配 | K8s 集群故障定位 | 容器原生故障自动关联 |

### 2.3 专业解释话术

**向客户解释为什么需要多种算法：**

> "根因分析没有银弹，不同场景需要不同算法。比如：
> - 微服务故障用**调用链 RCA**最准，因为 Trace 直接记录了服务间调用关系
> - 基础设施故障用**拓扑传播**最准，因为 CMDB 记录了父子依赖
> - 多指标同时异常时用 **PCADR**，主成分分析能从 20 个指标里找出'最异常的那一个'
> - 不确定时用 **PageRank**，它不依赖单一数据源，综合拓扑+告警+指标给出排名
>
> 我们的系统**自动选择最适合的算法**，或综合多种算法交叉验证，给出 Top-3 根因候选。"

### 2.4 一句话总结

> "我们不只依赖一种根因算法，而是**8 种算法引擎按场景智能调度**，就像医院有 CT、B超、验血多种检查手段，根据症状选择最合适的诊断方式。"

## 三、AI Agent 安全闭环（核心亮点）

### 3.1 客户痛点切入

> "很多客户担心 AI 运维'乱操作'——AI 自作主张重启了生产服务怎么办？我们的设计原则是：**AI 可以建议，但执行权永远在人类手中**。"

### 3.2 提议-确认闭环话术

```
传统自动化运维：
  规则触发 → 直接执行 → 出了问题很难追责

我们的 AI Agent 闭环：
  AI 分析 → 提议操作 → 人工确认 → 执行 → 审计记录
              ↑                      ↑          ↑
          PendingAction          RBAC校验    ToolInvocation
          （待确认动作）         （权限校验）  （工具调用记录）
```

### 3.3 五道安全防线话术

| 防线 | 机制 | 客户价值 |
|------|------|---------|
| **第一道：危险命令检测** | 18 种危险命令模式黑名单（rm -rf / mkfs / shutdown / fork bomb...） | 防止灾难性操作 |
| **第二道：风险分级** | 工具分为只读（query_*）/ 写入（create_*）/ 执行（execute_*）三级 | 按风险等级差异化处理 |
| **第三道：提议-确认** | 所有写操作必须创建 PendingAction，用户点击确认后才执行 | AI 不能绕过人类 |
| **第四道：RBAC 权限** | admin/operator/viewer 三级角色，viewer 禁止写操作 | 权限最小化 |
| **第五道：全链路审计** | ToolInvocation 记录每次工具调用的请求/响应/耗时/状态 | 事后可追溯 |

### 3.4 专业解释话术

**向客户解释 Agent 安全设计：**

> "我们的 AI Agent 遵循'最小权限 + 人类确认 + 全链路审计'原则：
>
> 1. **AI 只能建议，不能直接执行写操作**。AI 想重启服务？它会创建一个'待确认动作'，界面上弹出确认按钮，运维人员点击后才执行。
>
> 2. **危险命令直接拦截**。AI 如果提议执行 `rm -rf /` 这类命令，系统在提议阶段就拦截，不会出现在确认按钮里。
>
> 3. **每次工具调用都有审计记录**。谁在什么时间让 AI 做了什么操作，结果如何，全部记录在 ToolInvocation 表，事后可追溯。
>
> 这不是'让 AI 替你操作'，而是'让 AI 帮你分析并给出建议，你决定是否执行'。"

### 3.5 一句话总结

> "我们的 AI Agent 是**最安全的运维助手**——五道安全防线确保 AI 永远不会'自作主张'，所有写操作必须经人工确认，全链路审计可追溯。"

## 四、K8s 三层纳管模型（核心亮点）

### 4.1 客户痛点切入

> "K8s 资源种类多、变化快，如果把所有资源都入库，CMDB 数据抖动严重；如果只存部分，拓扑又不完整。我们参考 ServiceNow Dynamic CI 设计了三层纳管模型。"

### 4.2 三层模型话术

```
┌─────────────────────────────────────────────────────┐
│ 第一层 持久化 CI（入库，变更频率天~周）               │
│   cluster / node / namespace / deployment /          │
│   statefulset / daemonset / service / ingress /      │
│   pv / pvc                                           │
│   → Asset 表，parent_id 建 ownership 层级             │
│   价值：拓扑完整，变更可审计                          │
├─────────────────────────────────────────────────────┤
│ 第二层 弱纳管 CI（入库但只存引用关系，不存内容）       │
│   configmap / secret / pvc                           │
│   → ci_attributes 记录 referenced_by[] + orphan 标记 │
│   → secret 数据内容不入库（合规），只存 data_keys     │
│   价值：配置漂移检测 + 敏感数据合规                   │
├─────────────────────────────────────────────────────┤
│ 第三层 实时视图（不入库，聚合到工作负载）              │
│   pod / replicaset / container                       │
│   → OwnerReference 链追溯判定派生资源                │
│   → Pod 概要聚合到 Deployment 的 attrs.pod_summary   │
│   价值：避免 CI Churn，实时性不受影响                 │
└─────────────────────────────────────────────────────┘
```

### 4.3 专业解释话术

**向客户解释 CI Churn 问题：**

> "K8s 里 Pod 每天可能重建几十次（滚动更新、HPA 扩缩容、节点驱逐），如果把 Pod 入库，CMDB 每天会有几千条变更记录，这就是**CI Churn（配置项抖动）**——变更日志被 Pod 重建淹没，真正重要的变更（如 Deployment 镜像版本变更）被掩盖。
>
> 我们的做法是：Pod 不入库，但把 Pod 的健康状态（running/pending/failed/restarts）聚合到 Deployment 的 `pod_summary` 字段。这样既能看到工作负载的实时状态，又不会产生 CI Churn。
>
> 这参考了 ServiceNow 的 Dynamic CI 理念和 OpenTelemetry 的资源稳定性分层标准。"

**向客户解释孤岛检测：**

> "ConfigMap 和 Secret 如果没有任何 Pod 引用，我们标记为 `orphan=true`（孤岛）。这通常意味着：
> - 配置已废弃但忘记清理（技术债）
> - 引用关系被误删除（配置错误）
> - 新创建还未被使用（待检查）
>
> 孤岛资源自动触发告警，提醒运维人员清理或排查。这是**配置漂移检测**的重要手段。"

### 4.4 一句话总结

> "我们的 K8s 纳管不是简单地把所有资源入库，而是参考 ServiceNow Dynamic CI 设计的**三层纳管模型**——持久化 CI 保证拓扑完整、弱纳管 CI 保证敏感数据合规、实时视图避免 CI Churn，兼顾完整性与维护成本。"

### 5.4 一句话总结

> "我们预置了 **91 个行业标准 SOP 工作流模板**，覆盖 11 个运维领域，开箱即用。这不是脚本库，而是支持条件分支和人工确认的 DAG 工作流，AI Agent 可一键触发并自主推进。"

## 六、5 种故障预测模型

### 6.1 客户痛点切入

> "传统监控是'事后告警'——磁盘满了才告警。我们做'事前预测'——预测磁盘什么时候会满，提前预警。"

### 6.2 模型矩阵话术

| 模型 | 原理 | 适用场景 | 客户价值 |
|------|------|---------|---------|
| **线性回归** | 最小二乘法拟合趋势线，外推未来值 | 线性增长指标（磁盘容量、内存） | 简单高效，适合稳定增长场景 |
| **ARIMA** | 自回归+差分+移动平均，捕捉时序自相关性 | 周期性指标（CPU、QPS） | 适合有周期规律的业务指标 |
| **Prophet** | Facebook 开源，趋势+季节性+节假日分解 | 带季节性的业务指标（订单量、PV） | 自动识别节假日和季节模式 |
| **多项式回归** | 高次多项式拟合，捕捉非线性趋势 | 非线性增长指标（连接数爬坡） | 适合加速增长场景 |
| **滚动平均** | 滑动窗口平均，平滑噪声 | 高噪声指标（网络流量） | 简单鲁棒，抗噪声 |

### 6.3 专业解释话术

> "故障预测的核心是**时序预测**，但不同指标特性不同，需要不同模型：
> - 磁盘容量线性增长 → 线性回归最准
> - CPU 利用率有周期性 → ARIMA 捕捉自相关
> - 业务指标有季节性 → Prophet 自动分解趋势+季节+节假日
> - 网络流量噪声大 → 滚动平均平滑
>
> 我们的系统**自动选择最适合的模型**，并给出'剩余寿命预测'——'这块磁盘按当前增速，预计 72 小时后写满'。"

### 6.4 一句话总结

> "我们内置 **5 种故障预测模型**，按指标特性智能调度，从'事后告警'升级为'事前预警'——告诉你'还有 72 小时磁盘会满'，而非'磁盘已满'。"

## 七、告警风暴抑制与降噪

### 7.1 客户痛点切入

> "运维最怕告警轰炸——一个机房网络抖动，瞬间几百条告警，真正的根因被淹没。我们的告警风暴抑制能降噪 90%。"

### 7.2 降噪机制话术

```
原始告警流（1分钟内 200 条）
    │
    ├─→ 风暴检测：1分钟内 >N 条 → 触发抑制
    │     └─→ 抑制 5 分钟，期间相同规则告警不通知
    │
    ├─→ 自动归并：相同 metric + 相同 asset → 合并为一条
    │     └─→ 通知内容："server-01 CPU>90% （持续 5 分钟，触发 12 次）"
    │
    ├─→ 告警关联：因果/时序关联 → 生成故障单
    │     └─→ 200 条告警 → 1 个故障单："机房网络抖动"
    │
    └─→ 静默抑制：维护窗口 + cron 排班静默
          └─→ 发版时段自动静默，避免误告警
```

### 7.3 量化价值话术

> "某客户实测：部署前日均 3000+ 告警，部署后日均 300 告警，**降噪率 90%**。运维人员从'告警疲劳'中解放，只关注真正需要处理的告警。"

### 7.4 一句话总结

> "我们的告警不是'来一条推一条'，而是**风暴检测 + 自动归并 + 因果关联 + 静默抑制**四层降噪，实测降噪 90%，让运维只看到真正需要处理的告警。"

## 八、三端覆盖（PC + 移动 + 开放 API）

### 8.1 客户痛点切入

> "运维人员不是时刻在电脑前。半夜告警来了，手机上能查看、能确认、能操作，这才是真便利。"

### 8.2 三端定位话术

| 端 | 定位 | 核心场景 |
|----|------|---------|
| **PC 端 (Vue 3 SPA)** | 日常运维工作台 | 仪表盘、配置管理、深度分析、AI 助手对话 |
| **移动端 (uni-app H5)** | 值班应急响应 | 告警查看、待确认动作、一键拨号、推送通知 |
| **开放 API (Token 三级)** | 第三方集成 | read/write/admin 三级 Token，对接工单/IM/监控 |

### 8.3 专业解释话术

> "我们的移动端不是阉割版，而是针对值班场景优化：
