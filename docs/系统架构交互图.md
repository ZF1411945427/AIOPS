# AIOps 数据流链路一览

## 链路 1：核心数据流（观测→检测→告警→诊断→知识→AI→自愈→SRE）

资产自动发现页 ─(执行 DiscoverySchedule 的 protocol(ssh/tcp/icmp) 探测目标范围，target_range 支持逗号分隔 IP、短横线范围(192.168.1.1-254)、CIDR 占位，port 指定探测端口，credential_id 关联认证凭据，每条探测结果写入 DiscoveryResult(ip/port/status/hostname/os_type/services) 并同步创建 Asset(name/type/ci_type/ip/parent_id/status=online/connection_type/connection_config JSON {ssh_port,ssh_user,ssh_password}/health_status=green))─▶ 指标采集页 ─(读取 Asset 表 status=online 的全部资产，遍历每条 Asset 的 connection_config JSON 解析 ssh_port/ssh_user/ssh_password，用 paramiko SSHClient 连接后逐条执行 COLLECT_COMMANDS 中 19 条采集命令：cpu_usage="top -bn1|awk..."、memory_usage="free -m|awk..."、disk_usage="df -h|awk..."、loadavg、swap、network_rx/tx、tcp_established/time_wait、process_count、zombie_process、open_files、uptime、ssh/http/mysql_connections，每条结果解析为 float 后写入 MetricRecord(asset_id/name/value/unit/timestamp) 并同时调 metric_v2_service.write_metrics_batch() 双写到 VictoriaMetrics，再调 health_engine 根据 Asset.ci_type 映射到三层(api/middleware/infra)：api 层 Span错误率>5% 或 p99>1000ms→health_status=red，中间件层有活跃告警→red 否则 green，基础设施层 cpu>90% 或 memory>90% 或 disk>85%→red，回写 Asset.health_status)─▶ 可观测性页 ─(读取 MetricRecord 按 name 分组展示趋势图/TOP-N，用户按 asset/时间范围/指标名下钻查看具体历史数据，读取 Asset.health_status 展示健康状态红绿灯，读取 Asset.ci_type/ip 展示资产列表)─▶ 异常检测页 ─(读取 AnomalyConfig 表 enabled=True 的全部配置(name/metric_name/asset_id/algorithm/sensitivity/window_size/period)，逐条查 MetricRecord 按 name=metric_name 取最新 window_size+5 条，若记录数<window_size 则跳过(数据不足)，按 algorithm 路由到 7 种算法：sigma 计算 Z-Score=|latest.value-mean(values)|/stddev(values)→Z>sensitivity(默认3.0)则触发、EWMA 指数加权移动平均→标准化残差>3.0触发、MAD 中位数绝对偏差→修正Z值>sensitivity触发、STL 季节性分解→残差Z-Score超限触发、Prophet FB预测→实际值超预测上限触发、LSTM 滑动窗口线性回归近似→残差超阈值触发、Transformer 简易自注意力→残差超阈值触发，先调 _has_recent_alert 查 Alert 表 status IN(triggered,acknowledged) 的同 metric+同资产+同算法记录，有则跳过(去重防风暴)，确认无重复后写入 Alert(rule_id/asset_id/metric_name/actual_value/threshold/severity 由偏差程度决定 warning 或 critical/status=triggered/message 含算法名/偏差方向/Z值/当前值/阈值/created_at)，调 notification_service.notify_new_alerts() 发送通知)─▶ 告警列表页 ─(Alert.status=triggered 按 severity(critical>warning>info) 排序展示，每条显示 rule_name/asset_name/metric_name/actual_value/threshold/severity/created_at，支持批量确认(→acknowledged)/静默/升级故障，后台 correlate_alerts 每轮查 Alert 表 status IN(triggered,acknowledged) 且 created_at>now-15min，按 asset_id 分组，每组取最高 severity(critical=0>warning=1>info=2) 作为故障级别，标题格式 [严重/警告/提示]资产名异常-N条告警，查同 asset_id 且 status=open 且 created_at>15min 的 Incident，存在则更新 title/severity/alert_count，不存在则新建 Incident(status=open/severity/alert_count)，增量插入 IncidentAlert(incident_id/alert_id) 不重复，扫描所有 open 故障单若关联活跃告警数为 0 则自动 Incident.status=resolved)─▶ 故障列表页 ─(Incident.status=open 和 pending_approval 展示列表，每条显示 title/severity/alert_count/created_at/asset_name/approver_id，用户点击进入详情查看关联告列表和 IncidentAlert 关联的全部 Alert 明细，点击"RCA分析"按钮触发 RCA)─▶ RCA 分析页 ─(analyze_incident(incident_id) 查 Incident → 通过 IncidentAlert 获取关联 Alert 列表 → 加载全部 AssetRelation(parent_id/child_id) 构建拓扑图，执行 score_asset 递归传播：每个资产的分 = 自身告警 severity 加权和(severity映射 critical=3/warning=2/info=1) + 子节点分*0.5(递归深度≤10)，按分排序最高=root_asset，再执行 PageRank 算法用 numpy 矩阵迭代计算概率转移矩阵输出 Top-N 候选根因，构建 Investigation Package 6部分：(1)Facts：告警列表 metric/severity/value/threshold + 影响资产 + severity分布 (2)Timeline：故障窗口起止/持续分钟/期间 AssetChangeLog 变更记录 (3)Candidate Causes：按置信度排序的候选根因(topology评分+关联告警数) (4)Evidence：关键指标 Top10/拓扑传播路径/资产详情(ci_type/ip/health_status) (5)Exclusions：已排除方向(RCA 评分低的末梢资产) (6)Next Steps：操作建议(检查根因/核对变更/验证依赖/触发自愈))─▶ 知识审批页 ─(用户点击"沉淀为知识" → 调 LLM 从 Incident + 关联 Alert 生成 JSON {title/ symptom/ root_cause/ solution/ sop_steps: [{step/action/command/expectation}]/ tags/ severity/ asset_type}，写入 KnowledgeDraft(title/symptom/root_cause/solution/tags/severity/asset_type/source_type=sop/status=pending/created_at)，审批页读取 KnowledgeDraft(status=pending) 列表展示 title/severity/tags/created_at，审批人点击通过 → approve_draft() 将 KnowledgeDraft 转为 KnowledgeBase(source_type=auto/created_at) 并写入 AlertKbLink(alert_id/kb_id) 关联原始告警，驳回 → KnowledgeDraft.status=rejected)─▶ RAG 引擎 ─(query_knowledge_rag(query/asset_type/severity/tags/top_k) 入参 → TF-IDF 向量化：中文按字分词 _TOKEN_RE=r'[\u4e00-\u9fa5]|[a-zA-Z0-9]+'，TF-IDF=(词频/总词数)*log((N+1)/(文档频率+1)+1)，余弦相似度=稀疏向量点积/(norm1*norm2)，遍历全部 KbChunk(chunk_id/document_id/document_title/source_type/content/tags/asset_type/severity/embedding)，threshold=0.05 过滤，Top-K 排序返回 items[{document_title/content/similarity/source_type/tags/asset_type/severity}]，支持 asset_type 精确匹配/severity 精确匹配/tags ilike 模糊匹配)─▶ AI 智能体对话页 ─(用户输入问题 → 系统提示DEFAULT_SYSTEM_PROMPT 引导"操作步骤优先query_runbook、知识优先query_knowledge_rag、执行命令返回propose_action" → LLM 推理返回 tool_calls 或文本 → 若为 tool_call：call_mcp_tool(mcp_registry) 路由到具体handler，read_only 级别(query_knowledge_rag/query_runbook/get_recommend_algo/list_asset_metrics/get_incident_detail)自动执行无需确认，normal 级别(propose_action/create_alert_silence)需用户前端确认 → 工具结果注入 runtime_context.messages 附加 tool_result → LLM 下一轮根据结果生成回答或继续调用，query_runbook 细节：入参 search/category/asset_type/limit → SQL 对 Runbook.title/symptom/diagnosis/steps/tags 做 ilike 模糊匹配 + category 精确过滤 + asset_type 精确过滤 → 返回 Runbook(id/title/category/symptom/diagnosis/steps/tags/severity/asset_type) → LLM 读取 steps 字段操作步骤，如需执行则调 propose_action(action_type/target/command) 写入 PendingAction(status=pending) 前端弹窗确认，用户确认后读取 Asset.connection_config SSH 远程执行命令，执行后 PendingAction(status=executed/result)，若为 AgentWorkflow 预置工作流则走 8 种节点(start/end/llm/knowledge/tool/condition/code/http)拓扑排序执行引擎)─▶ 自愈执行页 ─(check_and_remediate() 读取 AutoRemediation 表 enabled=true 的全部规则(name/rule_id/action_type/params JSON/created_at)，逐条查关联 rule_id 下 Alert 表 status=triggered 最多 5 条，去重检查同 alert+同规则在本次轮次已执行则跳过，按 alert.asset_id 查 Asset(ip/connection_config JSON{ssh_port,ssh_user,ssh_password})，SSH paramiko 远程执行 execute_action()：action_type=restart→"systemctl restart {params.service}"、clean→"find {params.path} -type f -mtime +{params.days} -delete"、script→"bash {params.script_path}"、run_command→执行 params.command 但先过危险命令黑名单(正则硬拦截 rm\s+-rf\s+/|mkfs|dd\s+if=|shutdown|reboot|fork炸弹|chmod\s+-R\s+[0-7]{3,4}\s+/ 等 9 种模式，匹配则记录警告不执行)、notify→发送站内通知、scale→K8s扩缩容暂未实现返回错误，写 RemediationLog(remediation_id/alert_id/action_type/target/success/output/created_at)，更新 Alert.status=acknowledged 且 message 追加标记"已触发自动响应"，调 remediation_effect_service.track_effect() 30 分钟后查 MetricRecord 同指标最新值对比阈值，若恢复正常→RemediationEffect(status=resolved)，若未恢复→RemediationEffect(status=failed) 升级人工)─▶ 效果追踪页 ─(读取 RemediationEffect 表展示每条自愈记录的执行时间/目标资产/动作类型/结果(resolved/failed)/验证指标名/验证值/阈值，统计自愈成功率=resolved/(resolved+failed)，按日/周/月聚合趋势，展示失败原因分布)─▶ SRE 可靠性页 ─(读取 SLO 配置按时间窗口计算可用率=成功请求数/总请求数，Error Budget=(1-SLO目标)*总请求数-失败请求数，Burn Rate=Error Budget消耗量/时间，展示 SLO Dashboard 仪表盘含当前可用率/Error Budget剩余/Burn Rate 趋势/On-Call 排班/升级策略，读取自愈成功率作为 SLO 的底层保障指标)

## 链路 2：K8s 指标接入

K8s 管理页 ─(读取 K8s 集群配置(kube_config/api_server/token/namespace_list)，遍历各 namespace 下的 Pod 和 Node 列表，对每个 Pod 调 K8s Metrics API 采集 CPU 使用率(Pod CPU request 与实际 usage 对比比例)/内存使用率(Pod Memory request 与实际 usage 对比比例)/网络收发字节数(Network rx/tx bytes)/磁盘 IO 读写速率，对每个 Node 采集 Node CPU 使用率/Node 内存使用率/Node 磁盘使用率/Node 网络流量，每条数据写入 MetricRecord(asset_id 对应 K8s 节点资产/name=k8s_cpu_usage_pod/k8s_memory_usage_pod/k8s_network_rx_bytes/k8s_network_tx_bytes/k8s_node_cpu_usage/k8s_node_memory_usage/k8s_node_disk_usage 等/value/unit=percent或bytes或bps/timestamp，双写 VictoriaMetrics，Asset.ci_type=pod/container/node，Asset.parent_id 关联集群根资产)─▶ 可观测性页 ─(读取 MetricRecord 中 name 前缀为 k8s_ 的指标，按 asset_type=pod/node/container 过滤展示 K8s 资源视图，按 namespace 分组展示 CPU/内存/网络的 TOP-N 趋势图，异常检测 anomaly_service 读取 AnomalyConfig(algorithm/sensitivity) 对 k8s_ 前缀指标做同 7 种算法检测，生成 Alert(metric_name=k8s_xxx/asset_id=PodID/severity) 进入告警链路)

## 链路 3：K8s 事件接入告警

K8s 管理页 ─(调 K8s API 轮询 namespaces 下 Events 列表(each event: involvedObject.kind/name/namespace, reason, message, type=Normal/Warning/Error, count, firstTimestamp, lastTimestamp, source.component)，过滤 type=Warning/Error 的事件，映射到 Alert：metric_name=k8s_event, actual_value=count(事件重复次数), severity=type=Error→critical/type=Warning→warning, message=拼接 reason+":"+message+",涉及"+involvedObject.kind+"/"+involvedObject.name, asset_id 根据 involvedObject.name 查 Asset 表中匹配的 k8s_pod/k8s_node 资产 ID，status=triggered, 调 _has_recent_alert 查同 asset+同 metric_name+k8s_event+同 reason 的未解决告警，有则 count+=1 不新建，防止 K8s 抖动产生大量重复告警)─▶ 告警列表页 ─(K8s 事件告警在告警列表中与指标告警并列展示，severity=critical 的 K8s 事件告警优先级高于普通指标告警，在 correlate_alerts 中参与按 asset_id 分组聚合，可升级为 Incident(title="[严重] K8s 事件: "+reason))

## 链路 4：移动端告警推送与控制

移动端页 ─(uni-app H5 前端启动时建立 WebSocket 连接到 ws://host/api/ws/alerts，发送 auth token 鉴权，服务端 webocket_handler 实时推送 Alert 表 status=triggered 或 acknowledged 的最新告警(按 order by created_at desc 取前 50 条 + 增量推送)，移动端按 severity(warning/critical/info) 分栏展示卡片列表，每条卡片显示 title/message/asset_name/created_at/severity，用户点击确认 → 调 API PUT /api/alerts/{id}/acknowledge → 后端查 Alert(id) 设 status=acknowledged 并写入 acknowledged_at，用户点击静默 → 调 API POST /api/alerts/{id}/silence → 后端写入 SilenceRule(alert_id/rule_id/start_time=now/end_time=now+24h/reason 用户输入/created_by=移动端) 设 Alert.status=silenced，用户点击告警详情 → 调 API GET /api/alerts/{id} 返回完整告警信息含关联资产详情和关联故障单 ID)─▶ 告警列表页 ─(移动端确认/静默操作回写后端数据库，Alert.status 实时更新，告警列表页下次刷新或 WebSocket 推送后同步展示最新状态)

## 链路 5：事件变更关联故障

事件变更页 ─(用户创建变更审批单→写入 ChangeRequest(id/title/description/asset_id/change_type(应用发布/配置变更/扩缩容/重启)/status(pending/approved/rejected/executed/rolled_back)/planned_start_time/planned_end_time/created_by/approver_id/created_at)，当 ChangeRequest.status=executed 或 rolled_back 时触发的 30 分钟内若有新创建的 Incident，后台自动在 Incident.relation_changes JSON 字段追加 {change_id/change_title/change_type/executed_at}，用户手动关联时 → 故障详情页点击"关联变更"弹窗选择变更单 → 调 API POST /api/incidents/{id}/link-change/{change_id} → 写入 IncidentChangeLink(incident_id/change_id) 关联表)─▶ 故障列表页 ─(故障详情展示关联的变更记录列表(变更标题/类型/执行时间/状态)，RCA 分析时读取该故障关联的全部 AssetChangeLog(asset_id/change_type/description/created_at) 纳入 Investigation Package.Timeline 部分，帮助判断故障是否由变更引发)

## 链路 6：知识指导 SRE 实践

知识管理页 ─(KnowledgeBase 表存储经过审批的正式知识，每条含 title/symptom/root_cause/solution/sop_steps JSON[{step,action,command,expectation}]/tags/severity/asset_type/source_type(auto/manual/alert_case/incident_case)，SRE 页初始化时按 tags 匹配 SLO 相关标签(slo/reliability/error_budget/availability) 或 asset_type 匹配当前资产类型调 API GET /api/knowledge?tags=slo&asset_type=server 获取相关条目的 solution 和 sop_steps 字段，SLO 配置页的"参考知识"面板展示这些条目的 title 和摘要，用户点击查看完整的 sop_steps 操作步骤，Error Budget 违规时在告警消息中附带推荐的知识条目链接)─▶ SRE 可靠性页 ─(SLO 配置面板引用知识库中的 reliability 分类条目展示实际可用率目标值与知识库建议值的对比，On-Call 排班页的"常见故障处理"模块读取 incident_case 类型知识的 solution 字段供值班人员快速参考)

## 链路 7：SLO 违规触发自愈

SRE 可靠性页 ─(SLO 计算引擎按时间窗口(滚动 7 天/28 天)定期计算：可用率=成功请求数/总请求数，Error Budget=(1-SLO目标)*总请求数-失败请求数，Burn Rate=Error Budget消耗量/时间窗口长度(按小时)，当 Error Budget 剩余 ≤0 → SLO 违规，或 Burn Rate > 预设阈值(如 2/小时持续 6 小时) → 提前预警，写入 Alert(metric_name=slo_burn_rate/slo_availability/slo_error_budget_remaining, severity=critical 表示违规/warning 表示提前预警, actual_value=当前可用率%/当前 Error Budget 剩余量, threshold=SLO 目标值/Error Budget 预警线, message="SLO违规: 可用率"+value+"% 低于目标"+target+"%", status=triggered, asset_id 关联该 SLO 对应服务的资产)─▶ 告警列表页 ─(该 alert 进入 correlate_alerts 15min 窗口按 asset_id 分组聚合，可新建或关联已有 Incident，AutoRemediation 配置 rule_id 匹配 slo_ 前缀指标的规则，action_type=scale(扩缩容) 或 restart(重启服务) 或 notify(升级处理)，执行后写 RemediationLog 更新 Alert.status=acknowledged)─▶ 自愈执行页 ─(SSH 远程执行或 K8s 扩缩容操作(scale 暂未实现返回错误)，30 分钟后 track_effect 查对应服务的 slo_ 指标最新值，若可用率恢复到 target 以上→RemediationEffect(status=resolved)，若未恢复→failed 升级值班处理)

## 链路 8：告警直接触发自动响应（不经过故障链路）

告警列表页 ─(后台 remediation_service.check_and_remediate() 定时任务每轮扫描 AutoRemediation 表 enabled=true 的全部规则(name/rule_id/action_type/params JSON{service,path,command,days,script_path}/enabled)，逐条取关联 rule_id 下 Alert 表 status=triggered 的最新 5 条，先做去重检查(本轮次已处理过的 alert_id+remediation_id 组合跳过)，按 alert.asset_id 查 Asset(ip/connection_config JSON{ssh_port:22, ssh_user:root, ssh_password:xxx})，用 paramiko SSHClient 调用 execute_action：action_type=restart→"systemctl restart {service}" 执行重启服务、clean→"find {path} -type f -mtime +{days} -delete" 清理过期文件、run_command→"{command}" 执行自定义命令(先正则匹配危险命令黑名单 9 种模式：rm -rf /、mkfs、dd if=、shutdown、reboot、:(){ :|:& };:  fork炸弹、chmod -R 777 /、> /dev/sda、mv / 等，匹配则跳过记录警告不执行)、scale→返回"暂未实现"错误、notify→调 notification_service.send_notification(user_id/type/message/title)，写 RemediationLog(remediation_id/alert_id/action_type/target=asset.ip/success=true或false/output=命令执行stdout+stderr/created_at)，更新 Alert.status=acknowledged 并在 Alert.message 字段追加"[自动响应]已执行 {action_type}，结果:{success}"，调 remediation_effect_service.track_effect(remediation_log_id) 启动 30 分钟定时器)─▶ 自愈执行页 ─(30 分钟后定时器触发 read_effect(asset_id/metric_name)，查 MetricRecord 表取该 asset+该指标的最新的 3 条记录平均值，对比异常检测时使用的历史均值/标准差或对比预设阈值范围，若恢复→写入 RemediationEffect(remediation_log_id/status=resolved/verified_value=最新值/threshold=原阈值/verified_at)，若未恢复→写入 RemediationEffect(status=failed/verified_value/threshold) 并调 notification_service 升级通知到值班人)─▶ 效果追踪页

## 链路 9：故障自动沉淀知识（无需人工审批）

资产管理页 ─(调 GET /assets/api/health-score?asset_id= → asset_change_service.get_asset_health_score 查 Asset + 查 Alert(status=triggered/acknowledged 计数) + 调 metric_v2_service.query_latest_values 取 CPU/MEM/DISK 最新值 → 计算评分(base 40 + 资源分[CPU<70 +20/MEM<70 +20/DISK<80 +20] - 告警惩罚[critical -15/warning -5]) → 返回 {score 0-100/status healthy(>=80)/warning(60-79)/danger(<60)/metrics/alert_count}；调 POST /assets/api/health-scan → scan_asset_health_changes 批量遍历 online 资产计算评分 → 对比上次状态变化时 log_change 写入 AssetChangeLog(asset_id/change_type=health_status/from_value/to_value/operator=system:health_scan/created_at) → 调 GET /asset-changes/logs → get_change_logs 查 AssetChangeLog 按时间倒序分页)─▶ 资产管理页 ─(资产列表展示健康评分徽章(绿/黄/红)，变更历史抽屉展示 AssetChangeLog 时间线，实现资产状态自动感知+变更审计闭环)

## 链路 23：移动端运维闭环（Mobile Ops Closed-Loop）

移动端页 ─(① 日志搜索：tools 页"日志搜索" → pages/logs/index → 调 GET /logs/api/sources 取数据源列表 → 调 GET /logs/api/search?keyword=&level=&source=&start=&end=&page= → 返回日志列表 → 展开查看原始内容/JSON 格式化 → 复制导出到剪贴板；② 告警批量：alert/list 页长按进入多选模式(selected 数组) → 底部 batch-bar 显示已选数量 → 批量确认调 POST /alerts/api/batch-acknowledge{alert_ids:[]} → 批量解决调 POST /alerts/api/batch-resolve{alert_ids:[]}；③ 故障单创建：incident/index 页"+ 新建故障单" → pages/incident/create 表单(标题/级别/影响/联系人/描述/资产关联) → 调 POST /incidents/api/create → 成功 toast 返回；④ AI 会话列表：agent/chat 页右上角"历史" → pages/agent/sessions → 调 GET /agent/sessions 取历史会话列表 → 点击继续对话 → 删除调 POST /agent/session/{id}/delete)─▶ 移动端页 ─(四项移动端能力补齐与 Web 版功能差距，日志搜索对标 Web、批量操作提升效率、故障单移动创建、AI 会话历史管理，实现真正的移动运维闭环)

## 链路 24：Agent 能力中心（Agent Capability Center）

Agent 能力中心页 ─(调 GET /agent/api/capabilities → get_agent_capabilities 查 AgentConfig(name=default) + AIProvider(default_provider_id) → 遍历 _MCP_TOOLS 注册表(41 个工具) → 返回 JSON{tools[{name/description/input_schema/risk_level/expose_to_llm/category}]/stats{total/llm_tools/internal_tools/risk_counts{read_only/low/medium/high/critical/advisory}}/agent_config/provider_info} → 前端按风险等级筛选(read_only/low/medium/high/critical/advisory)+按范围过滤(LLM 暴露/Internal 内部)+关键词搜索 → 展开查看 input_schema 详情)─▶ Agent 能力中心页 ─(工具注册表可视化，41 个工具按风险等级着色(绿=read_only/蓝=low/黄=medium/橙=high/红=critical/灰=advisory)，展示安全策略说明，LLM 可调用工具与 Internal 工具分区，实现 Agent 能力透明化管理)
