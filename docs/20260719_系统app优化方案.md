> 扫描范围：`mobile/` 全量代码（20 个页面 + 11 个 API + 5 个组件 + 4 个工具类 + 配置文件）
> 问题统计：**P0 严重 Bug 7 项 / P1 性能体验 10 项 / P2 架构工程化 11 项 / P3 锦上添花 8 项**，共 36 项
> 落地优先级：**P0 → P1 → P2 → P3**，P0 全部修复预计 1 人天，P1 约 3 人天，P2 约 5 人天

## 一、P0 严重 Bug（必须立即修复）

### P0-1 `pages/logs/index.vue` 文件缺失导致白屏
- **位置**：`mobile/src/pages.json:82-86` 注册路由，`mobile/src/pages/tools/index.vue:42` 跳转
- **现象**：工具箱点"日志搜索"→ `uni.navigateTo('/pages/logs/index')` → 找不到页面 → **白屏**
- **根因**：`pages.json` 声明了 `pages/logs/index`，但 `mobile/src/pages/logs/` 目录不存在（Glob 验证 `mobile/src/pages/logs/**` 返回 No files found）
- **修复**：新建 `mobile/src/pages/logs/index.vue`，复用 `mobile/src/api/log.js` 已有的 `getLogSources` / `searchLogs`，UI 参考 `metrics/index.vue` 的 Tab + 列表布局
- **验证**：构建后访问 `#/pages/logs/index` 不白屏，能正常搜索日志

### P0-2 告警详情用全量拉取再 find
- **位置**：`mobile/src/pages/alert/detail.vue:86`
- **代码**：`const data = await getList({ per_page: 100 })` → `arr.find(a => String(a.id) === String(id))`
- **现象**：当告警总数 > 100 条时，目标告警可能不在前 100 条 → **详情页加载不出来**；且每次打开详情都拉 100 条，**严重浪费流量和后端查询**
- **根因**：`mobile/src/api/alert.js` 未封装单条告警接口，开发者用全量拉取兜底
- **修复**：
  1. 后端确认 `GET /api/alerts/{id}` 端点存在（`app/routers/alerts.py`）
  2. 前端 `api/alert.js` 新增 `getDetail(id)` 方法
  3. `detail.vue:fetchDetail` 改为 `const data = await getDetail(id)`，移除全量拉取
- **验证**：构造 200 条告警场景，打开第 150 条详情能正常加载

### P0-3 `oncall/my.vue` 残留调试背景色
- **位置**：`mobile/src/pages/oncall/my.vue:282`
- **代码**：`.page-wrap { background: #ff00ff !important; min-height: 100vh; }`
- **现象**：整个"我的值班"页**粉紫色背景**（`#ff00ff` 是纯品红），用户看到会以为是 Bug
- **根因**：调试时为验证 `oncall/my.vue` 组件缓存问题（AGENTS.md 已记录）临时加的 `!important` 高对比背景，**忘记删除**
- **修复**：删除 `background: #ff00ff !important;`，恢复为默认 `$bg` 或不设置
- **验证**：页面背景与其他页面一致（`#F4F7FB`）

### P0-4 推送开关是假的
- **位置**：`mobile/src/pages/settings/index.vue:73-76`
- **代码**：`togglePush` 只执行 `uni.setStorageSync('push_enabled', ...)`，**未调用 `/mobile/push/register` 或 `/mobile/push/unregister`**
- **现象**：用户在设置页打开"推送通知"开关 → 只写了本地 storage → 后端不知道 → **永远收不到推送**
- **根因**：UI 开关与后端推送注册接口未打通；`app/routers/mobile.py:103-117` 已有 `/push/register` 和 `/push/unregister` 端点但前端未调用
- **修复**：
  ```js
  async function togglePush(e) {
      pushEnabled.value = e.detail.value
      uni.setStorageSync('push_enabled', pushEnabled.value ? 'true' : 'false')
      try {
          if (pushEnabled.value) {
              const cid = await getPushClientId()  // uni.getPushClientId
              await registerPush({ cid, platform: uni.getSystemInfoSync().platform })
          } else {
              await unregisterPush()
          }
      } catch (err) {
          pushEnabled.value = !e.detail.value  // 回滚
          uni.showToast({ title: '推送设置失败', icon: 'none' })
      }
  }
  ```
- **验证**：开关切换后调 Network 看到 `/mobile/push/register` 请求，后端 `mobile_devices` 表有记录

### P0-5 `manifest.json` 的 appid 全为空
- **位置**：`mobile/src/manifest.json:3`（`"appid": ""`）、`:58`（`mp-weixin.appid: ""`）、`:47-51`（`igexin` 推送 appid/appkey/appsecret 全空）
- **现象**：
  - HBuilderX 云打包 App 会报错"appid 不能为空"
  - 微信小程序无法发布（微信后台要求 appid 与小程序后台一致）
  - 个推（igexin）推送无法注册（无 appkey）
- **根因**：项目初期未申请三方账号，占位空值未补齐
- **修复**：
  1. DCloud 后台申请 uni-app appid 填入 `manifest.json:3`
  2. 微信公众平台申请小程序 appid 填入 `mp-weixin.appid`
  3. 个推后台申请 appid/appkey/appsecret 填入 `app-plus.distribute.sdkConfigs.push.igexin`
- **验证**：HBuilderX 打包成功，微信开发者工具能预览，推送能注册

### P0-6 `incident/index.vue` 的"加载更多"是假的
- **位置**：`mobile/src/pages/incident/index.vue:89-93`
- **代码**：
  ```js
  function loadMore() {
      if (loadingMore.value || noMore.value) return
      loadingMore.value = true
      setTimeout(() => { loadingMore.value = false }, 500)  // ← 500ms 后什么都不做
  }
  ```
- **现象**：故障单列表滚到底部 → 显示"加载中..."500ms → 消失，**没有任何新数据加载**；且 `fetchList` 一次拉全量不分页（`:71` `getIncidentList(activeTab.value)` 无 page/per_page 参数）
- **根因**：开发者写了 `loadMore` 钩子但未实现分页逻辑，`api/incident.js` 的 `getIncidentList` 也未传分页参数
- **修复**：
  1. `api/incident.js` 的 `getIncidentList` 增加 `page` / `per_page` 参数
  2. `fetchList(reset)` 接受 reset 标志，reset 时 `list = arr`，非 reset 时 `list = list.concat(arr)`
  3. `loadMore` 改为 `page++; fetchList(false)`，根据 `arr.length < per_page` 判断 `noMore`
- **验证**：构造 100 条故障单，滚到底部能加载第 2 页

### P0-7 `asset/detail.vue` 的"重启服务"是假按钮
- **位置**：`mobile/src/pages/asset/detail.vue:286-294`
- **代码**：

  function goRestart() {
      uni.showModal({
          title: '重启服务', content: '确认重启该资产上的服务？',
          success: (r) => { if (r.confirm) uni.showToast({ title: '已提交', icon: 'success' }) }

- **根因**：占位实现未补齐；后端 `app/routers/script_exec.py` 或 `remediation.py` 有执行能力但前端未对接

- **验证**：点击后 Network 有请求，或按钮被移除

## 二、P1 性能 / 体验优化

### P1-1 首页 5 秒全量轮询 dashboard
- **位置**：`mobile/src/pages/index/index.vue:150` `setInterval(loadData, 5000)`
- **问题**：每 5 秒拉一次 `/mobile/dashboard` 全量数据（健康分+告警+值班+工作流），移动端耗电、后端压力、用户流量浪费
- **优化**：
  1. 轮询间隔从 5s → **30s**（移动端可接受）
  2. 已有 `utils/ws.js` 的 WS 推送通道，**告警数变化走 WS 增量**，dashboard 只在 onShow 时拉一次
  3. 加 `visibilitychange` 监听，页面隐藏时暂停轮询（已有 `onHide` 但只停了 timer，未停 WS）
- **收益**：后端 QPS 降 6 倍，用户流量降 6 倍

### P1-2 AI 助手无流式响应
- **位置**：`mobile/src/pages/agent/chat.vue:104` `await sendMessage(...)`
- **问题**：一次性 `await` 等待完整回复，大模型慢时（10s+）用户干等，只看到"AI 正在输入..."
- **优化**：
  1. 后端 `app/routers/agent.py` 的 `POST /api/agent/chat` 增加 SSE 流式端点（`text/event-stream`）
  2. 前端改用 `EventSource` 或 `uni.request` + `enableChunked` 接收
  3. `aiMsg.content` 边收边拼接，`scrollToBottom` 跟随
- **收益**：首字响应 < 1s，用户感知大幅提升

### P1-3 告警列表无虚拟列表
- **位置**：`mobile/src/pages/alert/list.vue:34-46` `v-for="item in list"`
- **问题**：千条告警全量渲染 DOM，小程序端会卡顿甚至 OOM
- **优化**：改用 `uni-recycle-list` 或 `<scroll-view>` + 自实现虚拟列表（只渲染可视区 ± buffer）
- **收益**：1000 条列表滚动 FPS 从 < 20 提升到 60

### P1-4 `main.js` 全局拦截"拨号"是 hack
- **位置**：`mobile/src/main.js:6-27`
- **问题**：在入口文件用 `document.addEventListener('click', ..., true)` 捕获阶段拦截所有点击，通过 `textContent.trim() === '拨号'` 匹配按钮 → **跨端不兼容**（小程序无 `document`）、**性能损耗**（每次点击都遍历 DOM）、**难维护**
- **根因**：AGENTS.md 已记录是绕过 `oncall/my.vue` 组件缓存问题的临时方案
- **优化**：
  1. 在 `oncall/my.vue` 的"拨号"按钮直接绑定 `@tap="callMember(item)"`（已有此方法）
  2. 删除 `main.js` 的全局拦截代码
  3. 如组件缓存问题仍存在，用 `npm run build:h5` 全量构建而非 dev server
- **收益**：代码可维护性 +10，跨端兼容性 +1

### P1-5 `agent/chat.vue` 直接操作 DOM
- **位置**：`mobile/src/pages/agent/chat.vue:82-84`（`document.getElementById('msg-list')`）、`:273-279`（`document.querySelector('uni-page-body')`）
- **问题**：小程序端、App 端无 `document` 对象 → **运行时报错**；且直接操作 `uni-page-body` 样式是 hack
- **优化**：
  1. 改用 `uni.createSelectorQuery().in(this).select('#msg-list').boundingClientRect()`
  2. 或用 Vue ref `<scroll-view ref="msgList">` + `msgList.value.scrollTo()`
  3. 删除 `uni-page-body` 的样式 hack，改用 CSS `height: 100vh` + `flex` 布局
- **收益**：跨端兼容，无运行时报错

### P1-6 `asset/detail.vue` 重复实现图表代码
- **位置**：`mobile/src/pages/asset/detail.vue:81-97`（chart-bars）、`:120-126`（barHeight）、`:272-276`（formatTime）
- **问题**：与 `mobile/src/pages/metrics/index.vue:25-33`、`:120-126`、`:114-118` **几乎完全重复**，维护两份
- **优化**：抽 `mobile/src/components/MetricChart.vue` 组件，props 接收 `chartData` / `latestValue`，两处复用
- **收益**：代码量 -120 行，统一图表样式

### P1-7 `asset/detail.vue` 直接用 `uni.request` 而非走 `api/metrics.js`
- **位置**：`mobile/src/pages/asset/detail.vue:213-221`、`:241-249`
- **问题**：`fetchMetricsNames` / `switchMetric` 内部 `new Promise` + `uni.request` 手写请求，未复用 `api/metrics.js` 的 `getMetricNames` / `getMetricData`，**绕过了统一 header / 错误处理**
- **优化**：改为 `import { getMetricNames, getMetricData } from '@/api/metrics.js'`，删除手写 `uni.request`
- **收益**：统一请求层，401 自动跳登录等错误处理生效

### P1-8 `asset/detail.vue` 全量拉 100 条告警再 filter
- **位置**：`mobile/src/pages/asset/detail.vue:171-182`
- **代码**：`getList({ per_page: 100 })` → `arr.filter(a => a.asset_id === assetId).slice(0, 5)`
- **问题**：与 P0-2 同类问题，告警 > 100 条时"最近告警"不全；且拉 100 条只为取 5 条，**浪费 95 条流量**
- **优化**：`api/alert.js` 的 `getList` 已支持 `asset_id` 参数，直接 `getList({ asset_id: id, per_page: 5 })`
- **收益**：流量降 20 倍，查询速度提升

### P1-9 `scan.vue` 的 NFC 实现未隔离
- **位置**：`mobile/src/pages/scan.vue:79-147`
- **问题**：
  1. NFC 代码 70 行写在页面组件里，混入业务逻辑
  2. `#ifdef APP-PLUS` 包裹但 H5 端访问会显示"请在 App 中使用"提示，**未做 UI 降级**（应隐藏 NFC 按钮而非点击后提示）
  3. `plus.android.importClass` 在 iOS 或小程序端调用会报错
- **优化**：
  1. 抽 `mobile/src/utils/nfc.js` 封装 NFC 读取逻辑
  2. `scan.vue` 的 NFC 按钮加 `<!-- #ifdef APP-PLUS -->` 条件编译，非 App 端不渲染
  3. 错误处理细化（NFC 未开启、无标签、读取超时分类提示）
- **收益**：代码可维护性 +1，跨端无报错

### P1-10 `workflow/list.vue` 一次拉 30 条无分页
- **位置**：`mobile/src/pages/workflow/list.vue:70` `params = { page: 1, per_page: 30 }`
- **问题**：固定 30 条，超过 30 条的工作流看不到；`loadMore` 未实现
- **优化**：参考 `alert/list.vue` 的分页模式，加 `loadMore` + `noMore` 状态
- **收益**：长列表可加载完整数据

## 三、P2 架构 / 工程化

### P2-1 离线缓存两套并存
- **位置**：`mobile/src/utils/offline.js`（`cache_` 前缀 + expire）vs `mobile/src/store/offline.js`（`offline_cache` 整存 + Pinia）
- **问题**：两套缓存职责重叠，`utils/offline.js` 的 `setCache/getCache` 与 `store/offline.js` 的 `cacheAlerts/cacheAsset` 并存，开发者不知道用哪套
- **优化**：统一为 `store/offline.js` 一套，`utils/offline.js` 只保留 `checkNetwork` / `onNetworkChange` 网络相关方法
- **收益**：单一数据源，缓存清理逻辑统一

### P2-2 `request.js` 过于简陋
- **位置**：`mobile/src/api/request.js`
- **问题**：无超时、无重试、无统一错误码处理、无请求取消；`api/log.js:5-28` 自己实现了 `cancelKey`，说明有需求
- **优化**：
  ```js
  export function request({ url, method = 'GET', data, header, hideError, timeout = 15000, retry = 1, cancelKey }) {
      // 1. timeout 超时控制
      // 2. retry 失败重试（仅 GET，仅网络错误）
      // 3. cancelKey 请求取消（复用 log.js 的实现）
      // 4. 统一错误码：401 跳登录、403 提示无权限、500 提示服务异常、502/503 提示维护中
      // 5. hideError 静默模式
  }
  ```
- **收益**：所有 API 共享超时/重试/取消能力，`api/log.js` 的 cancelMap 可删除

### P2-3 常量魔法值散落
- **位置**：`PAGE_SIZE=20`（alert/list.vue:78）、`5000`（index.vue:150）、`3000/30000`（ws.js:7,46）、`30/60/240/1440`（alert/detail.vue:126）、`120`（metrics/index.vue:87）、`60`（asset/detail.vue:253）、`100`（alert/detail.vue:86, asset/detail.vue:173）
- **优化**：抽 `mobile/src/constants.js`：
  ```js
  export const PAGE_SIZE = 20
  export const POLL_INTERVAL_DASHBOARD = 30000
  export const WS_RECONNECT_DELAY = 3000
  export const WS_RECONNECT_MAX = 30000
  export const SILENCE_OPTIONS = [{ label: '30 分钟', minutes: 30 }, ...]
  export const CHART_POINTS_MAX = 120
  ```
- **收益**：调参集中，魔法值消除

### P2-4 i18n 引入未使用
- **位置**：`mobile/package.json:19` `"vue-i18n": "^9.1.9"`
- **问题**：引入了 `vue-i18n` 但无 `locale/` 目录、无 `createI18n` 调用，**死依赖增加包体积**（vue-i18n gzip 约 30KB）
- **优化**：
  - 方案 A（推荐）：移除 `vue-i18n` 依赖，当前 UI 全中文无需国际化
  - 方案 B：若计划做国际化，创建 `locale/zh-CN.js` + `locale/en-US.js` + `main.js` 注册
- **收益**：包体积 -30KB

### P2-5 缓存清理硬编码 keep 列表
- **位置**：`mobile/src/pages/settings/index.vue:112`
- **代码**：`const keep = ['session_cookie', 'auth_token', 'server_base_url', 'biometric_token', 'user_info', 'device_id', 'push_enabled']`
- **问题**：新增 storage key 时极易遗漏加入 keep 列表 → **误清登录态/配置**
- **优化**：改"前缀策略"——所有缓存统一用 `cache_` 前缀（`utils/offline.js` 已用此约定），清理时只清 `cache_` 前缀：
  ```js
  keys.forEach(k => { if (k.startsWith('cache_')) uni.removeStorageSync(k) })
  ```
- **收益**：零遗漏风险，新加 storage key 无需改清理逻辑

### P2-6 无分包，主包过大
- **位置**：`mobile/src/pages.json` 20 个页面全在主包
- **问题**：微信小程序主包限制 2MB，当前 20 个页面 + 5 个组件 + uni-ui 全量引入，**build 后可能超限**
- **优化**：`pages.json` 增加 `subPackages`，按业务拆分：
  - 主包：index、alert、agent、tools、settings、login
  - 分包 `subpkg-ops`：logs、metrics、incident、workflow、oncall
  - 分包 `subpkg-asset`：asset/scan、asset/detail、asset/diagnose
- **收益**：主包体积降 40%，符合微信审核要求

### P2-7 健康分 trend 永远为空
- **位置**：`mobile/src/pages/index/index.vue:10` `:trend="''"`、`:trendValue="0`
- **问题**：`HealthCard.vue` 有 trend 展示能力（up/down/flat + 数值），但首页永远传空字符串，**组件能力浪费**
- **优化**：后端 `/mobile/dashboard` 增加 `health.trend` 字段（对比 1 小时前 SLA），前端传入 `:trend="dashboard.health.trend_dir"` `:trendValue="dashboard.health.trend_value"`
- **收益**：健康分趋势可视化，运维人员能感知系统状态变化方向

### P2-8 `config.js` 的 DEFAULT_BASE 在 H5 子路径下可能丢路径
- **位置**：`mobile/src/api/config.js:1-3`
- **代码**：`DEFAULT_BASE = window.location.origin`（只取 origin，不含 `/mobile-app/` 路径）
- **问题**：H5 部署在 `/mobile-app/` 子路径，`origin` 是 `http://host:port`，但后端 API 在 `http://host:port/api/...`（无 `/mobile-app/` 前缀），实际请求路径正确；但若用户手动改 `server_base_url` 为 `http://host:port/mobile-app`，会导致 API 404
- **优化**：明确注释 DEFAULT_BASE 的语义，加校验：保存 `server_base_url` 时若以 `/mobile-app` 结尾自动剥离
- **收益**：避免用户误配置

### P2-9 WS 401 不处理
- **位置**：`mobile/src/utils/ws.js:43-47` `onClose` 直接重连
- **问题**：token 过期时 WS 连接被服务端关闭（401），前端 `onClose` 无脑重连 → **无限重连风暴**，耗电耗流量
- **优化**：`onMessage` 解析服务端 `{"type":"error","code":401}` 时调 `userStore.logout()` 跳登录，停止重连；`onClose` 时检查 `reason` 是否为认证失败
- **收益**：避免重连风暴，用户体验友好

### P2-10 无全局错误边界
- **位置**：`mobile/src/main.js:29-32`
- **问题**：无 `app.config.errorHandler`，JS 运行时错误静默，线上问题难定位
- **优化**：
  ```js
  app.config.errorHandler = (err, vm, info) => {
      console.error('[Vue Error]', err, info)
      // 上报到后端 /mobile/error-report
  }
  uni.onError && uni.onError((err) => { console.error('[uni.onError]', err) })
  ```
- **收益**：线上 JS 错误可追踪

### P2-11 无单元测试
- **位置**：`mobile/` 下无 `test/` 目录，`package.json` 无 `vitest` / `jest` 依赖
- **问题**：工具类（`utils/offline.js`、`utils/crypto.js`、`utils/biometric.js`）、store（`user.js`、`offline.js`）无测试覆盖，重构无保障
- **优化**：
  1. 安装 `vitest` + `@vue/test-utils`
  2. 为 `utils/offline.js` 的 `setCache/getCache` 写单测（expire 逻辑）
  3. 为 `store/user.js` 的 `login`/`logout`/`checkBiometric` 写单测（mock `authApi`）
  4. CI 加 `npm run test`
- **收益**：重构有保障，回归风险降

## 六、附录

### 6.1 专业名词学习
- **虚拟列表（Virtual List / Windowing）** — 只渲染可视区域内的列表项，长列表性能优化标配，对应 `uni-recycle-list`
- **流式响应（Streaming Response / SSE）** — 服务端逐 token 推送，前端边收边渲染，解决 LLM 长等待
- **分包加载（Subpackage）** — 小程序将主包拆为多个子包按需下载，规避 2MB 主包限制
- **骨架屏（Skeleton Screen）** — 用灰色占位块预渲染页面结构，感知性能优于 spinner
- **请求取消（Request Cancellation）** — 通过 abort controller 中断未完成请求，用于搜索框防抖等场景
- **错误边界（Error Boundary）** — Vue 的 `errorHandler` / React 的 `componentDidCatch`，捕获组件树未处理异常
- **热更新（Hot Update / OTA）** — 小程序通过 `uni.getUpdateManager` 检查并下载新版本，App 通过 wgt 包差量更新
