# AIOps 用户操作手册 — 链路5-8：变更关联 & 知识指导 & SLO & 自动响应

> **适用角色**：运维工程师、SRE、变更管理员
> **前置条件**：已登录系统，拥有相关菜单权限
> **预计耗时**：约 15-20 分钟

---

## 概述

本手册覆盖四条链路：
- **链路5**：事件变更关联故障（RCA 辅助证据）
- **链路6**：知识指导 SRE 实践（告警研判）
- **链路7**：SLO 违规监控（手动触发计算）
- **链路8**：告警直接触发自动响应（完整闭环）

> **链路5 变更关联**　①资产变更 → ②RCA关联
> **链路6 知识指导**　①告警知识推荐 → ②知识库搜索 → ③RCA自动沉淀
> **链路7 SLO**　①配置SLO → ②手动触发计算
> **链路8 自动响应**　①配置规则 → ②自动触发(后台) → ③查看执行 → ④效果统计
>

# 链路5：事件变更关联故障

## 说明
⚠️ **链路5 当前实现**：变更记录仅作为 RCA 分析的辅助证据，**不会自动触发故障单**。
- 变更记录写入 `AssetChangeLog` 表（资产更新/健康扫描时自动记录）
- RCA 分析时查询关联资产的变更记录，作为时间线事件
- **无** `ChangeRecord` 模型，**无**变更直接触发故障的自动化链路
- 实际模型为 `ChangeRequest` + `ChangeTask`

## 第一步：记录资产变更

### 操作页面
**侧边栏** → **运维工作台** → **资产管理** → **资产列表**

### 操作步骤

1. **编辑资产信息**
   - 点击某条资产的「编辑」按钮
   - 修改资产属性（如 IP、状态、标签等）
   - 点击「保存」

2. **查看变更记录**
   - 在资产详情页，点击「变更历史」Tab
   - 页面显示该资产的所有变更记录（`AssetChangeLog`）
   - 每条记录显示：
     - **变更类型**：`field_update`（字段更新）/ `health_status`（健康状态变化）
     - **变更字段**：修改的字段名
     - **旧值 → 新值**
     - **操作人**：`user_id`（自动记录）
     - **变更时间**：`created_at`

### 看到什么
- 变更历史列表按时间倒序排列
- 每条变更记录完整追踪字段变化

## 第二步：故障单 RCA 关联变更

### 操作页面
**侧边栏** → **值班驾驶舱** → **故障管理** → 故障详情 → **RCA 分析**

### 操作步骤

1. **触发 RCA 分析**
   - 在故障详情页，点击「RCA 分析」按钮
   - 系统调用 `rca_service.analyze_incident(incident_id)`

2. **查看 Investigation Package**
   - RCA 分析结果分为 6 部分：
     - **Facts**：异常实体列表
     - **Timeline**：时间线 + 变更事件（来自 `AssetChangeLog`）
     - **Candidate Causes**：候选根因（按置信度排序）
     - **Evidence**：证据列表
     - **Exclusions**：已排除方向
     - **Next Steps**：建议操作

3. **查看变更关联**
   - 在 **Timeline** 部分，显示故障时间窗口内的变更事件
   - 格式：`[变更时间] 资产变更: {change_type} - {description}`
   - 帮助判断故障是否由变更引发

### 看到什么
- RCA 报告中 Timeline 部分显示关联的变更记录
- 候选根因中可能包含"核对变更记录"建议
- 如果存在变更关联，Next Steps 中提示"检查变更是否引入问题"

### 结果
变更记录作为 RCA 辅助证据，帮助定位根因。

## 第三步：RCA 结果自动沉淀知识

### 说明
故障 RCA 分析完成后，系统**自动触发**知识沉淀。

### 沉淀过程

1. **触发时机**：故障状态变为 `resolved` 后
2. **自动归档**：调用 `knowledge_graph_service.record_rca_result()`
3. **写入知识库**：
   - `title`：故障标题
   - `symptom`：告警消息汇总
   - `root_cause`：RCA 根因分析
   - `solution`：Next Steps 建议
   - `tags`：从告警指标名/资产类型提取
   - `source_type`：`incident_case`
4. **RAG 索引更新**：调用 `rag_service.index_document()` 更新 TF-IDF 向量

### 看到什么
- 无需手动操作
- 沉淀的知识条目出现在「知识库」列表中，`source_type=incident_case`

### 结果
RCA 结果自动沉淀为知识，可供后续 AI 问答和 RAG 检索使用。

# 链路7：SLO 违规监控

## 说明
⚠️ **链路7 当前实现**：SLO 计算结果仅作为**仪表盘展示**，**不会触发告警**，**不会触发自愈**。
- SLO 计算：手动触发 API 计算，更新 `SLOConfig.status`
- SLO 状态：`healthy` / `warning` / `critical`
- **未实现**：SLO 状态变化 → 创建告警 → 触发自愈的自动化链路

## 第一步：配置 SLO

### 操作页面
**侧边栏** → **可靠性工程** → **SLO 管理** → **SLO 配置**

### 操作步骤

1. **新建 SLO**
   - 点击「新建 SLO」按钮
   - **SLO 名称**：输入 `API 可用性 SLO`
   - **关联服务**：选择目标资产（如 API 网关）
   - **SLO 目标**：输入 `99.9`（可用性 99.9%）
   - **时间窗口**：选择「滚动 28 天」
   - **Error Budget**：自动计算 `(1 - 0.999) * 总请求数`

2. **保存 SLO**
   - 点击「保存」
   - SLO 配置出现在列表中，状态为 `未计算`

**下一步：手动触发 SLO 计算** →

## 第二步：手动触发 SLO 计算

### 操作页面
**侧边栏** → **可靠性工程** → **SLO 管理** → **SLO 仪表盘**

### 操作步骤

1. **查看 SLO 仪表盘**
   - 页面显示 SLO 卡片式仪表盘
   - 每张卡片显示：
     - **SLO 名称**
     - **目标值**：99.9%
     - **当前可用性**：实际计算值
     - **Error Budget 剩余**：剩余百分比
     - **燃烧速率**：当前 Burn Rate
     - **状态**：healthy / warning / critical

2. **手动触发计算**
   - 点击页面右上角「重新计算」按钮
   - 系统调用 `POST /api/sre/slo/calculate`
   - 后台执行 `slo_service.calculate_all_slo()`：
     - 查询 `SLOConfig` 表中所有 SLO
     - 对每个 SLO，从 VictoriaMetrics 查询实际指标
     - 计算可用性 = 成功请求数 / 总请求数
     - 计算 Error Budget = (1 - SLO目标) * 总请求数 - 失败请求数
     - 计算 Burn Rate = Error Budget 消耗量 / 时间窗口长度
     - 更新 `SLOConfig.status`：
       - Burn Rate > 10 或 Budget < 20% → `critical`
       - Burn Rate > 5 或 Budget < 50% → `warning`
       - 其他 → `healthy`
   - 页面刷新，显示最新计算结果

3. **查看 SLO 详情**
   - 点击某张 SLO 卡片
   - 弹窗显示：
     - 历史可用性趋势图（ECharts 折线图）
     - Error Budget 消耗趋势
     - 燃烧速率变化
     - 关联的告警列表（如有）

### 看到什么
- SLO 卡片显示最新计算结果
- 状态徽章：绿色（healthy）/ 橙色（warning）/ 红色（critical）
- 趋势图展示历史变化

### 结果
SLO 计算完成，展示当前可用性和 Error Budget 状态。

## 第二步：告警触发自动响应（后台自动）

### 说明
自动响应是**后台定时任务**，无需手动操作。

### 执行过程

1. **定时扫描**：后台任务 `remediation_service.check_and_remediate()` 每轮循环执行
2. **规则匹配**：查询 `AutoRemediation` 表中 `enabled=True` 的规则
3. **告警关联**：对每条规则，查询 `status=triggered` 的关联告警
4. **去重检查**：同一告警+同一规则最近已执行过则跳过
5. **SSH 执行**：
   - 根据 `alert.asset_id` 查 `Asset` 获取 IP/SSH 凭据
   - 用 paramiko SSH 连接远程资产
   - 按 `action_type` 执行动作：
     - `restart`：`sudo systemctl restart {service}`
     - `clean`：`find {path} -type f -mtime +7 -delete`
     - `run_command`：执行自定义命令（危险命令黑名单拦截）
     - `notify`：发送通知
6. **写入执行日志**：创建 `RemediationLog` 记录（成功/失败/输出）
7. **更新告警状态**：`Alert.status = acknowledged`
8. **效果追踪**：30 分钟后检查告警是否恢复，写入 `RemediationEffectRecord`

### 看到什么
- 无需手动查看
- 执行结果在「任务中心」→「自愈执行」页面可查看

## 常见问题

### Q1：SLO 计算结果不准确？
- **检查**：VictoriaMetrics 是否有对应的指标数据
- **检查**：SLO 配置的时间窗口是否正确
- **检查**：`slo_service.calculate_all_slo()` 是否在后台循环中执行

### Q2：自动响应没有执行？
- **检查**：`AutoRemediation` 是否 `enabled=True`
- **检查**：关联的 `AlertRule` 是否匹配当前告警
- **检查**：`RemediationLog` 是否有最近执行记录（去重机制会跳过）
- **检查**：SSH 凭据是否正确（`Asset.connection_config`）

### Q3：自动响应执行失败？
- **检查**：网络是否连通（能否 SSH 到目标资产）
- **检查**：服务名是否正确（`systemctl restart {service}`）
- **检查**：命令是否被危险命令黑名单拦截

### Q4：效果追踪显示未恢复？
- **检查**：告警是否在 30 分钟内解决
- **检查**：自愈执行是否成功（检查 `RemediationLog.success`）
- **检查**：指标是否恢复正常（检查 `MetricRecord` 最新值）

## 专业术语

| 术语 | 英文 | 说明 |
|------|------|------|
| Change Correlation | 变更关联 | 将基础设施变更事件与故障事件时间线对齐 |
| RCA | Root Cause Analysis | 根因分析，定位故障根本原因 |
| Investigation Package | 调查包 | 结构化 RCA 输出（Facts/Timeline/Candidates/Evidence/Exclusions/NextSteps） |
| SLO | Service Level Objective | 服务级别目标，定义可用性/延迟等指标目标值 |
| Error Budget | 错误预算 | SLO 允许的容错空间 |
| Burn Rate | 燃烧速率 | Error Budget 消耗速率，>1 表示正在消耗预算 |
| Auto-Remediation | 自动修复 | 根据预定义规则自动执行恢复操作 |
| Steady-State | 稳态 | 系统正常运行时的状态基线 |
