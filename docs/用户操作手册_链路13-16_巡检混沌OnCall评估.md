# 链路13：告警触发巡检自动化

## 第一步：查看告警并触发巡检

### 操作页面
**侧边栏** → **值班驾驶舱** → **告警中心** → **告警详情**

### 操作步骤

1. **选择告警**
   - 在告警列表中，点击某条告警
   - 进入告警详情页

2. **点击「触发巡检」按钮**
   - 在告警详情页，点击「触发巡检」按钮
   - 系统调用 `POST /inspection/api/trigger-by-alert/{alert_id}`

3. **自动匹配巡检模板**
   - 后台执行 `inspection_service.trigger_by_alert(alert_id, db)`：
     - 查 `Alert(id)` 取 `asset_id`
     - 查 `Asset(id)` 取 `ci_type`
     - 遍历 `InspectionTemplate` 表匹配 `target_ci_types` JSON 数组包含该 `ci_type` 的模板
     - 创建临时巡检任务：
       - `name`：`"告警触发巡检: " + alert.message`
       - `scope`：`dynamic`
       - `ci_types`：`[asset.ci_type]`
       - `created_by`：`system`

### 看到什么
- Toast 提示「巡检任务已创建」
- 系统自动执行巡检

## 第二步：巡检执行（后台自动）

### 说明
巡检执行是**后台自动任务**，无需手动操作。

### 执行过程

1. **逐资产执行 9 类检查项**：
   - `cpu_usage`：CPU 使用率
   - `memory_usage`：内存使用率
   - `disk_usage`：磁盘使用率
   - `active_alerts`：活跃告警数
   - `asset_status`：资产在线状态
   - `last_checked_at`：最近检查时间
   - `span_error_rate`：Span 错误率
   - `span_p99_ms`：Span P99 延迟
   - `span_avg_ms`：Span 平均延迟

2. **评估 worst_status**：逐项检查，取最严重的状态作为整体状态

3. **生成健康评分**：根据检查结果计算 0-100 分

4. **写入巡检记录**：
   - `task_id`：关联巡检任务
   - `triggered_by_alert_id`：关联原始告警
   - `status`：`completed`
   - `total_assets`：检查资产总数
   - `normal_count`：正常数
   - `warning_count`：告警数
   - `critical_count`：严重数
   - `health_score`：健康评分

5. **AI 生成巡检报告**：
   - 调用 LLM 生成专业巡检报告
   - 报告包含：摘要、评分解读、异常分析、趋势预判、处置建议、风险预警
   - 如果 LLM 不可用，降级为规则报告

### 看到什么
- 无需手动查看
- 巡检结果出现在「巡检记录」页面

# 链路14：混沌工程稳态验证与自动回滚

## 第一步：配置混沌实验

### 操作页面
**侧边栏** → **可靠性工程** → **混沌工程** → **混沌实验**

### 操作步骤

1. **点击「新建实验」按钮**

2. **填写实验信息**
   - **实验名称**：输入 `CPU 压力注入测试`
   - **目标资产**：选择目标服务器
   - **故障类型**：选择 `CPU 压力`（支持 14 种故障类型）
   - **注入命令**：系统自动生成（如 `stress --cpu 4 --timeout 60`）
   - **清理命令**：系统自动生成（如 `pkill -f stress`）
   - **持续时间**：输入 `60`（秒）
   - **稳态配置**：
     - **指标**：选择 `CPU 可用率`
     - **阈值**：输入 `20`（可用率低于 20% 视为异常）
     - **运算符**：选择 `<`

3. **点击「保存」**
   - 实验配置出现在列表中

**下一步：启动混沌实验** →

## 第二步：启动混沌实验

### 操作页面
**侧边栏** → **可靠性工程** → **混沌工程** → **混沌实验**

### 操作步骤

1. **启动实验**
   - 在实验列表中，点击「启动」按钮
   - 系统调用 `POST /api/chaos/experiments/{id}/start`

2. **自动执行流程**
   - 后台线程 `_inject_and_observe_async` 启动：
     - **第一步：采集 before 稳态基线**
       - SSH 连接目标资产
       - 采集 CPU/MEM/DISK 可用率
     - **第二步：注入故障**
       - SSH 执行 `inject_cmd`（如 `stress --cpu 4 --timeout 60`）
       - 注册 `cleanup_cmd` 供 abort 使用
     - **第三步：观察期**
       - `sleep(duration_seconds)`
     - **第四步：采集 after 稳态值**
       - 再次采集 CPU/MEM/DISK 可用率
     - **第五步：稳态验证**
       - `after_avail >= threshold` 则 `steady_state_passed=True`，否则 `False`
     - **第六步：主动清理**
       - 立即 SSH 执行 `cleanup_cmd`（如 `pkill -f stress`）
       - 标记 `auto_recovered=True`
     - **第七步：落库**
       - 创建 `ChaosRun` 记录：
         - `experiment_id`
         - `steady_state_passed`
         - `steady_state_before`：JSON（before 指标）
         - `steady_state_after`：JSON（after 指标）
         - `alerts_triggered`：注入期间触发的告警数
         - `duration_seconds`
         - `is_auto_recovered`：是否自动回滚

### 看到什么
- 实验状态从「就绪」变为「执行中」
- 执行完成后，状态变为「已完成」

## 第三步：自动轮转（后台自动）

### 说明
自动轮转是**后台自动任务**，无需手动操作。

### 轮转过程

1. **触发时机**：查询当前值班人时，检测到 `current_period_ended_at < now` 且 `is_auto_rotate=True`
2. **自动轮转**：调用 `_do_auto_rotate(db, schedule)`：
   - 解析 `members` JSON 数组
   - 找到当前 oncall 在成员列表中的索引
   - `next_idx = (current_idx + 1) % len(names)`（循环轮转）
   - 根据 `rotation_type` 设置新周期：
     - `weekly` → +7 天
     - `monthly` → +30 天
   - 更新 `current_oncall` / `current_period_started_at` / `current_period_ended_at`
3. **返回新值班人**

### 看到什么
- 无需手动查看
- 下次查询时，显示新的值班人

## 第四步：执行 A/B 测试

### 说明
A/B 测试是**自动分流**，无需手动操作。

### 分流过程

1. **确定分桶**：对 `session_id` 取 MD5 哈希，取前 8 位 hex 转 int，mod 100
2. **按比例分流**：`bucket < split_ratio` 则走 A 组，否则走 B 组
3. **调用 Provider**：按分桶结果调用对应的 LLM Provider
4. **记录结果**：写入 `ABTestRecord`（config_id/session_id/provider_id/success/latency_ms）

### 看到什么
- 无需手动查看
- 测试结果在「A/B 测试」页面展示

## 第三步：执行测试用例

### 操作页面
**侧边栏** → **AI运维智能体** → **Agent 评估** → **GroundTruth 测试**

### 操作步骤

1. **单用例执行**
   - 在用例列表中，点击某条用例的「执行」按钮
   - 系统调用 `POST /agent/api/ground-truth/tests/{id}/run`
   - 自动创建一个新的 AI 会话（ChatSession）
   - 将测试问题作为用户消息发送给 Agent
   - Agent 调用 MCP 工具、生成回答
   - 完成后返回评分结果

2. **查看执行结果**
   - 执行完成后，页面显示评分结果弹窗：
     - **语义相似度**：实际回答与期望答案的语义匹配度（0~1），基于 SentenceTransformer 计算余弦相似度
     - **工具匹配度**：实际调用工具与期望工具的匹配度（0~1），工具名称+参数精确匹配
     - **综合评分**：语义相似度 × 0.5 + 工具匹配度 × 0.5（加权平均）
     - **是否通过**：综合评分 ≥ 0.6 为通过
     - **实际工具**：Agent 实际调用的工具
     - **实际参数**：Agent 实际使用的参数
     - **响应时间**：毫秒

3. **批量执行**
   - 点击「批量执行」按钮
   - 系统逐个执行所有启用的用例
   - 进度条显示执行进度

- 批量执行：进度条显示执行进度，完成后自动刷新统计

## 常见问题

### Q1：告警触发巡检没有执行？
- **检查**：`InspectionTemplate` 是否有匹配的模板（`target_ci_types` 包含告警资产的 `ci_type`）
- **检查**：`trigger_by_alert()` 是否在后台循环中执行

### Q2：混沌实验失败？
- **检查**：目标资产是否在线
- **检查**：SSH 凭据是否正确
- **检查**：`inject_cmd` 和 `cleanup_cmd` 是否正确
- **检查**：`tc` 命令是否安装（网络故障注入需要）

### Q3：On-Call 自动轮转没有触发？
- **检查**：`is_auto_rotate` 是否开启
- **检查**：`current_period_ended_at` 是否小于当前时间
- **检查**：`_do_auto_rotate()` 是否在后台循环中执行

### Q4：A/B 测试结果不准确？
- **检查**：分流比例是否正确（如 `50/50`）
- **检查**：样本量是否足够（建议 > 100 次请求）
- **检查**：评估指标是否合理（如综合评分 = 成功率 - 延迟/1000）

### Q5：GroundTruth 执行评分很低？
- **检查**：期望答案/期望工具是否准确反映了正确操作
- **检查**：LLM Provider 是否配置正确（Agent 需要能正常调用）
- **检查**：语义相似度模型（SentenceTransformer）是否加载成功
- **检查**：工具参数格式是否正确（`expected_parameters` 必须是标准 JSON）

### Q6：GroundTruth 批量执行卡住？
- **检查**：是否有某个用例一直超时（单用例先执行排查）
- **检查**：Agent 会话数是否超过限制（默认每个用例创建新会话）
