## 一、业内实践与 AI RCA 整体架构

### 1\.1 业内 AI RCA 实践观察

调研数十家头部可观测厂商核心方案：

|厂商|核心 AI RCA 方案|
|---|---|
|Dynatrace|Grail 统一数据底座 \+ Smartscape 实时依赖图 \+ Causal AI 因果判断|
|PagerDuty / \[Incident\.io\]\(Incident\.io\)|告警归一化为 Incident 作为排障入口，深度集成 Slack 提供智能协作助手|
|Datadog|Bits AI SRE：生成故障假设 → 调用遥测数据 → 验证 / 推翻假设|
|Resolve AI|RCA 垂直工具，不持有监控 / 故障数据，串联多系统完成根因定位产品化|

**核心发现**
AI RCA 不能让 AI 自由探索（准确率极低），必须给 AI 提供**地图、元信息、标准化工具**，在受控环境内完成排障分析。

### 1\.2 AI RCA 整体架构数据底座

Workspace 统一聚合全维度数据输出故障排查结论，底层数据源包含：

1. **数据图谱**：Dynatrace Smartscape / Service Topology / Flashcat 灭火图 \(Firemap\)

2. **团队知识**：故障 SOP、静态依赖、排障 Skills 库

3. **可观测数据**：指标、日志、链路 Trace

4. **结构化资源数据**：CMDB、K8s、公有云资源信息

5. **变更事件数据**：二进制发布、配置、数据库、网络、运行时变更记录

6. **历史沉淀数据**：过往故障、复盘文档、根因案例库

## 二、RCA Layer1：Incident 层的 AI Agent 实践

### 2\.1 Incident 层定义

统一故障事件工作台，聚合全平台告警（ZABBIX、Skywalking、Sentinel 等），支持故障筛选、作战室创建、人员认领 / 协同。

### 2\.2 Incident 层两大核心职能

#### （1）故障治理职能

- 告警收敛：降噪、避免重复根因分析

- 归一化：统一告警严重等级、状态、语义结构

- 事件富化：补充标签、业务上下文

- 统一排障工作台：团队共享协作入口

#### （2）故障响应协作职能

围绕故障事件全流程自动化协同：

- 智能排班：精准匹配值班人，无打扰通知

- 故障认领 / 升级：保障事件闭环

- 告警静默 / 抑制 / 合并：智能降噪

- 自动建群、消息推送

- 故障自动总结、同步 Statuspage 公告

- 复盘沉淀：历史案例作为 AI 记忆库

> 关键：历史 Incident 复盘、相似故障案例是 AI RCA 核心参考素材，定期向向量库沉淀为 Agent Memory。
> 
> 

### 2\.3 Incident 层可落地 AI 能力（Flashduty 等 On\-call 产品）

AI 定位为**协作助手，不替代原有工作流**

1. **告警聚合**

    - 向量相似度匹配：无需规则，可解释弱

    - 规则匹配：可读性强，维护成本高

    - 落地现状：两种方案 5:5 并行使用

2. **自动复盘生成**
自动读取故障时间线、飞书 / Slack 聊天记录、事件全流程生成复盘文档

3. **通用 Assistant 能力**

    - 告警通知模板辅助编辑

    - 内部运维问答机器人

    - 代码生成运维文档

    - 知识库问答检索

### 2\.4 Incident 层作为 RCA 入口三大优势

1. **天然协同工作台**：团队共享排障技能、历史故障记忆，个人工具（Cursor/Claude）仅单人使用

2. **结构化标准对象**：`诊断Incident: 6a190e5e` 可直接提取故障时间、服务标签等结构化信息，远优于自然语言描述故障

3. **天然多数据源集成**：On\-call 产品原生对接数十类监控平台，集成链路可复用给 RCA 流程

## 三、RCA Layer2：O11y 层的 AI Agent 实践

### 3\.1 O11y 现状与痛点

O11y 包含三大类数据，行业普遍存在多系统并存、治理不完善问题：

1. **指标 Metrics**：Prometheus、VictoriaMetrics、Thanos；多套指标平台割裂

2. **日志 Logs**：ES、Loki、SLS、ClickHouse；仅做关键字检索，未发挥统计分析能力

3. **链路 Trace**：SkyWalking、Jaeger、ARMS；大量服务未接入链路追踪

**通用痛点**
数据分散、语义不统一、告警标准不一致、层级混乱；AI 跨平台取数成本高。

### 3\.2 核心关键结论

1. **存储统一不是刚需**：不同数据源 PromQL/LogQL/DSL 查询语言差异可被 LLM 兼容，仅跨库 Join 场景需要统一存储，占比极低

2. **语义规范统一远重于存储统一**：指标`service=payment`、日志`index=pay`需要额外映射知识库，语义不一致是 AI 最大障碍

### 3\.3 AI Agent 对 O11y 数据三大核心诉求

1. **索引信息**：从 Incident 提取 service、instance 等关键字段，作为检索 O11y 数据入口

2. **总 \- 分分层访问**：全局概览 → 单条明细；先大盘 SLI 指标，再下钻日志 / 链路详情

3. **全维度关联对象数据**

    - 当前故障服务自身指标日志

    - 依赖中间件、数据库、MQ、第三方接口

    - 底层宿主机、网络、K8s 运行环境

**最大挑战**：数据关联。生产环境链路不全、服务未埋点、字段不统一，依赖拓扑缺失。

### 3\.4 头部厂商 O11y 层 RCA 路线对比

1. **Dynatrace（理想建模路线）**
OneAgent 统一采集 → 标准化数据 → Smartscape 完整拓扑 → Causal AI 因果分析；局限：无法 100% 覆盖异构存量系统

2. **Datadog（务实假设验证路线）**
多线程并行验证故障假设，兼容生产环境数据不规范现状，灵活性更强

3. **Grafana（保守辅助路线）**
优先提供运维助手、标准化排查清单；强调过程可视化，拒绝输出错误根因结论

### 3\.5 核心洞察：海量数据，可枚举 Entity

- 指标 / 日志 / 链路是海量非结构化数据

- Entity（实体）是有限、可枚举单元：微服务、数据库实例、专线、业务功能

- Entity 具备可量化健康指标、可绑定告警、可关联日志链路

- 决定 Entity 健康的根因方向固定：流量突增、下游依赖异常、配置 / 代码变更

### 3\.6 Flashcat 灭火图（Firemap）：O11y 知识图谱载体

#### 实体分层结构

业务功能 → API 接口 → 微服务模块 → 容器 / K8s → 中间件 \(Redis/MySQL/Kafka\) → 宿主机 / 网络 / DNS

#### 灭火图四大上下文（AI RCA 完整信息骨架）

1. **对象上下文**：服务、数据库、缓存、主机等实体对象

2. **健康上下文**：成功率、延迟、错误率、慢查询、消息堆积

3. **关系上下文**：调用依赖、部署归属、链路拓扑

4. **数据路径上下文**：日志、Trace、大盘、变更事件下钻入口

> 设计价值：从海量监控数据提取骨架，同时给人类、AI 提供完整故障上下文。
> 
> 

### 3\.7 O11y 层 AI 输出：Investigation Package（故障调查包）

AI 不输出单一结论，输出完整证据链，包含 6 部分：

1. **事实**：异常实体、受影响业务范围

2. **时间线**：故障起止、同步变更事件

3. **候选根因**：按置信度排序，不确定则标注存疑

4. **证据**：关联指标、日志、链路、拓扑截图

5. **反证**：已排除的故障方向

6. **下一步操作**：指定实体深度验证方案

> 设计原则：诚实优先，AI 过度自信会导致错误归因；存疑场景必须明确标注。
> 局限：O11y 仅能输出候选根因，部分底层问题必须进入第三层实时诊断确认。

## 四、RCA Layer3：实时诊断层的 AI Agent 实践

### 4\.1 实时诊断层存在必要性

大量底层运行数据不会上报至 O11y 平台，必须直连机器 / 数据库现场取证：

- 文件 / 网络句柄占满：无法区分进程、句柄类型

- MySQL 锁等待：无法定位阻塞事务与 SQL

- Redis 大 Key 延迟：无法识别具体热点 Key

- 磁盘占用过高：无法定位大文件目录

- ES 分片异常：无法定位失败索引

### 4\.2 实时诊断层整体架构

AI Agent → 中心统一调用接口 → Websocket 穿透 → 机器本地 Agent → 结构化只读 Tool → 目标实例 \(主机 / 数据库 / MQ\)

#### 安全设计约束

1. 所有诊断能力**只读**，禁止修改操作

2. 命令白名单、AST 语法解析拦截高危 shell

3. 数据库凭据仅保存在本地 Agent，平台侧不存储密钥

4. 全操作审计留痕，同步超时、输出截断、数据脱敏

5. 默认不支持远端下发动态脚本（开放会大幅扩大攻击面）

### 4\.3 Tool 三层分层设计（颗粒度标准化）

单个 Tool 对应单一明确 SRE 诊断问题，避免大而全工具：

1. **Snapshot 快照工具（快速初筛）**
全局状态概览：`os.overview`、`mysql.overview`、`redis.overview`

2. **Focused 定向工具（验证假设）**
针对性深度诊断：`mysql.lock_contention`、`redis.slowlog`

3. **Flexible 灵活受控工具（长尾场景）**
受限 shell 执行 \(AST 白名单\)、SQL 解析查询，处理非标场景

### 4\.4 完整端到端 RCA 案例：MySQL 锁等待故障

1. Flashduty On\-call 平台接收故障 Incident：订单接口超时告警

2. Flashcat 灭火图：订单微服务、订单数据库实体标红

3. Trace 链路数据：请求阻塞在数据库调用阶段

4. O11y 指标大盘：数据库线程数、行锁等待指标持续上涨

5. 实时诊断 Agent 调用`mysql.lock_contention`工具现场取证

6. 最终输出完整根因信息：

    - 阻塞会话、源头阻塞事务

    - 冲突 SQL 与事务逻辑

    - 处置建议：Kill 会话、限流、回滚变更、持续观测

## 五、三层 AI RCA 整体总结

1. **Layer1 Incident（故障入口层）**
Flashduty On\-call 平台完成告警收敛、团队协作、故障沉淀；作为 AI RCA 统一入口，串联所有数据源。

2. **Layer2 O11y（证据分析层）**
Flashcat 可观测平台通过灭火图构建实体知识图谱，提供标准化监控证据链，输出候选根因与调查包。

3. **Layer3 实时诊断（现场验证层）**
分布式受控只读 Agent 工具，线下取证验证假设，补全监控系统缺失的底层运行数据。
