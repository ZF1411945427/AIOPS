# AI 智能助手 — 多轮对话测试场景

> 本文件定义 AI 智能助手在关联告警、关联资产、关联分析、知识自动沉淀四个场景下的多轮对话测试用例。

---

## 测试文件说明

本文件为**手工测试场景文档**，用于验证 AI 智能助手在以下四类核心场景中的对话能力。

| 场景 | 描述 | 验证要点 |
|------|------|---------|
| 关联告警 | 用户询问某告警，AI 主动关联同类告警、父子告警、拓扑上下游告警 | 告警数量/级别/资产是否准确关联 |
| 关联资产 | 用户询问某资产，AI 关联该资产的告警、指标、日志、链路、变更记录 | 关联信息是否完整、是否有遗漏 |
| 关联分析 | AI 主动调用 `query_correlation_analysis` 工具进行多维信号关联分析 | 工具调用、信号聚合、RCA 建议是否正确 |
| 知识自动沉淀 | 告警/故障解决后，AI 自动生成知识草稿并提交审批 | 草稿内容质量、关联准确性、审批流程 |

> 以上为 4 个核心正向场景（happy path）。以下场景 5-17 为扩展场景，覆盖模糊输入、异常路径、边界用例与复杂运维情境：

| 场景 | 类型 | 验证要点 |

| 8 多轮澄清 | 复杂 | 信息不足主动反问，避免无限澄清循环 |
| 9 告警风暴降噪 | 复杂 | 聚合/抑制/根因簇，不逐条刷屏 |
| 10 级联故障拓扑溯源 | 复杂 | 多跳下钻定位最底层根因节点 |
| 11 实时诊断与动态跟踪 | 复杂 | 在线跟踪增量变化，恢复确认 |

## 测试场景 1：关联告警多轮对话

### 场景目标

用户发现一个关键告警后，向 AI 助手咨询，AI 主动关联同资产、同服务、同时间窗口的其他告警，并给出处理建议。

### 测试数据准备

```
告警 A: ID=1, asset_id=server-001, severity=critical, service=payment,
        message="CPU 使用率超过 95%", created_at=当前时间-30min
告警 B: ID=2, asset_id=server-001, severity=warning, service=payment,
        message="内存使用率超过 80%", created_at=当前时间-25min
告警 C: ID=3, asset_id=database-01, severity=critical, service=payment,
        message="数据库连接数超限", created_at=当前时间-20min
告警 D: ID=4, asset_id=server-001, severity=info, service=payment,
        message="服务重启通知", created_at=当前时间-5min
```

### 多轮对话测试步骤

#### 第 1 轮：用户发起告警咨询

**用户输入：**
```
我刚才收到了一个 CPU 告警，发生在 server-001 这台服务器上，级别是 critical，这是什么情况？
```

**期望 AI 行为：**
- 理解用户描述的告警（CPU / server-001 / critical）
- 主动查询该资产的历史告警（最近 2 小时内）
- 展示关联告警列表（B: 内存 warning / D: 服务重启 info）
- 分析告警之间的时序关系（CPU 告警后出现内存告警，可能存在因果关系）
- 给出初步判断：可能是某个进程异常导致 CPU 飙升，进而影响内存

**期望输出示例：**
```
我查了一下 server-001 的历史告警，发现这个 CPU 告警不是孤立事件：

📊 同一资产关联告警（近 2 小时）：
| 告警ID | 级别   | 消息                    | 发生时间      |
|--------|--------|------------------------|---------------|
| #1     | 🔴严重 | CPU 使用率超过 95%      | 30 分钟前     |
| #2     | 🟡警告 | 内存使用率超过 80%       | 25 分钟前     |
| #4     | 🟢信息 | 服务重启通知             | 5 分钟前      |

🔍 告警关联分析：
1. CPU 和内存告警发生在 5 分钟内，可能由同一进程异常导致
2. 服务重启通知出现在 CPU 告警之后 25 分钟，可能是自动恢复或人工干预
3. 建议优先检查：进程列表 / 最近的变更记录 / 资源消耗峰值时刻的日志

需要我帮你进一步分析吗？
```

---

#### 第 2 轮：用户追问根因

**用户输入：**
```
能帮我分析一下这个告警的根因吗？可能是什么原因导致的？
```

**期望 AI 行为：**
- 结合告警的时序关系和资产信息，分析可能的根因
- 调取该资产的变更记录（如果有）
- 调取该资产的指标数据（CPU/内存/磁盘 I/O 的变化曲线）
- 给出 2-3 个候选根因，按置信度排序
- 提供下一步排查建议（查看日志 / 检查进程 / 确认变更）

**期望输出示例：**
```
基于告警时序和资产信息，我给出以下根因分析：

🎯 最可能根因（置信度 85%）：
进程异常导致资源耗尽
- 30 分钟前 CPU 突然飙升至 95%
- 紧接着内存告警（资源泄漏特征）
- 5 分钟后服务自动重启（可能是 OOM Kill）

🔍 支撑证据：
1. 告警级别从 critical → warning → info，符合"异常→触发保护→恢复"的典型模式
2. 告警集中在同一台服务器，不是批量问题
3. 服务重启后告警消失，初步判断为自愈

🛠 建议下一步：
1. 查看告警发生时刻的系统日志：`journalctl --since "30 minutes ago" | grep error`
2. 检查异常进程的启动时间和资源消耗
3. 确认是否有最近的变更记录（部署/配置修改）

需要我帮你查询日志吗？
```

- 返回草稿已创建的通知，包含草稿 ID 和待审批状态

**期望输出示例：**
```
好的，我来生成故障记录并提交到知识库...

✅ 故障知识已生成并提交审批！

📋 草稿摘要：
- 故障标题：server-001 CPU/内存告警分析与处理
- 涉及资产：server-001（payment 服务）
- 告警数量：3 条（1 严重 + 1 警告 + 1 信息）
- 根因：进程资源泄漏导致 OOM，触发服务自动重启
- 处理过程：观察 → 确认自愈 → 记录
- 持续时间：约 25 分钟

📌 知识草稿状态：待审批
   审批人：知识管理员
   审批入口：侧边栏 → 知识管理 → AI 知识草稿

你可以去审批入口确认内容后正式入库。
```

## 测试场景 2：关联资产多轮对话

### 场景目标

用户向 AI 询问某个资产的情况，AI 主动关联该资产的全量可观测数据（告警/指标/日志/链路/变更），给出一站式分析。

### 测试数据准备

```
资产：server-001（IP=10.0.1.101, ci_type=server, service=payment）
关联指标：
  - CPU: 最近 1 小时有 2 次 > 90% 峰值
  - Memory: 最近 1 小时稳定在 75%
  - Disk: 正常
关联日志（近 1 小时）：
  - ERROR 日志 12 条，主要为 "connection timeout"
  - WARN 日志 45 条
关联链路：
  - server-001 为 payment 服务的后端节点
  - 上游调用方：api-gateway
  - 下游依赖：database-01（数据库）
关联变更：
  - 30 分钟前有一次配置变更（修改了连接池大小）
```

### 多轮对话测试步骤

#### 第 1 轮：查询资产状态

**用户输入：**
```
server-001 这台服务器现在什么情况？帮我做个全面检查
```

**期望 AI 行为：**
- 解析资产名称/IP，定位到正确的资产记录
- 并行查询：告警 / 指标 / 日志 / 链路 / 变更
- 聚合所有数据，给出资产健康状态评分（0-100）
- 识别最严重的问题并优先展示

**期望输出示例：**
```
正在全面检查 server-001...

📊 资产基本信息：
- 名称：server-001
- IP：10.0.1.101
- 类型：服务器（Linux）
- 所属服务：payment（支付服务）
- 状态：✅ 运行中

🟢 指标状态（近 1 小时）：
| 指标     | 当前值 | 状态 |
|----------|--------|------|
| CPU      | 42%    | 正常 |
| 内存     | 75%    | 正常 |
| 磁盘     | 58%    | 正常 |
- CPU 历史峰值：92%（30 分钟前，已恢复正常）

🔴 告警状态（2 条）：
| 级别   | 消息               | 发生时间    |
|--------|-------------------|-------------|
| 🔴严重 | CPU 使用率超过 95% | 30 分钟前   |
| 🟡警告 | 内存使用率超过 80% | 25 分钟前   |

🔴 日志异常（近 1 小时）：
- ERROR：12 条（connection timeout 为主）
- WARN：45 条
- 日志样本：`[ERROR] 2026-07-17 14:30:01 - connection timeout to database-01`

🔗 拓扑关联：
- 上游：api-gateway（调用 server-001）
- 下游：database-01（被 server-001 调用）
- 下游错误率：database-01 错误率 3.2%（偏高）

📋 变更记录（近 24 小时）：
- 30 分钟前：配置变更（连接池大小 100→200）
- 2 小时前：应用部署（v2.3.1）

🏥 综合健康评分：72/100（⚠️ 中等）
主要扣分项：近期 CPU 峰值、日志 ERROR 偏多、下游数据库错误率高

建议优先关注：
1. database-01 的错误率问题（可能导致 server-001 的 connection timeout 日志）
2. 30 分钟前的配置变更是否有问题
```

---

#### 第 2 轮：追问下游依赖问题

**用户输入：**
```
database-01 错误率偏高是怎么回事？跟 server-001 有关吗？
```

**期望 AI 行为：**
- 聚焦 database-01 的详细状态
- 分析 server-001 → database-01 的调用关系
- 展示 database-01 的错误日志、响应时间分布
- 判断是否为 server-001 的异常查询导致数据库压力

**期望输出示例：**
```
让我详细查一下 database-01 的情况...

📊 database-01（数据库）详细状态：
- 类型：MySQL 8.0
- 状态：⚠️ 亚健康（错误率 3.2%）
- 当前连接数：485/500（偏高）

🔗 调用链路分析：
server-001 --(读/写)--> database-01
  ↑
  └── api-gateway（上游）

📈 关联分析：
1. server-001 的 12 条 connection timeout 日志
   时间与 database-01 连接数峰值高度吻合（相关系数 0.91）
2. 30 分钟前 server-001 配置变更（连接池 100→200）
   → 同时刻 database-01 连接数从 320 跳升至 485
3. 数据库连接数接近上限（500），部分请求因获取不到连接超时

🎯 根因判断：
配置变更后 server-001 连接池扩大 2 倍，但 database-01 最大连接数未调整，
导致数据库连接资源竞争，部分请求超时。

🛠 建议操作：
1. 立即：临时扩大 database-01 max_connections（建议 800）
2. 短期：评估是否需要扩大数据库实例规格
3. 长期：将连接池策略改为动态自适应

需要我帮你在知识库里搜索类似的故障案例吗？
```

## 测试场景 3：关联分析多轮对话

### 场景目标

用户请求对某个服务或资产进行关联分析，AI 主动调用 `query_correlation_analysis` MCP 工具，整合告警/指标/日志/链路四类信号，给出 RCA 建议。

### 多轮对话测试步骤

#### 第 1 轮：发起关联分析请求

**用户输入：**
```
帮我分析一下 payment 服务最近 2 小时的情况，看看有没有什么问题
```

**期望 AI 行为：**
- 识别服务名称（payment）
- 识别时间范围（最近 2 小时）
- 主动调用 MCP 工具 `query_correlation_analysis(hours=2, service="payment")`
- 解析工具返回结果（告警数/指标异常数/日志异常数/链路错误率/关联资产/RCA 建议）
- 以结构化方式展示分析结果

**期望输出示例：**
```
正在对 payment 服务进行多维关联分析...

🔍 分析范围：payment 服务，近 2 小时

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📊 关联分析结果（信号聚合）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

📌 告警（3 条）：
| 资产         | 级别   | 消息                   | 时间     |
|--------------|--------|------------------------|----------|
| server-001   | 🔴严重 | CPU 使用率超过 95%      | 30 分钟前 |
| server-001   | 🟡警告 | 内存使用率超过 80%      | 25 分钟前 |
| database-01  | 🔴严重 | 数据库连接数超限        | 20 分钟前 |

📈 指标异常（5 个）：
- server-001 CPU：峰值 95%（30 分钟前），已恢复正常
- server-001 内存：峰值 83%（25 分钟前），已下降至 75%
- database-01 连接数：峰值 490/500（20 分钟前），当前 485
- database-01 错误率：3.2%（当前仍偏高）
- payment 服务 P99 延迟：从 120ms 上升至 380ms（30 分钟前）

📝 日志异常（2 类）：
- ERROR：connection timeout（12 条，与 database-01 错误率峰值吻合）
- WARN：slow query > 1s（8 条，主要发生在 payment-api 连接数据库时）

🔗 链路异常（1 条）：
- payment-api → database-01：错误率 3.2%，P99 延迟 890ms
- 慢调用链：server-001 → database-01（单次最高 2.1s）

🏢 关联资产（4 个）：
server-001（严重）、database-01（严重）、server-002（正常）、api-gateway（正常）

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🎯 RCA 根因分析（置信度排序）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1️⃣ 根因 #1（置信度 88%）：
配置变更触发连接池耗尽
- 30 分钟前 server-001 连接池从 100→200
- database-01 max_connections 仍为 500
- 峰值连接数 490/500，触发"连接数超限"告警
- 后续引发 connection timeout 日志和 P99 延迟上升

2️⃣ 根因 #2（置信度 65%）：
数据库查询性能下降
```

## 测试场景 10：跨服务级联故障拓扑溯源（复杂）

### 场景目标

验证 AI 能沿调用链多跳溯源，从末端服务故障定位到根因服务，而非停留在表面现象。

### 测试数据准备

```
调用链：api-gateway → payment → database-01 → redis-01
故障源：redis-01 内存溢出（OOM）
表现：payment P99 飙升 → database-01 慢查询 → redis-01 宕机
```

### 多轮对话测试步骤

#### 第 1 轮：末端故障报告

**用户输入：**
```

```

**期望 AI 行为：**
- 不止查 payment 本身，沿调用链下钻
- payment → database-01 → redis-01，多跳溯源
- 识别 redis-01 为根因节点（最底层异常）
- 给出完整拓扑路径

**期望输出示例：**
```
payment P99 2s，我沿调用链下钻定位：

🔗 拓扑溯源路径：
api-gateway → payment（P99 2s）
              └→ database-01（慢查询 1.8s）
                  └→ redis-01（❌ 连接失败，OOM）

🎯 根因定位：redis-01 内存溢出（OOM）
- redis-01 当前内存：7.8G / 8G（97%，已触发淘汰）
- redis-01 告警：内存超限 + 连接拒绝
- 故障传播：redis-01 宕 → database-01 缓存未命中→ 慢查询 → payment P99 飙升

沿链路越往下越接近根因，redis-01 是最底层故障源。

需要我看 redis-01 的详细状态吗？
```

---

#### 第 2 轮：追问根因节点详情

**用户输入：**
```
redis-01 怎么了？详细说说
```

**期望 AI 行为：**
- 查询 redis-01 指标/日志/告警
- 给出 redis-01 故障详情与处理建议

**期望输出示例：**

```
📊 指标：

- 内存超限（critical）
- 连接拒绝激增（warning）

[ERROR] maxmemory policy: allkeys-lru triggered 1240 times

🎯 判断：缓存数据量超 maxmemory，LRU 淘汰跟不上 → 大量 key 被淘汰 → 命中率暴跌 → 上游全部回源数据库

🛠 建议：
1. 立即：扩容 redis-01 maxmemory 或扩容实例
```

## 验收标准

### 关联告警场景
- [ ] AI 能准确识别用户描述的告警（资产/级别/类型）
- [ ] 主动查询并展示同资产/同时段的其他告警
- [ ] 告警数量和级别与数据库实际数据一致
- [ ] 能分析告警之间的时序关系和因果关系
- [ ] 给出有参考价值的处理建议

### 关联资产场景
- [ ] 能根据资产名称或 IP 精确定位资产
- [ ] 并行查询告警/指标/日志/链路/变更，数据聚合完整
- [ ] 给出资产健康评分（0-100）及扣分原因
- [ ] 能分析上下游依赖关系和调用链路
- [ ] 能调用知识库搜索相似案例

### 关联分析场景
- [ ] AI 主动调用 `query_correlation_analysis` MCP 工具
- [ ] 正确聚合告警/指标异常/日志异常/链路异常四类信号
- [ ] 按置信度输出 RCA 根因建议（Top 3）
- [ ] 结合变更记录进行时序分析
- [ ] 给出可操作的下一步建议

### 知识自动沉淀场景
- [ ] 能从故障单和对话内容中提取知识要素（现象/根因/方案）
- [ ] 生成的知识草稿结构完整（标题/分类/标签/内容）
- [ ] 能正确调用知识创建接口并提交审批
- [ ] 能搜索知识库并返回相关知识列表
- [ ] 知识搜索支持关键词和分类过滤

### 模糊指代与输入纠错场景（场景 5）
- [ ] 对模糊指代（无 ID/资产名）能查询候选并让用户选择，不直接拒绝
- [ ] 对拼写不规范的资产名（少横杠/中文/下划线）能容错匹配
- [ ] 能解析口语化时间表达（"刚才"/"昨天那会儿"）
- [ ] 不因输入不完整而报错

### 闲聊与单点查询场景（场景 6）
- [ ] 纯打招呼不触发任何 MCP 工具调用
- [ ] 单点事实查询不误触发重型关联分析工具
- [ ] 闲聊追问能复用上下文，不重复查全量数据

### 空结果与异常输入场景（场景 7）
- [ ] 查询不存在的资产时明确提示，不编造数据
- [ ] 知识库空搜索时明确告知无匹配，不返回幻觉内容
- [ ] 无效时间窗时提示可查范围，不报错
- [ ] 不返回空表假装有结果

### 多轮澄清场景（场景 8）
- [ ] 信息严重不足时主动反问关键问题，不盲目调用工具
- [ ] 部分信息时按合理假设查询并标注假设，避免无限澄清循环
- [ ] 信息完整后正常执行分析

### 告警风暴降噪场景（场景 9）
- [ ] 能识别告警风暴模式（短时大量告警）
- [ ] 按资产/类型/时间窗聚合，抑制重复告警
- [ ] 输出 Top N 根因簇，而非逐条罗列
- [ ] 支持展开被抑制告警明细，支持分页

### 级联故障拓扑溯源场景（场景 10）
- [ ] 能沿调用链多跳下钻，不止停留在表面服务
- [ ] 识别最底层根因节点
- [ ] 给出完整拓扑路径与故障传播链
- [ ] 追问根因节点时返回详细状态

### 实时诊断与动态跟踪场景（场景 11）
- [ ] 识别故障未解决状态，支持实时跟踪
- [ ] 间隔追问时对比上一轮，聚焦增量变化
- [ ] 故障恢复时验证指标并确认结束时间
- [ ] 主动建议生成知识沉淀

### 工具调用失败降级场景（场景 12）
- [ ] 重型工具超时时明确提示，不假装成功
- [ ] 提供降级方案（轻量查询拼接）
- [ ] 工具报错时不编造数据，建议重试
- [ ] 不因工具失败而崩溃或返回 500

### 用户纠正根因场景（场景 13）
- [ ] 接受用户纠正作为新证据，不固执己见
- [ ] 重新查询验证并修正根因判断
- [ ] 基于新根因继续推进（如回滚建议）
- [ ] 致谢/确认用户补充信息

### 权限与脱敏场景（场景 14）
- [ ] 越权查询时明确拒绝并说明，不绕过权限
- [ ] 配置中密码/Token/密钥自动掩码
- [ ] 日志中凭证/连接串自动打码
- [ ] 保留非敏感字段便于排查

### 对比分析与趋势预测场景（场景 15）
- [ ] 支持横向对比（A vs B），结构化对比表
- [ ] 支持纵向趋势（本周 vs 上周），计算变化率
- [ ] 基于历史外推做容量预测
- [ ] 给出可操作的扩容建议

### 上下文跳跃与多人协同场景（场景 16）
- [ ] 识别话题切换信号，不混淆上下文
- [ ] "回到刚才"能正确指代原话题
- [ ] 接受多人补充信息作为新证据
- [ ] 补充信息能更新根因判断

### 知识冲突与误报识别场景（场景 17）
- [ ] 识别知识库矛盾建议并明确标注冲突
- [ ] 结合资产现状给出加权建议，不盲目采纳
- [ ] 对比告警值与实测指标，识别背离
- [ ] 判断误报并建议静默/检查采集器，不按错误告警排查

## 测试场景 20：自愈闭环验证——自动恢复后的根因确认（复杂）

### 场景目标

验证 AI 在系统自动恢复（自愈）后，能主动分析自愈是否真正解决了根因，还是只是暂时掩盖了症状。这是自动化运维中"信任但验证"的核心能力。

### 测试数据准备

```
线上故障记录：
- 时间线：
  14:00:00 — server-001 CPU 95% 告警触发
  14:00:15 — 自动自愈流程触发（restart nginx）
  14:00:45 — nginx 重启完成，CPU 降至 40%
  14:01:00 — 告警自动恢复
- 当前时间：14:30（故障已自动恢复 30 分钟）
- server-001 CPU 当前：42%（正常范围）
- 但 nginx 错误日志显示：重启前有大量 "worker_connections exceed" 警告
- 该告警在过去一周内触发了 4 次，每次都通过自愈恢复
```

### 多轮对话测试步骤

#### 第 1 轮：查看自愈恢复后的总结

**用户输入：**
```
刚收到通知，server-001 的 CPU 告警已经自动恢复了。
但我看到这个告警这周已经出现了好几次，是不是自愈只是临时解决了？
```
