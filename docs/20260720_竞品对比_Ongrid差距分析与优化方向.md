## P2 Edge Agent 反向隧道（已完成 2026-07-20）

### 架构
```
浏览器 ─(WebSocket)─> 云端 webssh.py ──┐
                                       ├─(内存查找)─> edge agent ─> 本地 PTY
HTTP /edge/command ─> _PENDING_COMMANDS ─┘                    ↑
                                  edge agent HTTP 轮询获取 ────┘
                                  edge agent ─(WebSocket)─> 云端 exec_result
```

### 关键设计决策
1. **HTTP 轮询模式**（非 WebSocket 推送）：云端把命令放入 `_PENDING_COMMANDS` 内存队列，edge agent 每 1s HTTP 轮询 `/edge/commands/pending` 获取
   - 原因：Starlette/uvicorn 在 Windows 上 WebSocket 跨协程 `send_json` 在 `receive_text` 等待时不能立即发送到 TCP（已知问题），改用 HTTP 轮询避开
   - 结果：命令执行延迟 < 200ms（含网络往返）
2. **WebSocket 仅用于 receive**：edge agent 通过 WebSocket 回传 `exec_result` / `pty_output`，云端 WebSocket handler 只做 receive + 转发
3. **PTY 交互式终端**：WebSSH → `pty_open` → edge agent 启动本地 PTY（Linux `pty.openpty` + bash，Windows 退化为 cmd）→ `pty_output` 回传 → 浏览器 xterm.js 渲染
4. **心跳监控**：edge agent 每 30s 发心跳，云端后台线程扫描 >90s 无心跳的 session 标记 offline
5. **断线重连**：edge agent 指数退避（1s → 60s），重连后自动重新注册
6. **全程审计**：所有命令执行写入 `edge_command_logs` 表（user/command/exit_code/stdout/stderr/duration/status）

### 已完成 P0-2:PromQL 子集解析器

**做了什么:**
- 新增 `app/services/promql_parser.py`(PromQL 子集解析器)
- `query_metrics` 工具新增 `promql` 参数,支持两种模式:
  - 字段模式(兼容):`metric_name + asset_id + hours + limit`
  - PromQL 模式(新):`promql="topk(3, cpu_usage)"`
- 支持的 PromQL 语法:
  - `metric_name` — 简单指标
  - `metric_name{asset_id="1"}` — 标签过滤
  - `topk(N, expr)` / `bottomk(N, expr)` — Top/Bottom N
  - `rate(expr[5m])` — 速率
  - `avg_over_time(expr[1h])` / `max_over_time` / `min_over_time` / `sum_over_time` — 窗口聚合
  - `avg(expr)` / `max` / `min` / `sum` / `count` — 全资产聚合
  - `topk(3, rate(cpu_usage[1h]))` — 嵌套聚合(先算 rate 再取 topk)
- 解析失败返回明确 error(不支持二元运算/直方图/子查询)

**入口位置:**
- AI 智能助手对话中,Agent 调用 `query_metrics` 工具时传 `promql` 参数
- 工具描述已更新,LLM 可自主构造 PromQL 表达式
- 后端文件:`app/services/promql_parser.py` + `app/services/mcp_tools.py:query_metrics`

## 三、差距排序(从大到小)

### 差距 1:设备纳管架构(最大差距)

**Ongrid 做法**
- 主机装轻量 edge agent,主动向云端拨出一条加密连接(反向隧道)
- 主机零监听端口,防火墙入站规则可全关,22/80/443 一个都不用开
- 浏览器点"终端"按钮直接弹真实 SSH 会话,无跳板机、不分发密钥、全程审计
- 攻击面从"每台机器一堆开放端口"收敛成"一条出站连接"

**本系统现状**
- 云端主动 SSH 连主机(`ssh_helper.get_ssh_client` → `ssh.connect(host, port=22)`)
- 主机必须开放 22 端口(或 WinRM 5985/5986),防火墙需放行入站
- 容器/K8s Pod 有 WebSSH(`containers.py:157`、`k8s_resources.py:1301`),**主机无浏览器 SSH**
- 混沌/诊断工具执行靠 paramiko 主动连,每台主机都是攻击面

**改造方向**
1. 开发 edge agent(Python 轻量守护进程,打包成 systemd service 或单二进制)
2. edge agent 启动时主动 WebSocket 拨出到云端(`wss://cloud/edge/connect`),建立反向隧道
3. 云端维护 `edge_session` 表(设备 ID + 隧道 ID + 心跳 + 在线状态)
4. WebSSH 前端 → 云端 → 反向隧道 → edge agent → 本机 `bash`/`powershell`
5. 主机侧零监听,所有命令走已建立的反向隧道
6. 全程审计:命令日志写入 `edge_command_log` 表

**涉及代码**
- 新增 `edge_agent/`(独立子项目,Python 单文件 + requirements)
- 新增 `app/routers/edge_tunnel.py`(云端隧道管理 API)
- 新增 `app/routers/webssh.py`(浏览器 WebSSH WebSocket)
- 改造 `app/models.py` 加 `EdgeSession`、`EdgeCommandLog` 模型
- 改造 `frontend/src/views/AssetsView.vue` 加"终端"按钮

**工作量评估**:大(2-3 周),涉及新子项目 + 协议设计 + 隧道稳定性

**Ongrid 做法**
- Coordinator 协调者接收任务
- 分派给 SRE / 网络 / 数据库子专家 Agent
- 每个子专家有自己的工具集和 System Prompt
- 时间线可审计、可复盘

**本系统现状**
- 单 Agent + 45 个工具(`mcp_tools.py`),Agent 自己选工具
- 无子专家路由,所有任务由同一个 Agent 处理
- `agent_sse.py` max_rounds=15 单轮编排

**改造方向**
1. 定义子专家配置(SRE/网络/数据库/中间件/K8s)
2. 每个子专家绑定工具子集 + 专属 System Prompt
3. Coordinator Agent 第一步:判断任务域 → 选子专家
4. 子专家执行 → 结果回传 Coordinator → Coordinator 汇总
5. 时间线记录每步分派决策

**涉及代码**
- 新增 `app/services/sub_agent_registry.py`(子专家注册)
- 改造 `app/services/agent_service.py` 加 Coordinator 编排逻辑
- 改造 `app/routers/agent_sse.py` 加子专家分派事件类型
- 新增 `app/models.py` `SubAgent` 模型(name/domain/tools/system_prompt)

**工作量评估**:中(1-2 周),主要是编排逻辑 + 子专家配置

---

### 差距 4:可观测全家桶(标准栈)

**Ongrid 做法**
- 内置 Prometheus + Grafana + Loki + Tempo
- Agent 自己写 PromQL(`topk(8, ...)`)、LogQL
- 面板标题挂真实查询语句
- 一键跳 Grafana 深度分析

**本系统现状**
- 自建指标采集 `metric_v2_service.py`,按字段查不按 PromQL
- 无内置 Prometheus/Grafana/Loki/Tempo
- Agent 调 `query_metrics` 工具按字段查,不会写 PromQL

**改造方向**(两种路线)
- A 路线(轻量):保持自建采集,新增 PromQL 解析器,让 `query_metrics` 支持 PromQL 子集
- B 路线(重量):集成 Prometheus + Grafana,指标双写,Agent 走 PromQL
- 推荐 A 路线(打磨期不引入新组件)

**涉及代码**
- 改造 `app/services/mcp_tools.py` `query_metrics` 加 PromQL 解析
- 新增 `app/services/promql_parser.py`(PromQL 子集解析 → 内部指标查询)

**工作量评估**:小到中(3-5 天,A 路线)

### 差距 5:工具安全元数据(Capability Metadata)

**Ongrid 做法**
- 33 个工具每个标 `safe`、运行位置(设备端/云端)、是否"仅 AI 调用"
- 沙箱化只读策略(`host_bash` 跑只读 shell)
- 全程审计

**本系统现状**
- 45 个 MCP 工具有 `display_name`、`description`,无 safe/位置/仅 AI/只读元数据
- `AgentCapabilitiesView.vue` 可视化管理但不区分安全等级
- 危险命令黑名单 52 条(`AGENTS.md` 提到),但工具本身不标 safe

**改造方向**
1. `@register_mcp_tool` 装饰器加参数:`safe=True/False`、`location="cloud"/"edge"`、`ai_only=True/False`、`read_only=True/False`
2. `MCPTool` 数据结构加这四个字段
3. Agent 调用前校验:safe=False 工具需人工确认
4. `AgentCapabilitiesView.vue` 展示这些元数据(图标/标签)
5. 前端配置页支持按安全等级过滤

**涉及代码**
- 改造 `app/services/mcp_registry.py` `register_mcp_tool` 加参数
- 改造 `app/services/mcp_tools.py` 45 个工具补元数据
- 改造 `frontend/src/views/AgentCapabilitiesView.vue` 展示元数据

**工作量评估**:小(2-3 天),主要是补元数据 + UI

## 五、专业名词学习表

| 大白话 | 专业名词 |
|--------|---------|
| 主机主动连云端,不用开端口 | Reverse Tunnel(反向隧道) |
| 主机一个入站端口都不开 | Zero-inbound-port(零入站端口) |
| 只允许出站连接 | Outbound-only(仅出站) |
| 云端边的轻量小程序 | Edge Agent(边缘代理) |
| 群里既能收告警又能发指令 | Bidirectional IM Channel(双向即时通信通道) |
| 聊天式运维 | ChatOps |
| 一个大脑分派给多个小专家 | Multi-Agent Orchestration(多智能体编排) |
| 子智能体路由 | Sub-agent Dispatch |
| 工具标明安不安全/在哪跑/只读 | Capability Metadata(能力元数据) |
| 工具在隔离环境跑 | Tool Sandboxing(工具沙箱化) |
| 只能查不能改的策略 | Read-only Policy(只读策略) |
| 监控日志链路全家桶 | Observability Stack(可观测性栈) |
| 改配置不用重启 | Hot-swap(热切换) / Hot-reload(热重载) |
| Agent 每步决策可追溯 | Auditability(可审计性) / Reproducibility(可复现性) |

## 九、结论

本系统在**知识库/SOP 数量、灭火图分层健康、工具数量、安全加固、移动端、混沌工程、智能巡检**七方面已对齐或强于 Ongrid。

主要差距集中在**设备纳管架构(反向隧道)、IM 双向通道、子专家分派、可观测标准栈、工具安全元数据**五项,按 P0/P1/P2 分级推进。

本系统定位为闭源商业产品(License 硬件绑定),与 Ongrid 开源自托管路线不同,差距不等于劣势,但**反向隧道 + ChatOps** 是运维平台的核心体验差异,建议长期规划。
