> 目标：解决可观测性维度的核心瓶颈（SQLite + 轮询），预计可观测性从 7.0 → 9.0

**VictoriaMetrics 已就绪**：二进制已放入 `bin/`，`run.py` 启动时自动拉起子进程（端口 8428），后端随 FastAPI 一起管理生命周期。

### 3.1 指标时序存储迁移（VictoriaMetrics / ClickHouse）

**现状**: `MetricRecord` 表存于 SQLite，无分区、无归档，高并发写入冲突；VictoriaMetrics 进程已就绪（端口 8428）

**目标**: 专业时序库存储，支持百万级指标/秒写入

**架构变更**:
  原: MetricCollector → SQLite (MetricRecord)
  新: MetricCollector → VictoriaMetrics (主存储) + SQLite (元数据/模板)

| 步骤 | 内容 | 状态 |
|------|------|------|
| 3.1.1 | VictoriaMetrics 下载并嵌入 run.py 子进程管理 | ✅ 已完成 |
| 3.1.2 | 新建 `metric_v2_service.py`，实现 VM 写入/查询封装 | ❌ |
| 3.1.3 | 修改 `metric_collector.py`：采集后双写（VictoriaMetrics + SQLite 模板表） | ❌ |
| 3.1.4 | 新增 `/metrics/api/v2/query` 接口，使用 VictoriaMetrics PromQL 查询 | ❌ |
| 3.1.5 | 前端 `MetricsView.vue` 适配新 API | ❌ |
| 3.1.6 | 时序数据自动归档策略：>90天降采样到 1h 粒度 | ❌ |

**新增文件**:
- `app/services/metric_v2_service.py` — VictoriaMetrics 读写封装
- `app/routers/metrics_v2.py` — 新版指标 API（PromQL 兼容）
- `docker-compose.monitoring.yml` — VictoriaMetrics + vmagent 编排

**预计效果**:
- 写入性能：从 SQLite ~500 ops/s → VictoriaMetrics ~100,000 ops/s
- 查询性能：从全表扫描 ~2s → 时序索引 ~50ms
- 存储成本：自动压缩，1 年数据仅占原始 1/10

## 十二、技术选型参考

| 需求 | 推荐方案 | 备选方案 | 选择理由 |
|------|---------|---------|---------|
| 时序存储 | VictoriaMetrics | InfluxDB / TimescaleDB | 性能最优、PromQL 兼容、资源占用低 |
| 日志存储 | ClickHouse | Elasticsearch | 列式存储压缩比高、SQL 查询、成本低 |
| 消息队列 | Redis Streams | Kafka / RabbitMQ | 轻量、已有 Redis 依赖、足够此规模 |
| 缓存 | Redis | Memcached | 功能更全、支持数据结构 |
| 拖拽仪表盘 | gridstack.js | vue-grid-layout | 轻量、响应式、社区活跃 |
| 国际化 | vue-i18n | 自研 | Vue 官方推荐、生态成熟 |
| 移动推送 | FCM + 自建网关 | 极光/个推 | 无第三方依赖、私有化部署 |
| 数据库迁移 | Alembic | 自研幂等脚本 | SQLAlchemy 官方工具、版本管理 |

### 改进项详情

**知识图谱推理引擎** — 三大推理能力

1. **故障传播分析 (Impact Analysis)**
   - 算法：BFS 沿依赖图（`depends_on`/`parent`）向下游传播，传播概率衰减因子 0.7^depth
   - 输入：故障源 `asset_id` + 传播深度 (1-5)
   - 输出：受影响资产列表（影响概率/影响评分/告警关联）+ 传播路径可视化 + 智能建议
   - API：`GET /knowledge/graph/api/impact-analysis?asset_id=&depth=`

2. **根因定位推理 (Root Cause Inference)**
   - 算法：networkx 构建拓扑图 → 三维评分融合
     - PageRank（图中心性，权重 0.4）
     - 入度中心性（被依赖越多越可能根因，权重 0.25）
     - 告警传播评分（子树告警数加权衰减，权重 0.35）
   - 输入：`alert_ids` 或 `asset_ids`（留空自动取最近 24h 告警）
   - 输出：根因候选列表（置信度高/中/低）+ 推理依据 + 拓扑传播路径
   - API：`POST /knowledge/graph/api/root-cause`

3. **知识推荐推理 (Knowledge Recommendation)**
   - 四条推理链路：
     - 告警 → 资产 → ci_type → Runbook(asset_type 匹配)
     - 告警 → metric_name → KnowledgeBase(tags/symptom 关键词重叠)
     - 资产 → 依赖链 → 上下游历史 RCA 知识
     - 资产 → 同类型资产历史告警 → 相似故障知识
   - 输入：`alert_id` 或 `asset_id`
   - 输出：推荐知识/Runbook/历史故障列表 + 关联推理路径 + 图谱上下文
   - API：`GET /knowledge/graph/api/recommend?alert_id=&asset_id=&limit=`
