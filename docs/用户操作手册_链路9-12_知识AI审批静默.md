# AIOps 用户操作手册 — 链路9-12：知识沉淀 & AI检索 & 审批流 & 静默管理

> **适用角色**：知识管理员、SRE、运维工程师
> **前置条件**：已登录系统，拥有相关菜单权限
> **预计耗时**：约 15-20 分钟

---

## 概述

本手册覆盖四条链路：
- **链路9**：故障自动沉淀知识（LLM 生成知识草稿 + 版本追踪）
- **链路10**：AI 直接检索知识（RAG 问答 + 统一搜索）
- **链路11**：故障内部审批流（状态机）
- **链路12**：告警静默管理

> **入口补充**：知识管理相关页面入口位于 **侧边栏 → 中级运维 → 知识管理**，包括「知识文档」(kb-documents)、「知识图谱」(kb-graph)、「智能推荐」(smart-recommend) 三个子页面。

> **链路9 知识沉淀**　①触发(后台) → ②查看草稿 → ③审批草稿 → ④查看版本
> **链路10 AI检索**　①AI问答 → ②采纳为知识 → ③统一知识搜索
> **链路11 审批流**　①创建故障单 → ②提交审批 → ③审批通过 → ④驳回
> **链路12 静默**　①创建规则 → ②静默生效 → ③到期恢复 → ④取消静默 → ⑤查看
> **菜单指引**　知识文档/知识图谱/智能推荐入口位于「中级运维→知识管理」
>

# 链路9：故障自动沉淀知识

## 第一步：触发知识生成（后台自动）

### 说明
故障关闭后，系统**自动触发**知识沉淀，无需手动操作。

### 沉淀过程

1. **触发时机**：故障状态变为 `resolved` 后 5 分钟内
2. **自动归档**：后台任务调用 `knowledge_autogen_service.generate_draft(alert_id, db)`
3. **LLM 生成**：
   - 查询故障单详情（`Incident`）+ 关联告警（`IncidentAlert`）+ 关联资产（`Asset`）
   - 构造 Prompt 调用 LLM 生成结构化知识：
     - `title`：故障标题衍生（如 "生产环境 CPU 使用率过高"）
     - `symptom`：告警 message 汇总（如 "CPU 使用率持续超过 90%"）
     - `root_cause`：RCA 根因分析（如 "nginx 进程 CPU 占用过高"）
     - `solution`：修复建议（如 "重启 nginx 服务"）
     - `tags`：从告警指标名/资产类型提取（如 `["cpu", "nginx", "production"]`）
     - `severity`：继承故障级别（如 `critical`）
     - `asset_type`：继承资产类型（如 `server`）
4. **写入草稿**：创建 `KnowledgeDraft` 记录：
   - `status`：`pending`
   - `source_type`：`auto`（AI 自动生成）
   - `sop_steps`：LLM 生成的 SOP 步骤（JSON 数组）

### 看到什么
- 无需手动查看
- 生成的知识草稿出现在「AI 知识草稿」页面

## 第三步：审批知识草稿

### 操作页面
**侧边栏** → **知识库** → **AI 知识草稿**

### 操作步骤

# 链路10：AI 直接检索知识

## 第一步：发起 AI 问答

### 操作页面
**侧边栏** → **AI运维智能体** → **AI 智能助手**

### 操作步骤

1. **输入问题**
   - 在聊天输入框输入问题（如 "CPU 使用率过高怎么处理？"）
   - 点击发送或按回车

2. **AI 检索知识**
   - 系统自动调用 `query_knowledge_rag` 工具
   - **TF-IDF 向量化**：
     - 中文按字分词，英文/数字按词分词
     - 计算 IDF（逆文档频率）
     - 构建 TF-IDF 稀疏向量
   - **相似度计算**：
     - 遍历所有 `KbChunk` 计算余弦相似度
     - 过滤相似度 < 0.05 的结果
     - 按相似度降序取 Top-5
   - **结果注入 LLM**：
     - 检索到的知识内容注入 `runtime_context.messages`
     - LLM 根据知识内容生成回答

3. **查看回答**
   - 聊天界面显示 AI 回答
   - 回答中引用知识库条目（如 "根据知识库《XXX》..."）
   - 回答下方显示「采纳为知识」按钮

### 看到什么
- AI 回答基于检索到的知识库内容
- 回答中引用知识来源
- 回答下方有「采纳为知识」按钮

## 第二步：采纳为知识

### 操作页面
**侧边栏** → **AI运维智能体** → **AI 智能助手**

### 操作步骤

1. **点击「采纳为知识」**
   - 在 AI 回答下方点击「采纳为知识」按钮
   - 系统调用 `create_knowledge_draft`
   - 将当前对话内容转为知识草稿：
     - `title`：问题摘要
     - `symptom`：问题描述
     - `solution`：AI 回答内容

2. **审批入库**
   - 草稿出现在「AI 知识草稿」页面
   - 审批人阅读后通过 → 转为正式 `KnowledgeBase`

### 看到什么
- Toast 提示「已生成知识草稿，等待审批」
- 草稿出现在「AI 知识草稿」页面

AI 回答可被采纳为知识，进入审批流程。

## 第三步：统一知识搜索

### 操作页面
**侧边栏** → **AI运维智能体** → **AI 智能助手**（或直接访问搜索入口）

### 说明
统一搜索（`GET /knowledge/api/unified-search`）可同时检索知识库（KnowledgeBase）、知识文档（KbDocument）、Runbook 三种知识来源，支持文本模糊匹配和向量语义检索两种模式。

### 操作步骤

1. **在 AI 助手中使用统一搜索**
   - 在聊天输入框输入含搜索意图的问题，如 `"统一搜索：nginx 故障处理方案"`
   - AI 助手自动调用 `unified_search` 工具
   - 系统同时检索：
     - **知识库**：`KnowledgeBase` 表的 title/symptom/solution 字段 LIKE 匹配
     - **知识文档**：`KbDocument` 表的 title/content 字段 LIKE 匹配
     - **Runbook**：`Runbook` 表的 title/steps 字段 LIKE 匹配

2. **切换搜索模式**
   - **文本模式**（`mode=text`）：使用 SQLite LIKE 模糊匹配，速度快
   - **向量模式**（`mode=vector`）：使用 Milvus 向量检索，语义理解更准确

3. **查看搜索结果**
   - 搜索结果按相关性排序，每项显示：
     - **来源类型**：知识库 / 文档 / Runbook
     - **标题**：匹配条目的标题
     - **内容片段**：匹配内容的高亮摘要
     - **相关性评分**：匹配程度的分数

### 看到什么
- 搜索结果在 AI 对话中展示
- 多来源结果统一呈现，来源类型以标签区分
- 向量模式下语义匹配更准确，能处理同义词和近似表达

### 结果
统一搜索闭环：搜索请求 → 多来源检索 → 结果聚合 → 回答用户。

# 链路12：告警静默管理

## 说明
告警静默支持两种模式：
1. **即时静默**：通过 `POST /alerts/{alert_id}/silence` 设置 `AlertSilence.expires_at = now + minutes`，过期自动失效
2. **定时静默计划**：通过 `AlertSilenceSchedule` 表配置 cron 表达式实现周期静默，`duration_minutes` 定义每次持续时长

## 第一步：创建静默规则

### 操作页面
**侧边栏** → **值班驾驶舱** → **告警中心** → **告警列表**

### 操作步骤

1. **选择告警**
   - 在告警列表中，点击某条告警
   - 进入告警详情页

2. **点击「静默告警」按钮**
   - 弹窗填写：
     - **静默时长**：下拉选择（30分钟 / 1小时 / 6小时 / 24小时 / 自定义）
     - **静默原因**：输入静默原因（必填）
     - **按规则匹配**：勾选则静默该告警规则下所有告警（而非仅当前告警）

3. **点击「提交」**
   - 系统调用 `POST /alerts/{alert_id}/silence`
   - 写入 `AlertSilence` 记录：
     - `alert_id`：当前告警 ID
     - `rule_id`：告警规则 ID（如勾选按规则匹配）
     - `expires_at`：`now + 用户选择时长`
     - `reason`：用户输入的原因
   - 如果 `alert_id` 非空：设 `Alert.status = silenced`
   - 如果 `rule_id` 非空：批量设该规则下全部 `Alert.status = silenced`

### 看到什么
- 告警列表中，静默的告警显示灰色/斜体样式 + "静默中"标签
- 鼠标悬停显示静默原因和结束时间

## 第二步：静默生效期间

### 说明
静默期间，以下行为被跳过：
1. **告警聚合**：`correlate_alerts` 扫描时跳过 `status=silenced` 的告警
2. **自动响应**：`remediation_service.check_and_remediate()` 跳过静默告警
3. **通知推送**：`notification_service` 过滤静默告警不发送通知

### 看到什么
- 静默中的告警在列表中以灰色/斜体样式展示
- 标注「静默中」标签
- 鼠标悬停显示静默原因和结束时间

---
← **上一步：创建静默规则**　**下一步：静默过期自动恢复** →

## 第三步：静默过期自动恢复

### 说明
静默过期后，系统**自动恢复**告警状态：
1. 后台定时任务每分钟扫描 `AlertSilence` 表
2. 过期的静默规则（`expires_at < now`）被忽略
3. 关联告警的 `status` 从 `silenced` 恢复为：
   - `triggered`（未处理状态）
   - `acknowledged`（已确认状态，取决于原始状态）

### 看到什么
- 静默过期后，告警恢复为正常样式
- 状态从「静默中」恢复为「待处理」或「已确认」

## 第四步：手动取消静默

### 操作页面
**侧边栏** → **值班驾驶舱** → **告警中心** → **告警列表**

### 操作步骤

1. **选择静默中的告警**
   - 在告警列表中，找到显示「静默中」标签的告警

2. **点击「取消静默」**
   - 在告警详情页或列表操作栏，点击「取消静默」按钮
   - 系统调用 `DELETE /alerts/{alert_id}/silence`
   - 立即删除静默记录
   - 告警状态恢复

### 看到什么
- 告警状态从「静默中」恢复为「待处理」或「已确认」
- 「静默中」标签消失

## 常见问题

### Q1：知识草稿没有生成？
- **检查**：故障单状态是否变为 `resolved`
- **检查**：告警是否已解决（`alerts.py:api_resolve_alert` 触发）
- **检查**：LLM Provider 是否配置正确

### Q2：AI 检索结果不准确？
- **检查**：知识库是否有相关条目
- **检查**：知识条目的 `tags`/`asset_type` 是否匹配
- **检查**：RAG 索引是否更新（`rag_service.index_document()`）

### Q3：故障审批流报错？
- **检查**：故障状态是否正确（`open` 才能提交，`pending_approval` 才能审批）
- **检查**：审批人是否有权限
- **检查**：`IncidentApproval` 模型是否正常

### Q5：统一搜索返回结果为空？
- **检查**：知识库/文档/Runbook 中是否有匹配的数据
- **检查**：搜索关键词是否太短（LIKE 模式至少需要 1 个字符）
- **检查**：向量模式是否已启用（Milvus 是否在运行）
- **检查**：文本模式下 SQLite 全文索引是否正常

### Q6：版本号没有自动递增？
- **检查**：编辑 API 是否传了 `version_number` 字段（不传则自动 +1）
- **检查**：`change_log` 字段是否填写了变更说明
- **检查**：知识条目是否存在（新建时也会触发版本号）

### Q4：告警静默没有生效？
- **检查**：`AlertSilence` 记录是否创建成功
- **检查**：`expires_at` 是否大于当前时间
- **检查**：后台静默清理任务是否在运行
