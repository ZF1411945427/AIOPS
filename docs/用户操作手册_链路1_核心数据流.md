# AIOps 用户操作手册 — 链路1：核心数据流

> **适用角色**：运维工程师、SRE、系统管理员
> **前置条件**：已登录系统（admin / admin123），拥有「运维工作台」菜单权限
> **预计耗时**：完整走完约 15-20 分钟（含等待采集和检测周期）

---

## 概述

本手册引导您完整体验 AIOps 的核心数据流闭环：

```
资产发现 → 指标采集 → 可观测性展示 → 异常检测 → 告警生成 → 告警确认 → 故障单 → 知识沉淀 → 自愈执行
```

> **链路1 核心数据流**　①资产发现 → ②指标采集(后台) → ③可观测性 → ④异常检测配置 → ⑤异常检测(后台) → ⑥告警列表 → ⑦故障单 → ⑧审批流 → ⑨知识沉淀(后台) → ⑩知识草稿审批 → ⑪AI检索验证 → ⑫自愈(后台)
>

## 第一步：资产自动发现

### 操作页面
**侧边栏** → **运维工作台** → **资产管理** → **资产自动发现**

### 操作步骤

1. **点击「新建发现任务」按钮**
   - 页面右上角的蓝色按钮

2. **填写表单**
   - **任务名称**：输入 `测试自动发现`
   - **发现协议**：下拉选择 `SSH`（或 TCP / ICMP）
   - **目标范围**：输入 `192.168.1.1-10`（支持逗号分隔IP、短横线范围、CIDR）
   - **探测端口**：输入 `22`（SSH默认端口）
   - **认证凭据**：选择已配置的 SSH 凭据（或新建）

3. **点击「立即执行」或「保存」**
   - 保存：创建定时任务（按 cron 表达式定期执行）
   - 立即执行：触发一次扫描

### 看到什么

- **任务列表**：显示刚创建的任务，状态为「执行中」
- **执行结果**：扫描完成后，「结果」列显示发现的资产数量
- **资产自动入库**：发现的资产自动写入 Asset 表，`status=online`，`health_status=green`

### 结果
系统自动创建 Asset 记录，包含 `name`、`ci_type`、`ip`、`connection_config`（SSH 端口/用户名/密码）等字段。

## 第二步：指标采集（后台自动执行）

### 说明
指标采集是**后台定时任务**，无需手动操作。系统每轮循环自动执行：

### 采集过程

1. **读取在线资产**：查询 `Asset` 表中 `status=online` 的全部资产
2. **SSH 连接**：从 `Asset.connection_config` JSON 解析 `ssh_port`/`ssh_user`/`ssh_password`，用 paramiko 建立 SSH 连接
3. **执行 21 条采集命令**：

| 分组 | 指标名 | 命令 |
|------|--------|------|
| CPU（5条） | cpu_usage | `top -bn1\|awk '/Cpu/{print $2}'` |
|  | cpu_iowait | `top -bn1\|awk '/Cpu/{print $4}'` |
|  | loadavg_1min | `cat /proc/loadavg\|awk '{print $1}'` |
|  | loadavg_5min | `cat /proc/loadavg\|awk '{print $2}'` |
|  | loadavg_15min | `cat /proc/loadavg\|awk '{print $3}'` |
| 内存（3条） | memory_usage | `free -m\|awk '/Mem/{printf "%.1f",$3/$2*100}'` |
|  | memory_available | `free -m\|awk '/Mem/{print $7}'` |
|  | swap_usage | `free -m\|awk '/Swap/{if($2>0) printf "%.1f",$3/$2*100;else print 0}'` |
| 磁盘（2条） | disk_usage | `df -h /\|awk 'NR==2{print $5}'\|tr -d '%'` |
|  | disk_inode_usage | `df -i /\|awk 'NR==2{print $5}'\|tr -d '%'` |
| 网络（4条） | network_rx_bytes | `cat /proc/net/dev\|awk '/eth0/{print $2}'` |
|  | network_tx_bytes | `cat /proc/net/dev\|awk '/eth0/{print $10}'` |
|  | tcp_established | `ss -s\|awk '/TCP:/{print $2}'` |
|  | tcp_time_wait | `ss -s\|awk '/timewait/{print $1}'` |
| 系统（4条） | uptime_seconds | `awk '{print $1}' /proc/uptime` |
|  | process_count | `ps aux\|wc -l` |
|  | zombie_process | `ps aux\|awk '{if($8=="Z") count++}END{print count+0}'` |
|  | open_files | `cat /proc/sys/fs/file-nr\|awk '{print $1}'` |
| 应用连接（3条） | ssh_connections | `ss -s\|awk '/estab/{print $4}'` |
|  | http_connections | `ss -s\|awk '/estab/{print $4}'` |
|  | mysql_connections | `ss -s\|awk '/estab/{print $4}'` |

4. **双写存储**：
   - 写入 SQLite `MetricRecord` 表（`asset_id`/`name`/`value`/`unit`/`timestamp`）
   - 同时写入 VictoriaMetrics（`metric_v2_service.write_metrics_batch()`）

5. **健康评分更新**：根据 `Asset.ci_type` 映射到三层模型：
   - **功能接口层 (api)**：Span 错误率 > 5% 或 P99 > 1000ms → `health_status=red`
   - **中间件层 (middleware)**：有活跃告警 → `red`，否则 `green`
   - **基础设施层 (infra)**：CPU > 90% 或内存 > 90% 或磁盘 > 85% → `red`

### 看到什么
- 无需手动查看，采集过程在后台自动完成
- 可在「可观测性」页面看到采集到的指标数据

## 第四步：异常检测配置

### 操作页面
**侧边栏** → **智能分析室** → **异常检测** → **异常检测配置**

### 操作步骤

1. **查看现有规则**
   - 页面显示已配置的异常检测规则列表
   - 每条规则包含：指标名、算法、灵敏度、状态（启用/禁用）

2. **新建检测规则**（可选）
   - 点击「新建规则」按钮
   - **指标名称**：输入 `cpu_usage`（或选择其他指标）
   - **资产**：选择目标资产（或选择「全部资产」）
   - **算法**：下拉选择（推荐自动选择）：
     - **Sigma（3σ）**：适合平稳指标（如磁盘 I/O）
     - **EWMA**：适合渐变趋势（如内存使用）
     - **MAD**：适合噪声干扰强的指标（如 CPU）
     - **STL**：适合周期性波动（如 QPS）
     - **Prophet**：适合业务时序（如订单量）
     - **LSTM**：适合复杂模式（numpy 模拟）
     - **Transformer**：适合自注意力检测（numpy 模拟）
   - **灵敏度**：默认 `3.0`（Z-Score 阈值）
   - **窗口大小**：默认 `30`（最近 30 个数据点）
   - 点击「保存」

3. **启用规则**
   - 在规则列表中，点击「启用」开关
   - 规则状态变为「已启用」

### 看到什么
- 规则列表更新，新规则显示为「已启用」状态
- 系统开始按配置的算法对该指标进行异常检测

### 结果
异常检测规则已配置，系统将在后台定时执行检测。

## 第五步：异常检测执行（后台自动）

### 说明
异常检测是**后台定时任务**，无需手动操作。

### 检测过程

1. **查询启用的规则**：遍历 `AnomalyConfig` 表中 `enabled=True` 的全部配置
2. **获取指标数据**：对每条规则，查 `MetricRecord` 取最新 `window_size+5` 条记录
3. **算法检测**：按 `algorithm` 字段路由到对应检测函数：
   - **Sigma**：计算 Z-Score = |latest - mean| / stddev → Z > sensitivity 则触发
   - **EWMA**：指数加权移动平均 → 标准化残差 > 3.0 则触发
   - **MAD**：中位数绝对偏差 → 修正 Z 值 > sensitivity 则触发
   - **STL**：季节性分解 → 残差 Z-Score 超限则触发
   - **Prophet**：FB 预测 → 实际值超预测上限则触发
   - **LSTM**：滑动窗口线性回归 → 残差超阈值则触发
   - **Transformer**：自注意力检测 → 残差超阈值则触发
4. **去重防风暴**：检查 `Alert` 表中是否已有同指标+同资产+同算法的未解决告警，有则跳过
5. **生成告警**：确认无重复后，创建 `Alert` 对象：
   - `metric_name`：指标名
   - `actual_value`：当前值
   - `threshold`：阈值
   - `severity`：根据偏差程度决定 `warning` 或 `critical`
   - `status`：`triggered`
   - `message`：含算法名/偏差方向/Z值/当前值/阈值

### 看到什么
- 无需手动查看
- 异常检测会在「告警列表」页生成新告警

### 结果
异常检测自动完成，触发的告警进入告警列表。

## 第六步：告警列表查看与确认

### 操作页面
**侧边栏** → **值班驾驶舱** → **告警中心** → **告警列表**

### 操作步骤

1. **查看告警列表**
   - 页面显示所有告警，按 `created_at` 倒序排列
   - 每条告警显示：
     - **严重级别**：橙色（warning）/ 红色（critical）徽章
     - **告警标题**：`message` 字段内容（如 "CPU使用率异常: 当前值95.2%, 超过阈值90%"）
     - **关联资产**：资产名称 + IP 地址
     - **触发时间**：`created_at` 时间戳
     - **状态**：triggered（待处理）/ acknowledged（已确认）/ resolved（已解决）

2. **筛选告警**
   - **状态筛选**：点击 Tab 切换（全部 / 待处理 / 已确认 / 已恢复）
   - **严重级别**：下拉选择（warning / critical / info）
   - **搜索**：输入关键词搜索告警标题或资产名称

3. **确认告警**

   - 点击「确认」
   - **结果**：告警状态变为 `acknowledged`，`acknowledged_at` 记录确认时间

4. **查看告警详情**
   - 点击告警标题或「详情」按钮
   - 右侧弹出详情抽屉，显示：
     - 告警完整信息（指标名/当前值/阈值/算法/消息）
     - 关联资产详情（名称/IP/类型/健康状态）
     - 关联知识库推荐（如有）
     - 关联故障单（如有）

### 看到什么
- 告警列表显示刚生成的告警（红色或橙色徽章）
- 确认后状态从「待处理」变为「已确认」
- 详情抽屉显示完整的告警上下文

### 结果
告警已确认，可进入下一步创建故障单。

## 第九步：知识自动沉淀（后台自动）

### 说明
故障关闭后，系统**自动触发**知识沉淀，无需手动操作。

### 沉淀过程

1. **触发时机**：故障状态变为 `resolved` 后 5 分钟内
2. **自动归档**：后台任务调用 `knowledge_autogen_service.generate_draft(alert_id, db)`
3. **LLM 生成**：
   - 查询故障单详情 + 关联告警 + 关联资产
   - 构造 Prompt 调用 LLM 生成结构化知识：
     - `title`：故障标题衍生
     - `symptom`：告警 message 汇总
     - `root_cause`：RCA 根因分析
     - `solution`：修复建议
     - `tags`：从告警指标名/资产类型提取
     - `severity`：继承故障级别
     - `asset_type`：继承资产类型
4. **写入草稿**：创建 `KnowledgeDraft` 记录，`status=pending`，`source_type=auto`

### 看到什么
- 无需手动查看
- 生成的知识草稿出现在「AI 知识草稿」页面

### 结果
知识草稿已生成，等待人工审批。

## 第十一步：AI 检索知识（验证闭环）

### 操作页面
**侧边栏** → **AI运维智能体** → **AI 智能助手**

### 操作步骤

1. **发起问答**
   - 在聊天输入框输入问题（如 "CPU 使用率过高怎么处理？"）
   - 点击发送或按回车

2. **AI 检索知识**
   - 系统自动调用 `query_knowledge_rag` 工具
   - TF-IDF 向量化问题（中文按字分词）
   - 遍历所有 `KbChunk` 计算余弦相似度
   - 过滤相似度 < 0.05 的结果
   - 按相似度降序取 Top-5

3. **AI 生成回答**
   - LLM 根据检索到的知识内容生成回答
   - 回答中引用知识库条目作为信息来源

### 看到什么
- 聊天界面显示 AI 回答
- 回答中可能引用知识库条目（如 "根据知识库《XXX》..."）
- 回答下方显示「采纳为知识」按钮

### 结果
验证知识沉淀闭环：故障 → 知识生成 → 审批入库 → AI 检索 → 用户问答。

## 第十二步：告警自动响应（自愈）

### 说明
告警自动响应是**后台定时任务**，无需手动操作。

### 响应过程

1. **定时扫描**：后台任务 `remediation_service.check_and_remediate()` 每轮循环执行
2. **规则匹配**：查询 `AutoRemediation` 表中 `enabled=True` 的规则
3. **告警关联**：对每条规则，查询 `status=triggered` 的关联告警
4. **去重检查**：同一告警+同一规则最近已执行过则跳过
5. **SSH 执行**：
   - 根据 `alert.asset_id` 查 `Asset` 获取 IP/SSH 凭据
   - 用 paramiko SSH 连接远程资产
   - 按 `action_type` 执行动作：
     - `restart`：`sudo systemctl restart {service}`
     - `clean`：`find {path} -type f -mtime +7 -delete`
     - `run_command`：执行自定义命令（危险命令黑名单拦截）
     - `notify`：发送通知
6. **写入执行日志**：创建 `RemediationLog` 记录（成功/失败/输出）
7. **更新告警状态**：`Alert.status = acknowledged`
8. **效果追踪**：30 分钟后检查告警是否恢复，写入 `RemediationEffectRecord`

### 看到什么
- 无需手动查看
- 执行结果在「任务中心」→「自愈执行」页面可查看

### 结果
告警自动响应完成，形成「告警 → 自愈 → 效果追踪」闭环。

## 常见问题

### Q1：指标采集没有数据？
- **检查**：资产是否 `status=online`
- **检查**：`connection_config` JSON 是否正确（ssh_port/ssh_user/ssh_password）
- **检查**：网络是否连通（能否 SSH 到目标资产）

### Q2：异常检测没有生成告警？
- **检查**：`AnomalyConfig` 是否 `enabled=True`
- **检查**：`MetricRecord` 是否有足够数据（至少 `window_size` 条）
- **检查**：是否有重复告警（去重机制会跳过）

### Q3：告警自动响应没有执行？
- **检查**：`AutoRemediation` 是否 `enabled=True`
- **检查**：关联的 `AlertRule` 是否匹配当前告警
- **检查**：`RemediationLog` 是否有最近执行记录（去重机制会跳过）

### Q4：知识草稿没有生成？
- **检查**：故障单状态是否变为 `resolved`
- **检查**：告警是否已解决（`alerts.py:api_resolve_alert` 触发）
- **检查**：LLM Provider 是否配置正确
